## 2. 책 내용 핵심 요약

1. **모니터링 vs 관측 가능성**: 모니터링은 미리 알고 있는 문제를 감지하는 것이고, 관측 가능성은 알지 못했던 문제(unknown unknowns)를 외부 신호로 추론해낼 수 있는 능력이다.
2. **메트릭의 구조**: Counter / Gauge / Histogram / Summary 네 가지 유형이 있고, 구글 골든 시그널(Latency / Traffic / Errors / Saturation)은 서비스 상태를 판단하는 최소한의 기준이 된다.
3. **Prometheus 히스토그램의 버킷 구조**: 분포를 표현할 때 미리 정의된 버킷에 관측값을 누적시켜 저장한다. 이 구조 덕분에 `histogram_quantile` 같은 집계가 가능하지만, 버킷 경계 설정이 정확도를 결정한다.
4. **추적(Trace)의 작동 원리**: 분산 시스템에서 하나의 요청이 여러 서비스를 거칠 때, Trace ID와 Span ID를 통해 전체 흐름을 하나의 맥락으로 묶는다. 컨텍스트 전파(Context Propagation)가 핵심이다.
5. **상관관계(Correlation)가 실제 가치**: 메트릭에서 이상 감지 → Exemplar로 추적으로 점프 → Trace ID로 로그 필터링. 이 흐름이 장애 대응 시간을 결정적으로 줄여준다.

## 3. 중요하다고 생각한 개념

### 개념 1. Prometheus 히스토그램의 버킷 구조

#### 개념 설명

Prometheus 히스토그램은 관측값을 `le`(less than or equal) 레이블이 붙은 버킷에 누적 카운터로 저장한다. 예를 들어 HTTP 응답 시간을 측정할 때, `le="0.1"`, `le="0.5"`, `le="1.0"` 같은 버킷마다 해당 임계값 이하의 요청 수가 누적된다. `histogram_quantile(0.99, ...)` 함수는 이 버킷 분포에서 선형 보간으로 백분위수를 추정한다.

#### 중요하다고 생각한 이유

Prometheus를 실제로 사용하면서 `histogram_quantile`의 결과가 왜 근사값인지, 버킷 경계를 잘못 설정하면 왜 왜곡이 생기는지 명확히 이해하지 못했었다. 책에서 버킷 구조를 설명해주면서 이 동작 방식이 비로소 이해됐다.

#### 운영 관점에서의 의미

서비스 지연 시간 알람을 설정할 때 p99 기준으로 임계값을 잡는 경우가 많은데, 버킷 경계가 알람 임계값 근처에 촘촘하게 설정되어 있지 않으면 p99가 실제보다 낙관적으로 추정될 수 있다. 처음 Prometheus를 세팅할 때 버킷 경계 설계를 의도적으로 해야 한다는 점이 현업에서 중요하다.

### 개념 2. 상관관계(Correlation) — Exemplar와 Trace ID 연결

#### 개념 설명

메트릭 하나만 봐서는 "에러율이 올랐다"는 사실은 알 수 있어도, "어떤 요청 흐름에서 에러가 났는지"는 알 수 없다. Exemplar는 메트릭 데이터 포인트에 Trace ID를 첨부해서, 이상 지점에서 바로 추적으로 드릴다운할 수 있게 해준다. 추적 안에서는 로그를 Trace ID로 필터링해 로그까지 연결된다. 메트릭 → 추적 → 로그 전체가 하나의 맥락으로 이어지는 것이 핵심이다.

#### 중요하다고 생각한 이유

관측 가능성과 단순 모니터링의 차이가 사실 이 하나로 수렴한다고 느꼈다. 알람이 울렸을 때 대시보드를 이리저리 넘기며 수동으로 타임스탬프를 맞춰보는 과정이 얼마나 비효율적인지 경험으로 알고 있어서, 이 부분이 가장 강렬하게 와닿았다.

#### 운영 관점에서의 의미

장애 대응(MTTR)을 줄이는 가장 직접적인 수단이다. "메트릭 이상 감지 → 원인 있는 Trace 특정 → 해당 로그에서 스택 트레이스 확인"이 자동으로 연결되면, 장애 상황에서 수동 탐색에 쓰는 시간이 대폭 줄어든다. 특히 마이크로서비스 환경에서는 서비스 경계를 넘는 요청 흐름을 추적하는 것이 사실상 Trace 없이는 불가능하다.

### 개념 3. 메트릭의 카디널리티 관리

#### 개념 설명

메트릭에 붙이는 레이블(label)의 조합 수가 카디널리티(cardinality)다. 레이블 값이 무한히 늘어날 수 있는 값(예: user_id, request_id)을 레이블로 쓰면 시계열 수가 폭발적으로 증가한다. Prometheus는 시계열 수에 비례해 메모리·디스크·쿼리 비용이 증가하기 때문에 카디널리티 관리는 운영 안정성과 직결된다.

#### 중요하다고 생각한 이유

직접 겪어본 영역이라 더 와닿았다. 레이블 설계를 잘못하면 운영 중 Prometheus가 메모리를 과도하게 쓰기 시작하는데, 이미 수집 중인 시계열을 줄이는 것은 새로 추가하는 것보다 훨씬 어렵다.

#### 운영 관점에서의 의미

메트릭 레이블은 처음 설계할 때 신중해야 한다. 특히 고유한 값이 될 수 있는 필드는 레이블이 아니라 로그나 추적으로 분리해서 다뤄야 한다. 레이블은 그룹핑에 쓸 고정 카테고리만 남기는 것이 원칙이다.
