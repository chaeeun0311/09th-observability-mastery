## 2. 프로메테우스의 구성

- 프로메테우스 서버
    - http Server: PromQL 쿼리를 처리
      - PromQL: 프로메테우스에 저장된 시계열 데이터를 조회하고 계산하는 쿼리 언어
    - TSDB: 시계열 메트릭 저장
    - Retrieval(수집 엔진): 대상 목적지로 부터 메시지를 Pull방식으로 주기적으로 가져옴
    
- 프로메테우스 오퍼레이터
    - 쿠버네티스 자원관리, 프로비저닝
    - 동적으로 증가하는 서비스와 파드를 발견하고 프로메테우스의 구성파일을 업데이트
    
- 프로메테우스 익스포터
    - 해당 서버에서 메트릭 수집 후 프로메테우스가 이해할 수 있는 정보로 바꿔서 서버로 전송
    - 커스텀 익스포터를 개발할때 이에 필요한 API, SDK 제공
    
- 프로메테우스 어댑터
    - 익스포터로 수집된 메트릭을 측정하고 HPA 수행
    

- Service Discovery
    - 쿠버네티스나 파일 설정을 통해 모니터링해야 할 대상들의 IP와 포트 목록을 자동으로 알아냄
- Pushgateway
    - 예외 케이스 처리 / 배치 작업이나 몇 초만에 끝나는 작업은 프로메테우스가 주기적으로 Pull 하기 전에 종료될 수 있기때문에 예외적인 데이터를 Pushgateway에 Push로 넣고 프로메테우스가 데이터를 Pull해옴
- Node
    - TSDB에 들어온 최종 데이터는 안전한 디스크에 저장됨

## 3. 프로메테우스의 Pull 방식

스크래핑: 프로메테우스 서버가 데이터를 수집하는 방식

```
[Prometheus Server]                                     [Target (예: Node Exporter)]
        │                                                           │
        │ 1. 서비스 디스커버리로 Target IP/Port 파악                      │
        │──────────────────────────────────────────────────────────>│
        │                                                           │
        │ 2. HTTP GET /metrics 요청 (예: 15초 주기)                    │
        │──────────────────────────────────────────────────────────>│
        │                                                           │
        │                                  3. 자신의 메트릭을 Text로 생성 │
        │                                                           │
        │ 4. HTTP 200 OK + 메트릭 데이터 응답                            │
        │<──────────────────────────────────────────────────────────│
        │                                                           │
        ▼
 5. TSDB에 저장 및 처리 (데이터 파싱)
```

프로메테우스 서버가 주도권을 갖고 대상에서 데이터를 당겨오는 방식

| **구분** | **Pull 방식 (프로메테우스)** | **Push 방식 (Datadog, Telegraf 등)** |
| --- | --- | --- |
| **데이터 전송 주체** | **모니터링 서버**가 대상에게 요청 | **에이전트**가 모니터링 서버로 전송 |
| **네트워크 방향** | 서버 ➔ 대상 (Inbound 열려있어야 함) | 대상 ➔ 서버 (Outbound 열려있어야 함) |
| **서버 확장성/안정성** | 서버가 수집 능력을 스스로 조절하여 안전함 | 에이전트가 급증하면 서버에 부하 발생 가능 |
| **동적 환경 대응** | 서비스 디스커버리 연동이 필수적임 | 에이전트가 서버 주소만 알면 자동 등록됨 |
| **단기 실행 작업** | 수집하기 어려움 (Pushgateway 필요) | 작업 종료 직전 전송하면 되므로 매우 유리함 |
| **주요 적합 환경** | **쿠버네티스, 마이크로서비스(MSA)** | 전통적인 레거시 서버, 분산된 외부 클라이언트 |

- 장점
1. 과부하 방지</br>
    push 방식은 트래픽이 몰리는 상황에서 수천 개의 에이전트가 동시에 데이터를 푸시하면 모니터링 서버가 뻗을 수 있음. pull 방식으로 서버가 감당할 수 있는 속도와 주기로 데이터를 당겨옴
    
2. 대상 안정성</br>
    에이전트를 대상에 설치하면, 모니터링 서버가 어디있는지 주소가 바뀌었는지 알 필요가 없음. push 방식은 수천 개의 에이전트에서 모니터링 서버의 IP를 변경하야하는 번거로움이 있다.
    
- 단점
1. 정밀한 실시간성의 한계 </br>
    수집주기(ex. 15초) 사이에 순간적이 피크상태 같은 변화는 놓칠 수 도 있음

## 4. 프로메테우스의 시계열 데이터

1. 데이터 형식
    
    ```bash
    - 키 값
    http_requests_total {status ="200", method="GET"} @143317560138 94355
    ----------------------키--------------------------------------- --값--
    메트릭 이름{레이블_이름="값"}
    
    ```
    
    - 카디널리티
    
    프로메테우스 인스턴스에 할당되는 컴퓨팅 리소스에 따라 여러 개의 시계열을 점진적으로 처리
    
    의미: 전체 행에 대한 특정 칼럼의 중복 수치를 나타내는 지표
    
    ```
    
    웹 서버의 요청 횟수를 측정하는 메트릭이 있다고 가정
    
    - 메트릭 이름: http_requests_total
    - 레이블: method (GET, POST), status (200, 500)
    
    이 경우 프로메테우스가 관리해야 하는 고유 시계열은 총 **4개**(2 x 2)가 됨.
    1. http_requests_total{method="GET", status="200"}
    2. http_requests_total{method="GET", status="500"}
    3. http_requests_total{method="POST", status="200"}
    4. http_requests_total{method="POST", status="500"}
    이 4개의 가짓수(수학에서의 집합의 크기)를 카디널리티(Cardinality)라고 한다.
    
    ```
    
    - 카디널리티 문제 </br>
    메트릭에 수 없이 조합된 여러 차원은 프로메테우스에서 카디널리티 폭발을 야기한다. (너무 많은 카디널리티) 한계치가 명확하지 않은 값이 무한히 증가하거나 예상치 못하는 개수를 초과
    
    ```
    
    사용자마다 다른 고유 ID인 `user_id`나 요청자의 `ip` 주소를 레이블로 추가하면?
    
    http_requests_total{method="GET", status="200", user_id="유저ID"}
    
    - 메서드 종류: 2개 (GET, POST)
    - 상태 코드: 2개 (200, 500)
    - 사용자 수: 100,000명 (무한히 증가할 수 있는 값)
    
    이제 프로메테우스가 생성하고 메모리에 유지해야 하는 고유 시계열의 수는 2 X 2 X 100,000 = 400,000개로 급증합니다.
    
    만약 사용자가 1,000만 명이 된다면 조합의 수는 4,000만 개가 되겠죠. 이렇게 레이블 값의 가짓수가 곱하기 연산으로 엮이면서 데이터 종류가 기하급수적으로 늘어나는 현상을 "카디널리티 폭발"이라고 한다..
    
    ```
    
    프로메테우스는 빠른 조회를 위해 모든 시계열의 메타데이터(수집중인 모든 고유 조합)를 메모리에 올려두고 관리하는 아키텍처를 가지고 있다.
    
    따라서 메모리 고갈과 조회 시 성능 저하가 발생할 수 있다.
    
2. 데이터 관리
    - TSDB(Time Series Database): 시계열 데이터베이스
    - LRU알고리즘: 가장 오랫동안 참조하지 않은 페이지 교체
    - 메모리 페이징: 일정크기 페이지로 분할해 적재
    - 샘플(수집 데이터)을 수집하고 블록 형태로 디스크에 저장
    - 데이터(샘플)대신 ‘청크’라는 용어 사용.
    - 메모리에 우선 저장하고, 주기적으로 플러시 해서 디스크에 청크 생성
    - 메모리에 있는 데이터는 백업용도로 WAL 생성, 장애 발생 시 WAL로 복구
    - TSDB와 WAL 파일로 복구
    
   ```text
    청크
    
    하나의 고유한 시계열(예: 특정 지점의 `GET/200` 메트릭)이 가지는 [타임스탬프 + 숫자 값]의 연속된 묶음
    
    - 샘플 (Sample): 데이터의 최소 단위. (예: 12시 00분 00초의 CPU 사용량 = 25.4%)
    - 청크 (Chunk): 한 종류의 메트릭에 대한 샘플 수십~수백 개를 압축한 묶음. 보통 2시간 정도 분량의 데이터가 하나의 청크가 됨.
    - 블록 (Block): 디스크에 저장되는 가장 큰 단위. 특정 시간대(예: 2시간) 동안 수집된 모든 메트릭의 '청크'들을 한데 모아둔 폴더
    ```

    ```text
      TSDB의 데이터 관리 방식: 블록
  
      프로메테우스 TSDB가 대용량의 데이터를 관리하기 위한 방법으로
      디스크에 저장되는 기본 디렉터리 단위이다.
      
      프로메테우스는 데이터를 기본적으로 2시간 단위의 블록으로 쪼개서 저장한다. 따라서 최근 2시간동안 데이터를 조회한다면 전체 데이터를 찾을 필요없이 해당 시간의 폴더만 열면 된다.
      
      블록 디렉터리 내부에는 청크, 인덱스, 메타 데이터 정보들이 있다.
      
      컴팩션: 자잘한 블록들을 더 큰 단위로 합치는것 (2시간 단위 블록 → 8시간 단위 블록)
      
    ```
  
    ```text
      
      데이터 압축률
      
      메트릭 데이터는 `수집 시간`, `값` 으로 이루어진 숫자 쌍으로,  페이스북이 개발한 Gorilla 알고리즘을 변경해 사용
      
      또한 시간 값의 변화량의 변화량 (델타-델타 압축)을 기록해 크기를 대폭 줄임
      
    ```
    

1. 블록 관리
    
    시계열 데이터: 프로메테우스 저장을 위한 필수 요소로, 타임스탬프(시간)와 float64(데이터 값)로 구성된 샘플 형태입니다.
    샘플: 특정 시간에 측정한 수집값 하나
    
    여러 블록들이 병합되어서 더 큰 적당한 블록 크키로 만들고 효율적으로 저장
    
    - 로컬스토리지(TSDB) 데이터 흐름
    
    1단계: 메모리 (Memory) & WAL
    
    - **최신 데이터 보관:** 모든 최신 데이터는 우선 메모리에 저장됨
    - **WAL (로그 선행 기입):** 시스템이 비정상 종료되었을 때 데이터가 날아가는 것을 방지하기 위해, 메모리 상태를 유지하는 로그를 디스크에 먼저 기록
    - **체크포인트:** 주기적으로 체크포인트 파일을 기록하며, 시스템 재시작 시 이를 이용해 메모리로 데이터를 빠르게 복구
    
    2단계: 디스크 (Disk) 저장
    
    - **2시간 주기:** 데이터가 쌓인 지 2시간이 지나면 메모리에서 디스크로 넘어가며 하나의 '블록(Block)'으로 기록된다.
    - **삭제 표시:** 디스크에 저장된 데이터를 삭제할 때는 즉시 지우지 않고, '삭제 표시 파일(Tombstones)'을 생성하여 처리.
    
    3단계: PromQL 조회
    
    - **온디맨드 로드:** PromQL 쿼리 실행 시 현재 메모리에 없는 과거 데이터(청크)가 필요하면, 디스크에서 해당 데이터를 찾아 메모리로 로드한다.
    
    - 블록 관리 및 병합
    
    시간순으로 생성된 작은 블록들을 효율적으로 관리하기 위해 저장 공간을 최적화하는 과정
    
    - **블록 병합:** 시간이 지나면서 짧은 시간 단위의 작은 블록들을 합쳐서 더 큰 블록으로 만든다. 저장 및 검색 효율성이 높아짐
    - **주요 설정 옵션:**
        - `-storage.tsdb.min-block-duration`: 하나의 블록이 생성되는 최소 시간 단위 (기본값: 2시간).
        - `-storage.tsdb.max-block-duration`: 블록들이 병합되어 최대로 커질 수 있는 시간 범위.
    
    ```yaml
    예) min=2시간 max=24시간
    
    - 새벽 02:00가 됨: [00:00 ~ 02:00] 블록 1 생성
    - 새벽 04:00가 됨: [02:00 ~ 04:00] 블록 2 생성
    - 아침 06:00가 됨: [04:00 ~ 06:00] 블록 3 생성
    
    작은 블록이 너무 많으면 프로메테우스는 백그라운드에서 이 블록을 합친다. 보통 인접한 3개를 묶어서 합친다.
    아침 06:00 직후: 프로메테우스가 블록 1, 2, 3을 합쳐서 6시간짜리 큰 블록이 됨 [00:00 ~ 06:00] 
    
    또 시간이 지나서 6시간 블록이 3개 생기면 
    [00:00 ~ 06:00] (6시간짜리 블록)
    [06:00 ~ 12:00] (6시간짜리 블록)
    [12:00 ~ 18:00] (6시간짜리 블록)
    3개의 블록을 묶어서 [00:00 ~ 18:00]으로 합침.
    
    max 값은 블록이 최대로 커질 수 있는 시간으로 이 시간이상으로 합칠 수 없다.
    ```
    
    - 아키텍처적 한계
    
    **참고 사항**
    / 프로메테우스는 시계열 데이터 저장 및 조회에 특화되어 있어, 엘라스틱서치(Elasticsearch) 같은 전문 분산 스토리지 시스템에 비해 **백업, 복구, 스냅샷 관리 기능이 상대적으로 부족!**

## 5. 프로메테우스 기반 쿠버네티스 오토스케일링

- 오토스케일링 데이터 흐름
1. 프로메테우스가 메트릭을 수집 </br>
   애플리케이션 파드나 인프라로부터 매트릭 수집후 TSDB에 저장
2. 쿠버네티스 HPA 확인 주기 </br>
   HPA 컨트롤러는 오토스케일링 할지를 판단하기 위해 `Custom Metrics API` 서버에 메트릭을 요청하게됨
3. 통역사(프로메테우스 어댑터) </br>
   쿠버네티스는 프로메테우스의 PromQL을 알지 못하기 때문에 `Custom Metrics API`로 들어온 요청을 프로메테우스 어댑터가 가로챔
4. 프로메테우스에 쿼리 조회 </br>
   어댑터는 쿠버네티스 요청을 PromQL로 변환하여 프로메테우스 서버에 보내고, 서버는 TSDB에서 값을 꺼내 어댑터에게 반환
5. 오토스케일링 </br>
   어댑터가 쿠버네티스가 읽을 수 있는 형태로 HPA에 최종전달 -> 스케일 아웃 / 스케일 인

- 정교한 오토스케일링(HPA): CPU/메모리 같은 시스템 메트릭이 아닌, 애플리케이션 고유의 커스텀 메트릭(예: HTTP 요청 수, 큐 대기 시간 등)을 기준으로 서버를 늘리거나 줄이기 위해 사용함.
- 프로메테우스 어댑터의 역할: 프로메테우스에 쌓인 PromQL 기반 메트릭을 쿠버네티스가 읽을 수 있는 API 형태(`custom.metrics.k8s.io`)로 포맷을 변환하여 노출해 주는 다리 역할을 함.

쿠버네티스 서비스 디스커버리
변하는 파드(Pod)의 IP를 자동으로 추적해서, 클라이언트가 원하는 파드에 언제나 올바르게 접근할 수 있도록 연결해 주는 메커니즘
- 실시간 탐색: 쿠버네티스 환경에서는 파드(Pod)의 IP가 수시로 바뀌기 때문에, API 서버를 통해 수집 대상을 실시간으로 찾아내야 함
- 수집 필터링: 탐색된 모든 파드를 수집하는 것이 아니라, 레이블 재지정(`relabel_configs`) 설정을 통해 수집 활성화 조건이 붙은 파드만 골라내어 메트릭을 긁어옴.

| **단계** | **항목명** | **역할** |
| --- | --- | --- |
| **1. Discovery** | `seriesQuery` </br> `seriesFilter` | 프로메테우스에서 가져올 대상 메트릭을 규칙으로 지정함 (예: `nginx_`로 시작하는 메트릭만 필터링). |
| **2. Association** | `resources` | 찾아낸 메트릭을 쿠버네티스의 어떤 리소스(Pod, Namespace 등)와 매칭할지 결정함. |
| **3. Naming** | `name` | 쿠버네티스 커스텀 메트릭 API에서 사용할 직관적인 이름으로 가공함. |
| **4. Querying** | `metricsQuery` | HPA가 메트릭을 요청할 때, 어댑터가 실제로 프로메테우스 서버에 날릴 PromQL 쿼리를 정의함. |
