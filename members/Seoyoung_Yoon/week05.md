# 1. 그라파나 관측 가능성

관측가능성은 **메트릭, 로그, 분산 추적** 이 세가지 데이터가 유기적으로 연결되어야한다. (=**LGTM Stack)**

# 2. 로키 로그 관리

## 로키의 과정

1. 로그를 저장하는 과정(쓰기 경로)
    
    ```bash
    [로그 생성 (App/Flog)] ──> [로그 수집기 (Promtail)] ──> [Distributor] ──(해시링 분산)──> [Ingester] ──> [장기 스토리지 (S3 등)]
    ```
    
    1. **로그 수집:** 애플리케이션이나 에이전트(Flog, Promtail)가 로그 생성 및 수집.
    2. **배포:** Distributor가 로그 데이터를 받아 유효성을 검증하고 Ingester로 전달.
    3. **압축 및 저장:** Ingester가 메모리에 로그를 쌓아두고 청크(Chunk) 단위로 압축 후, 일정 시간(또는 크기)이 지나면 장기 스토리지에 기록.
2. 로그를 검색하는 과정(읽기 경로)
    
    ```bash
    [유저 (Grafana)] ──> [Query Frontend] ──(쿼리 분할/엔진)──> [Querier] ──+──> [Ingester (메모리 내 실시간 로그 검색)]
                                                                           +──> [장기 스토리지 (과거 로그 검색)]
    ```
    
    1. **쿼리 접수:** 유저가 그라파나 UI에서 쿼리를 던지면 Query Frontend가 가장 먼저 접수.
    2. **쿼리 가공:** 대량의 시간 범위를 가진 쿼리를 작게 분할(Sharding)하고 대기열(Queue)에 정렬.
    3. **검색 수행:** Querier들이 분할된 쿼리를 배정받아 최신 데이터(Ingester 메모리)와 과거 데이터(장기 스토리지)를 동시에 조회하여 병합 후 반환.

## 로키 기능

- 그라파나 로키 마이크로서비스 모드에서 데이터가 저장되는 흐름

- 용어 정리
    - 스트림: 나열된 로그데이터의 줄기, 동일한 레이블의 집합
    - 쿼리: 조건 검색 명령문

- 로키 컴포넌트
    - Distributor
        - 역할: 로그 수집의 첫 단계 처리이다.
        - 일관된 해시 링: 레이블을 기반으로 로그 분산 타깃 Ingester를 결정함.
    - Ingester
        - 역할: 로그 수집 및 압축이다.
        - 청크 형성: 메모리에 로그를 쌓아두고 압축 덩어리를 만들어 스토리지에 전달함. 읽기 요청에 대해 메모리 데이터를 즉각 반환함.
    - Query Frontend
        - 역할: 사용자 쿼리 조율 및 최적화이다.
        - 쿼리 분할: 긴 시간 대역의 쿼리를 분할하여 검색 효율을 높임.
        - 캐싱: 동일 검색 결과는 캐시에서 바로 꺼내어 응답함.
        - 큐잉: 대기열을 통한 요청 제어로 시스템 과부하를 방지함.
    - Querier
        - 역할: 실제 데이터 탐색이다.
        - 동시 검색: 분할된 작업을 병렬로 동시 수행하여 검색 속도를 극대화함.
    - 인덱스 게이트웨이
        - 역할: 장기 스토리지 인덱스 조회를 빠르게 돕는 컴포넌트이다.
- 그라파나 로키레이블
    
    키와 값의 조합으로 로그 스트림을 정의함. 레이블 하나만 변경되어도 새로운 스트림이 생성된다.
    
    - 스트림 변동 최소화 작업
        - {app="frontend", env="production"}과 같이 카디널리티가 낮은 변하지 않는 메타데이터를 레이블로 지정해야 함.
    - 병렬 처리를 통한 로키 성능 향상
        - 일래스틱서치
        색인(데이터를 검색가능한 구조로 변경하기 위해 원본 문서를 검색어 토큰으로 변환하여 저장)으로 사용자가 원하는 질의에 대한 정확한 결과를 반환
        인덱스 사용
        - 로키의 강력한 기능
            
            쿼리를 작은 조각으로 분할하고 병렬로 전달하여 짧은 시간애 방대한 양의 로그 데이터를 쿼리할 수 있음. (=쿼리 프론트엔드)
            
            병렬 다운스트림 처리: 쪼개진 작은 쿼리들은 수십, 수백 개의 쿼리어 노드에 동시에 분산
            
        - 로키는 스트림 변동을 최소화하는 작업을 잘 수행하고 있다면 수집된 로그에 비해 인덱스가 매우 느리게 증가한다.
            
            ```bash
            레이블
            {userid="123123"} -> 수시로 변하는 값은 스트림 종류 증폭! 
            {app="frontend", env="production"} -> 최소화 작업
            ```
            
- Promtail: 서버에 직접 설치되어 로그 파일을 모니터링하고 로키로 전송하는 수집기이다.
- Flog: 테스트 및 개발을 위해 임의의 로그 스트림을 생성하는 에이전트이다.

# 3. 미미르 메트릭 관리

미미르: 수평 확장형 엔터프라이즈급의 시계열 데이터베이스(TSDB)

단일 서버 구조를 사용하는 프로메테우스의 한계를 극복하기 위해 등장

프로메테우스의 데이터를 안전하게 저장하는 용도

## 미미르 기능

- 미미르 구성 컴포넌트
    - Distributor: 프로메테우스 메트릭을 유효성 검증한 후 Ingester로 분산하는 입구이다.
    - Ingester: 메트릭을 받아 메모리에 저장하고 2시간 주기로 스토리치에 전송하는 쓰기 주체이다.
    - 스토어 게이트웨이: 장기 스토리지에 압축되어 보관 중인 과거 데이터 블록을 빠르게 찾을 수 있도록 인덱스를 제공함.
    - 콤팩터: 여러 Ingester가 올린 작은 데이터 블록들을 정렬 및 병합하여 중복을 없애고 저장 공간을 관리함.
    - Query Frontend 및 Querier: PromQL 요청을 최적화하고 분할하여 실제 스토리지와 메모리 데이터를 고속 병렬 처리함.

- 쓰기 경로
    1. 프로메테우스가 미미르에 메트릭을 Push
    2. 디스트리뷰터가 레이블을 기준으로 해시링(Hash Ring)을 돌려 담당 **인제스터** 3곳을 찾아냄.
    3. 인제스터들의 메모리에 데이터가 기록(TSDB블록형태)되고, 동시에 복구용 쓰기 로그(WAL)에 저장
    4. 2시간이 지나면 블록을 스토리지로 업로드
- 읽기 경로
    1. 유저가 그라파나에서 PromQL로 메트릭을 검색
    2. 쿼리 프론트엔드가 쿼리를 분석하고 시간, 라벨별로 샤딩(분할)
    3. 분할된 쿼리는 쿼리어에게 전달
    4. 쿼리어는 인제스터 메모리나 스토어 게이트웨이를 거쳐 스토리지를 뒤져 데이터를 병합한 뒤 반환
    

# 4. 템포 추적 관리

### 템포 핵심 개념

- 스판: 작업의 최소 단위로 서비스 이름, 실행 상태, 소요 시간 데이터를 포함함.
- 트레이스: 여러 개의 스판들이 모여 하나의 온전한 요청 흐름을 형성하는 계층 구조이다. 고유한 Trace ID를 통해 식별됨.
- No-Index 아키텍처: 자체 인덱스를 생성하지 않고 오직 Trace ID로만 데이터를 검색하는 구조이다. 메트릭이나 로그에서 ID를 알아낸 뒤 직접 조회를 시도하여 인프라 비용을 극적으로 낮춘다.

### 템포 동작 흐름

### 쓰기 경로

1. 소스 코드에 탑재된 추적 SDK가 스판 데이터를 수집하여 전송함.
2. 템포 Distributor가 데이터를 받아 유효성을 판단하고 Ingester로 보냄.
3. Ingester가 메모리에 데이터를 임시 정렬하고 장기 스토리지로 업로드함.

### 읽기 경로

1. Grafana에서 사용자가 특정 Trace ID를 입력함.
2. Query Frontend가 쿼리를 접수하고 Querier에 작업을 할당함.
3. Querier가 Ingester 및 장기 스토리지에서 해당 Trace ID 데이터를 직접 매핑하여 읽어온 뒤 타임라인 트리 뷰로 출력함.

# 5. 예거 추적 관리

예거는 마이크로서비스 환경에서 트랜잭션을 추적하고 모니터링하기 위해 만들어진 대표적인 분산 추적 플랫폼이다.

### 예거 구성 컴포넌트

```bash
[예거 데이터 수집 아키텍처]
Application -> Jaeger Agent -> Jaeger Collector -> 영구 저장소 (Elasticsearch)
```

- 예거 에이전트: 로컬 호스트에 상주하며 UDP 프로토콜로 전송되는 스판 정보를 수집 및 정렬하여 콜렉터로 전달함.
- 예거 콜렉터: 다수 에이전트로부터 전달받은 트레이스 데이터를 검증 및 병합하여 영구 저장소에 기록함.
- 예거 인제스터: 대규모 유입 제어를 위해 카프카 같은 메시지 큐를 운영할 경우, 큐로부터 데이터를 직접 소비해 영구 저장소에 기록함.
- 쿼리 및 UI: 저장 공간에서 트레이스 데이터를 직접 쿼리하여 사용자 웹 화면에 그래프 형태로 보여줌.
