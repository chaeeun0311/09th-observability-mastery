### 개념 1. 데이터의 복원력을 높이는 방법

### 개념 설명

복원력을 높이는 4가지 방법

1. 재시도
2. 비율 제한    
    구현하기 위해서 이스티오를 사용할 수도 있지만, 가능하다면 다양한 인그레스와 게이트웨이 로드밸러너싱을 활용하는것을 권장한다.
    
    이스티오는 인그레스 후에 클러스터내부에서 차단하는 방식이기 때문에 트래픽이 서버로 들어오게 되어 영향을 준다.
    
    ingress/Gateway는 입구에서 바로 차단하는 방식 가능하면 더 바깥쪽에서 막아라
    
3. 벌크헤드    
    벌크헤드 패턴? 다운스트림 서비스를 **서로 격리(제한)**하고 각 서비스의 동시 처리능력을 제어
    
    - **업스트림 (Upstream - 상류):** 트래픽이 **시작되거나 들어오는 곳** (요청을 **보내는** 서비스)
    - **다운스트림 (Downstream - 하류):** 트래픽이 **도달하거나 흘러가는 곳** (요청을 **받는** 서비스)
    
    $\text{클라이언트} \longrightarrow \text{A 서비스 (업스트림)} \longrightarrow \text{B 서비스 (다운스트림)}$
    
    ```bash
    [사용자] 
       │
       ▼
    [주문 서비스 (Order Service)] ── (업스트림)
       │
       ├──▶ [결제 서비스 (Payment Service)] ── (Order의 다운스트림)
       │
       └──▶ [배송 서비스 (Delivery Service)] ── (Order의 다운스트림)
    ```
    
    이 개념이 필요한이유: 마이크로서비스를 운영할때 하나의 (다운스트림)서비스의 가용성만 저하되어도 여기에 의존하는 다른 서비스까지 응답 불가 상태에 놓일 위험이 있다.
    
    예) 주문API, 리뷰작성API가 있다. 여기서 리뷰API가 폭주하게 이미지 저장 서버가 느려지면서 되면 리뷰API의 스레드들이 반환되지 않고 묶여버린다.
    해당 서버를 리뷰API가 모두 차지해버리면서 주문API마저도 할당받지 못하고 먹통이된다.
    벌크헤드 적용: 스레드풀 격리 주문API 70개 할당, 리뷰API 30개 할당
    
4. 서킷 브레이커
    
    특정 다운스트림 서비스에 장애가 발생했을때 요청을 즉시 차단하여 시스템 전체가 무너지는것을 막는 장치
    
    폴백: 서킷브레이커가 열였을때 500에러 페이지를 보지 않도록 우회할 수 있는 대안책을 준비해두는것
    
    서킷 브레이커 + 분한 메시지 시스템(카프카)
    
    동기 요청을 비동기 이벤트로 전환하여 시스템의 결합도를 낮출수있음
    
    기본적인 HTTP 통신은 동기적이다. 주문 서비스가 결제 서비스를 호출하면 결제가 끝날 때까지 주문 서비스는 멈춰서 기다려야함. 이때 결제가 터지면 주문도 터지기 때문에 서킷브레이커가 문을 닫아버림.
    
    이 중간에 카프카를 둔다?
    
    주문 서비스는 주문 접수됨 메시지를 카프카 토픽에 던지고 자기 할 일을 끝내버림. 따라석 결제 서비스는 자신의 속도에 맞게 처리하면 된다.
    

|  | 차이 |
| --- | --- |
| 벌크헤드 | 자원 격리, 상한선 제한 |
| 서킷 브레이커 | 자원 차단 |

### 중요하다고 생각한 이유

마이크로서비스 환경에서 트래픽이 몰려 오류가 발생했을때 이를 대응하는 방안이기 때문이다.

특히 마이크로서비스 환경에서 특정 서비스에서 문제가 발생했을때 장애대응을 하기 어렵기 때문에 이런 복원력을 높이는 벌크헤드, 서킷브레이커 방식에 집중했다.

### 개념 3. 쿠버네티스 HAP

### 개념 설명

1. 메트릭 측정

    APM에서 제공하는 기본 메트릭을 초당 처리개수(TPS), 처리 지연시간(latency) 등

- 요청 수
    
    애플리케이션에서 요청수 개산은 가장 기본적인 방법. 요청수를 파악해 애플리케이션을 자동으로 확장해야하는지 판단할 수 있음
    
- 요청 기간
    
    서비스의 상태를 이해하는데 중요한 지표
    
- 동시 요청
    
    동시에 처리할 수 있는 동시 요청 수, 병목현상 급증 요청에 대한 증상 확인 가능
    
1. 메트릭 선정

HPA를 설정할 때 고려해야할 주요 영역

1. 메트릭 선택
    
    메트릭값과 파드 레플리카 수 사이의 **상관관계**가 있어야한다.
    
    ```bash
    예를 들면, 파드에 1000TPS가 오는 경우라면 파드를 2로 늘렸을때 부하량이 500TPS로 줄어드는 경우
    ```
    
    파드 수 와 직접 관련된 메트릭을 선택해야한다. 
    
    ```bash
    메모리 메트릭은 부적적한 이유
    - Java나 NodeJS 같은 환경은 가비지 컬렉션이 발생하기 전까지 메모리를 붙잡고 있기때문에 요청이 끝나도 메모리가 줄어들지 않는다.
    - 파드를 아무리 늘려도 파드가 사용하는 메모리 자체는 줄지 않기때문에 파드를 늘려도 메모리가 줄지않는다.
    ```
    
2. 스레싱 방지
    
    스레싱: 오케스케일링 파드가 늘었다 줄었다를 너무 빠르게 반복하는 현상
    
    부하가 안정적이지 않을때 레플리카 수의 잦은 변동을 야기할 수 있는 실행을 막는다.
    
3. 지연 반응
    
    HPA스케일링 공식은 변동/스레싱을 방지하기 위해 약간의 지연된 반응을 도입한다.
    
    → 수치 변동이 있을때마다 칼같이 반응하면 스레싱이 뛰기때문에 일정시간동안 데이터 추이를 지켜보고 반응한다.

### 개념 3. 안정적인 데이터 관리

### 개념 설명

1. 데이터 파티션
    
    **안정해시**를 사용해 데이터를 여러 서버 고르게 분산
    
    - 샤딩과 파티셔닝의 차이
    
    샤딩: 주로 메모리 분리
    
    파티셔닝: 시간 순서에 따라서 적재되며 주로 디스크
    
2. 데이터 다중화
    
    높은 가용성과 안정성을 확보하기 위해 N개의 서버에 비동기적으로 다중화
    
    - S3: 스토리지 파일을 3개 복제
    - 카프카: 레플리카로 메시지 복제
    - 일래스틱서치: 프라이머리와 레플리카 구성
3. 데이터 일관성
    
    정족수 합의: R(쓰기 성공 수)+W(읽기 성공 수) > N 인경우 강한 일관성이 보장된다.
    
    데이터의 복제본을 유지하면서 높은 성능을 유지하고 정합성을 높이는 것은 어려운 작업
    
4. 장애감지
    
    가십프로토콜
    
    heartbeatcount 를 주기적으로 증가 시킴. 다른 무작위 노드에게  목록을 전달. 받은 노드는 값을 갱신. 그중 어떤 멤버 값이 갱신되지 않으면 해당 멤버는 장애!
    
5. 일시적 장애처리
    
    장애상태인 서버로 가는 요청을 다른 서버가 잠시 맡아 처리하는데, 변경 사항은 해당 서버가 복구 되었을때 일괄적으로 데이터를 반영하여 데이터의 일관성을 보존한다. 이 때 임시로 쓰기를 처리한 서버는 그에 관한 힌트를 남겨두는데 이를 위탁기법이라고 부른다.

### 어려웠던 부분

1. KEDA
    
    이벤트 기반 오토스케일러, HPA의 확장판
    
    | 구분 | HPA | KEDA |
    | --- | --- | --- |
    | 기반 | CPU/메모리 메트릭 | **이벤트/외부 메트릭** |
    | 스케일 최솟값 | 1개 이상 | **0개까지** 가능 |
    | 확장성 | 제한적 | 매우 다양한 트리거 |
    - kafka와 연계
        
        ```bash
        외부 이벤트 소스
        (Kafka, SQS, Redis 등)
            ↓
        KEDA Scaler가 감지
            ↓
        HPA에 메트릭 전달
            ↓
        파드 수 자동 조절
        ```
        
    - 파드를 0개까지 스케일링이 가능하다.

1. 객체 스토리지의 분산 처리 개념
- 객체 스토리지는 데이터의 양이 많을때는 우수하지만, 소량의 데이터는 양에 비해 통신 오버헤드가 크기때문이 이 경우에는 블록 형태로 저장한다.
- 객체 스토리지에 파티셔닝을 적용하여 애플리케이션에서 병렬처리를 하면 SSD(블록스토리지)의 IOPS 정도로 성능을 높일 수 있다.
    - SSD, IOPS, 객체 스토리지의 파티셔닝
        
        SSD: 데이터 저장 장치로 자석판을 돌리는 HDD(하드디스크)와 다르게 전기적 신호로만 작동하기때문에 속도가 빠르다.
        
        IOPS: 초당 입출력 작업 횟수
        
        파티셔닝: 객체 스토리지에서는 파일 경로에 따라서 파티셔닝이 분리되어 분리된 경로를 향해 동시에 병렬로 읽기/쓰기 요청을 보냄으로써 IOPS를 확보할 수 있다.
        
- 시계열데이터는 스토리지에 저장할때 데이터를 적당히모아서 한번에 저장, 하지만 너무 크면 전송에 오류가 있을 수 있으니 적당히 분할하여 전송
- 파일 크기 줄이는법: CSV,JSON → 파케이(parquet),ORC 로 변환

## 6. 저번 주차 회고

- APM과 Observability는 어떻게 다른가?
    
    APM(Application Performance Monitoring): 애플리케이션 내부 상태를 모니터링하는 **도구**
    
    Observability: MLT를 조합하여 이전에 겪지 못한 장애를 추적하고 설명할 수 있는 시스템 **능력**
    
- Prometheus exporter와 OpenTelemetry collector는 어떻게 다른가? OpenTelemetry가 Metrics/Logs/Traces를 다룬다면 Prometheus는 왜 쓰는가?
    
    
    |  | Prometheus Exporter | OpenTelemetry collector |
    | --- | --- | --- |
    | 데이터 수집 방식 | Pull 방식 ( 주기적으로 애플리케이션에 데이터를 요청) | Push, Pull 모두 지원 |
    | 범위 | 메트릭 | 메트릭, Logs, Traces |
    | 용도 | 데이터 저장 및 조회 </br> 프로메테우스 저장소에 보내서 저장하는 용도 | 데이터 수집이 표준 </br> 즉, 데이터를 수집해서 전달해주는 역할만 수행 </br> Jaeger나 Tempo에 저장 |
