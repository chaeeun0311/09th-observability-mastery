### 개념 1. 모니터링과 관측가능성은 다르다.

### 개념 설명

> 모니터링은 전체 시스템을 이해하는데 적합하다. 모든것을 수집하기보다는 적은 데이터를 사용해 의미있고 예측이 가능한 결과를 도출해낸다면,
관측가능성은 복잡하고 분산된 시스템 환경에서 발생하는 이벤트에 대한 통찰력을 기반으로 서비스의 정보(context)를 분석한다.
> 

### 중요하다고 생각한 이유

> 모니터링과 Observability의 개념을 확실히 알아야 상황에 맞게 용어를 사용할 수 있을 것 같다.
상태 감지: 모니터링
원인 분석: 관측가능성
> 

관측가능성은 메트릭, 로그, 추적을 통해 장애 원인까지 분석할 수 있는 능력을 의미하고, 모니터링을 통해 무슨 문제가 생겼는지 확인을 하는 활동이다.

```bash
Monitoring ⊂ Observability
모니터링은 관측가능성의 일부이다.
```

### 개념 2. 매트릭과 골든 시그널

### 개념 설명

메트릭은 가용성을 측정하는 관찰 수단이며, SLI를 측정하는 지표이다.

구글의 SRE팀에서는 메트릭을 수집할 때 4가지 골든 시그널에 집중한다.

1. 지연

시계열, 히트맵, 히스토그램 차트로 지연을 시각화한다.

지연은 잠재적인 장애를 예측할 수 있는 신호로, 서비스 요청 후 완료까지 걸리는 시간을 측정한다.

여기서 1초의 지연을 포함하는 정상응답 ≠ 60초의 지연을 포함하는 정상응답

(60초 지연은 에러로 판단하거나 타임아웃으로 처리하는것이 현명하다.)

1. 에러

에러는 문제점과 에러 내용을 직접적으로 출력한다.

에러에는 정상(normal), 대기(pending, 에러 처음 발견 시), 파이어링(firind, 에러가 계속 유지되는 상태)의 세가지 상태가 있다.

1. 트래픽

발생하는 요청의 양

트래칙의 양이 급격히 많아지거나 임계치를 초과하면 지연과 에러가 발생하며 최종적으로 장애를 야기시킴

골든시그널의 원칙: 트래픽이 증가해도 지연과 에러를 최소화하고 장애에 이르지 않게 설계하는것

1. 포화

리소스가 현재 얼마나 채워졌는지, 전체 총대역폭과 처리가능한 트래픽에 대비해 현재 트래픽이 어느정도인지 알 수 있는 지표

### 중요하다고 생각한 이유

> 매트릭은 장애를 수치적으로 표현해줄수있는도구이며 가장 정확하게 파악할 수 있다. 매트릭의 증상들과 봐야할 매트릭들을 알아야 장애에 대응할 수 있다.
>

### 개념 3. 로그

### 개념 설명

시스템이나 애플리케이션에서 발생한 이벤트를 기록한 데이터

분산시스템에서 다양하게 퍼져있는 로그데이터는 로그 데이터 형식을 표준화할 필요가 있다.

로그에 포함할 유용한 정보: **타임스탬프, 식별자, 소스, 레벨 또는 카테고리**가 있다.

```json
{
  "timestamp": "2024-01-15T10:23:45.123Z",
  "level": "error",
  "message": "User login failed",
  "service": "auth-service",
  "version": "1.2.3",
  "environment": "production",
  "trace_id": "abc123",
  "span_id": "def456",
  "user_id": "user_789",
  "error_reason": "invalid_password",
  "attempt_count": 3
}
```

### 중요하다고 생각한 이유

데이터 형식의 표준화와 로그와의 관계:

프론트엔드와 백엔드 주문ID가 다르다면 로그파일을 분석할때 이 둘의 상관관계를 알기 어려워 로그 분석을 한번에 확인할 수 없다.

### 개념 4.  추적

### 개념 설명

여러 시스템 구성 요소를 통과하는 요청 또는 트랜잭션 경로

MSA 환경에서 수많은 서비스들 중 정확히 어느 지점에서 문제가 발생했는지 찾아내기 위해 분산 추적이 필요함. 이 분산 추적의 작은 단위 ‘스팬(Span)’

- 스팬(Span)

예) 사용자의 단일 요청을 처리하는 과정에서 발생하는 모든 개별 작업(API 호출, DB쿼리, 함수 실행 등)

```bash
스팬 정보
1. 작업 이름
2. 시작 및 종료 시간
3. 속성: HTTP 메소드, URL 상태 등
4. 이벤트: 작업 중간에 발생한 이벤트
5. 태그, 로그, 스팬 콘텍스트, 배기지 정보 등
```

대표도구: 예거, 오픈텔레메트리, 템포 등

## 4. 이해가 어려웠던 부분

책을 읽으면서 이해가 어려웠던 개념, 단어, 문장, 구조를 적어주세요.

### 어려웠던 부분

- 추적을 나타내는 Go언어 부분

자동계측으로 잡히지 않는 비즈니스 로직 구간을 직접 코드로 표시할때 사용함.

수동계측의 경우 코드로 직접 스팬을 추가하게됨(Go버전)

```go
func formatString(rootSpan opentracing.Span, helloTo string) string {
    // 부모 스팬과 연결해서 자식 스팬 생성
    span := rootSpan.Tracer().StartSpan(
        "formatString",
        **opentracing.ChildOf(rootSpan.Context())**, // ← 핵심
    )
    defer span.Finish()

    helloStr := fmt.Sprintf("Hello, %s!", helloTo)

    // 이 스팬에 메타데이터 추가
    span.LogFields(
        log.String("event", "string-format"),
        log.String("value", helloStr),
    )
    return helloStr
}
```

### 왜 어려웠는지

책에 있는 Go언어을 처음보았음. 코드로 스팬을 직접 제어하는 경우는 다른 언어 프레임워크에서도 가능하다.

```java
@Service
public class HelloService {

    // 어노테이션을 활용해서 스팬을 지정
    @NewSpan("formatString")
    public String formatString(@SpanTag("hello-to") String helloTo) {
        return "Hello, " + helloTo + "!";
    }

    @NewSpan("printHello")
    public void printHello(String helloStr) {
        System.out.println(helloStr);
    }
}
```


## 5. 추가로 찾아본 내용

### 추가로 찾아본 키워드

- **장애의 기준이 무엇일까? 지연의 기준이 무엇일까? 많은 요청들마다 지연의 기준을 각각다 정해놓는것일까?**

**장애의 기준**

1. **가용성**: 에러율이 x%를 초과하는 경우
2. **지연**: 응답시간이 임계값을 초과
3. 완전중단: 서비스 자체가 응답 없음

- **서비스 신뢰성 SLI, SLO, SLA**

지연의 기준은 보통 **SLO**로 정의한다.

**SLI (Service Level Indicator, 서비스 수준 지표)**
→ 무엇을 측정할까
**예시**
- API 요청 성공률 (성공한 요청 ÷ 전체 요청)
- 응답 시간 (p95 latency ≤ 200ms)
- 오류율 (5xx 응답 ≤ 0.1%)

**SLO (Service Level Objective, 서비스 수준 목표)**
→ 우리가 달성해야하는 목표
SLI 지표에 대해 내부적으로 달성해야 할 목표치
**예시**
- API 성공률 ≥ **99.99%**
- 응답 시간 p95 ≤ **200ms**
    - p95: 상위 95 가200ms라면 전체데이터의 95%는 200ms보다 빨리 처리된것
        - p90: 상위 10%를 제외한, **일반적인 상황**의 성능을 보여준다
        - p95: 상위 5%를 제외한, **조금 더 엄격한 기준**의 성능을 보여준다
        - p99: 상위 1%를 제외한, **매우 까다로운 기준**의 성능을 보여준다
- 에러율 ≤ **0.1%**

```
💡 SLO 대시보드 개발시 주의사항(책 내용)

필수적인 SLO는 에러율, 지연시간, 가용성등이다.

오류예산과 번 레이트를 다양한 윈도(분, 시, 일, 월 등의 주기)에 따라 알람과 연계해 출력해야한다.

프로메테우스 레코딩 규칙과 알람 규칙으로 구현할 수 있고, 이 규칙은 자동 생성 방식을 선호한다.

```

**SLA (Service Level Agreement, 서비스 수준 협약)**
→ 고객에게 보장하는 수준
법적 효력이 있음
**예시**
- 고객과 계약: “우리는 API 가용성을 **99.9% 이상** 보장합니다. 이를 지키지 못하면 서비스 크레딧 지급.”

<br/>

- 스팬을 많이 저장하면 오버헤드가 클까?

분산트레이싱에서는 하나의 요청에도 부모-자식등의 많은 스팬이 생긴다. 서비스 규모가 커질수록 생성되는 **스팬의 수는 기하급수적으로 증가**하며, 각 스팬은 스토리지 공간을 필요로 한다.

대규모 MSA 환경에서는 하루에 수억개의 스팬이 생성되기도 한다. 따라서 **스팬 관리는 인프라 비용 관리의 핵심요소**가 되었다.

스팩 폭주의 주요원인

1. 불필요한 스팬 생성
    디버그 모드나 상세 로깅 설정이 기본으로 활성화된 경우 모든 내부 호출이 스팬에 기록됨
    
2. 중복 스팬 생성
    여러 계층의 라이브러리가 독립적으로 트레이싱을 활성화하면 동일한 작업에 대해 중복으로 생성됨
    
3. 과도한 속성추가
    각 스팬에 불필요한 메타데이터 추가
    
4. 서비스 Mesh의 도입으로 트레이싱 범위 확대
    사이드카 프록시로 인해 네트워크 레벨의 스팬이 추가적으로 생성되는경우
    

스팬 최적화 방법

1. 불필요한 스팬 제거
    디테일한 디버그 스팬은 비활성화하고 주요 비즈니스 트랜잭션에 집중
    
2. 샘플링 전략
    모든 요청을 다 추적하면 비용이 많이 들기 때문에 어떤 요청을 추적할지 선별하는 작업
    요청의 5%는 랜덤하게 추적을 진행(**Head-based 샘플링**)
    나머지는 요청이 완전히 끝나고 문제가 있으면 저장/ 아니면 버림 (**Tail-based 샘플링**)
    
3. 속성 최적화
    필수속성인 이름, 작업 유형, 응답 상태코드는 기본적으로 포함시키고 메타데이터는 필요시에 동적으로 추가
    특히, 대량의 사용자 데이터나, 긴 문자열은 피하고, 필요한 경우 처리된 식별자만 저장하거나 별도의 로깅 시스템으로 분리한다.

## 6. 내 생각, 경험, 질문

### 자유정리

> 관측가능성과 모니터링의 개념을 거의 동일하다고 생각했는데 차이가 있다는점이 인상 깊었습니다.
> 그리고 메트릭과 로그는 알고있었지만 트레이싱(추적)의 개념을 처음 알게 되었습니다. 대규모 시스템에서 추적과 스팬의 개념은 장애를 빠르게 파악하는데
> 주요 지표가 될수있음을 알게 되었고, 책에서 나온 트레이싱 예시들을 보며 직접 해보고 추적을 구현하고 오픈텔레메트리나 예거를 사용해 시각화해보면 도움이 많이 될 것 같습니다.
> 또한 관측가능성의 큰 흐름을 보며 앞으로 배울 컬렉터나 데이터 저장, 그리고 예측과 분석까지 배우기 위한 관측가능성의 필수 개념으로 발판을 다지는 시간이 되었습니다.
