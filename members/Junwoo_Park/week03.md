## 1. 이번 주 전체 흐름

이번 주 내용은 관측 가능성(Observability)을 가능하게 하는 주변 기반 기술들을 다룬다.

관측 가능성은 메트릭, 로그, 트레이스를 수집하는 것에서 끝나지 않는다. 실제 운영 환경에서는 트래픽을 어떻게 분산할지, 장애가 났을 때 어떻게 격리할지, 부하가 늘어날 때 어떻게 확장할지, 쌓이는 데이터를 어떻게 저장하고 조회할지도 함께 중요해진다.

크게 보면 이번 주 내용은 다음 흐름으로 이해했다.

1. 트래픽을 안정적으로 분산한다.
2. 장애가 전체 시스템으로 번지지 않게 막는다.
3. 서비스 간 통신을 관찰하고 제어한다.
4. Kubernetes 환경에서 부하에 따라 자동 확장한다.
5. 대량의 관측 가능성 데이터를 분산 저장하고 빠르게 조회한다.

## 2. 트래픽 관리와 복원성

### 멀티 테넌트와 멀티 테넌시

- 멀티 테넌트(Multi-tenant): 하나의 시스템이나 서비스를 여러 사용자 또는 조직이 함께 사용하는 구조
- 멀티 테넌시(Multi-tenancy): 멀티 테넌트 환경을 가능하게 하는 아키텍처 또는 운영 방식

관측 가능성 서비스는 여러 애플리케이션이나 조직의 데이터를 함께 받을 수 있으므로 멀티 테넌시를 고려해야 한다. 테넌트별 데이터 격리, 권한 관리, 저장소 분리, 사용량 제한 같은 부분이 중요해질 수 있다.

### SPOF

- SPOF(Single Point of Failure): 해당 지점이 실패하면 전체 시스템에 영향을 주는 단일 장애 지점

관측 가능성 시스템이 SPOF가 되면 장애 상황에서 오히려 장애를 분석할 수 없게 된다. 그래서 수집기, 저장소, 대시보드, 알림 시스템까지 고가용성을 고려해야 한다.

### 관측 가능성 서비스

- 관측 가능성 서비스: 비즈니스 서비스를 지원하기 위해 메트릭, 로그, 트레이스 등을 수집하고 분석하는 서비스

관측 가능성 서비스는 보조 시스템처럼 보이지만 실제로는 비즈니스 서비스보다 더 많은 트랜잭션과 데이터를 처리할 수 있다. 모든 서비스가 계속 데이터를 보내기 때문에 자체 확장성과 안정성이 중요하다.

### 로드 밸런서

- 로드 밸런서(Load Balancer): 여러 서버나 서비스 인스턴스에 트래픽을 분산해 가용성과 처리량을 높이는 구성 요소

로드 밸런서는 한 인스턴스에 요청이 몰리는 것을 막고, 장애가 난 인스턴스로 트래픽이 계속 들어가지 않도록 도와준다.

### 로드밸런싱 방식

- 플랫폼 로드밸런싱: AWS ELB, Nginx, IIS, Apache처럼 플랫폼 또는 인프라 계층에서 트래픽을 분산하는 방식
- 게이트웨이 로드밸런싱: API Gateway, Spring Cloud Gateway, Netflix Zuul처럼 게이트웨이 계층에서 요청을 라우팅하고 분산하는 방식
- 클라이언트 측 부하 분산: 호출 주체가 서비스 인스턴스 목록을 보고 직접 호출 대상을 선택하는 방식
- 디스커버리 서비스: Eureka, Consul처럼 서비스 인스턴스 목록과 위치 정보를 제공하는 시스템

로드밸런싱은 한 계층에서만 하는 것이 아니라, 시스템 구조에 따라 여러 계층에서 나누어 담당할 수 있다. 외부 요청은 플랫폼이나 게이트웨이에서 받고, 내부 서비스 호출은 클라이언트 측 부하 분산이나 서비스 메시가 담당할 수 있다.

### 트래픽 제어

- 트래픽 제어: 애플리케이션 코드, 로드밸런서, 게이트웨이, 클라이언트 로드밸런싱, 서비스 메시 등을 통해 요청 흐름을 조절하는 방법

트래픽 제어는 요청을 단순히 많이 받는 것보다, 장애 상황에서 어디로 보내고 어디를 막을지 결정하는 역할에 가깝다.

### 다운스트림 서비스

- 다운스트림 서비스(Downstream Service): 현재 서비스가 요청을 전달하거나 의존하는 하위 서비스 또는 외부 서비스

현재 서비스가 정상이어도 다운스트림 서비스가 느리거나 실패하면 전체 요청이 지연될 수 있다. 그래서 다운스트림 호출에는 타임아웃, 재시도, 격리 전략이 필요하다.

### 복원성 패턴

- 복원성 패턴(Resilience Pattern): 장애가 발생해도 전체 시스템으로 장애가 전파되지 않도록 타임아웃, 재시도, 벌크헤드, 서킷 브레이커, 폴백 등을 적용하는 설계 방식
- 벌크헤드 패턴(Bulkhead Pattern): 다운스트림 서비스를 격리하고 동시 처리량을 제한해 한 서비스의 장애가 전체로 번지는 것을 막는 패턴
- 서킷 브레이커(Circuit Breaker): 특정 서비스 호출 실패가 지속될 때 호출을 차단해 장애 전파와 자원 낭비를 막는 패턴
- 폴백(Fallback): 기본 처리 경로가 실패했을 때 사전에 정한 대체 응답이나 로직을 반환하는 방식

복원성 패턴은 장애를 완전히 없애는 방법이라기보다는 장애의 영향을 줄이는 방법이다. 특히 마이크로서비스 환경에서는 한 서비스의 문제가 여러 서비스로 번지기 쉬워서 중요하다.

## 3. 가시성과 서비스 메시

### 가시성과 관측 가능성

- 가시성(Visibility): 시스템이나 네트워크 상태를 외부에서 확인할 수 있게 만드는 능력
- 관측 가능성(Observability): 메트릭, 로그, 트레이스를 통해 시스템 내부 상태와 장애 원인을 추론하는 능력

가시성은 "보이는가"에 가깝고, 관측 가능성은 "왜 그런 일이 생겼는지 판단할 수 있는가"에 가깝다.

### 서비스 메시

- 서비스 메시(Service Mesh): 서비스 간 통신을 프록시나 사이드카 계층에서 제어해 트래픽 관리, 보안, 관측 가능성을 제공하는 인프라 계층
- 사이드카(Sidecar): 애플리케이션 컨테이너 옆에서 프록시나 보조 기능을 수행하는 별도 컨테이너 패턴

서비스 메시를 사용하면 애플리케이션 코드에 직접 넣지 않아도 서비스 간 통신 제어, 보안, 관측 가능성 기능을 공통으로 적용할 수 있다. 다만 프록시가 추가되므로 리소스 사용량과 운영 복잡도도 함께 늘어난다.

### Istio와 Kiali

- Istio: 서비스 메시 구현체로 트래픽 관리, 보안, 관측 가능성 기능을 제공하는 도구
- Kiali: Istio 서비스 메시의 트래픽 흐름과 서비스 간 관계를 시각화하는 도구

Istio는 서비스 간 통신 정책을 제어하고, Kiali는 그 통신 흐름을 눈으로 확인할 수 있게 해준다. 마이크로서비스가 많아질수록 서비스 간 관계를 시각적으로 보는 것이 중요해진다.

### eBPF와 Cilium

- eBPF: 커널 수준에서 네트워크와 시스템 동작을 관찰하고 제어할 수 있게 하는 기술
- Cilium: eBPF 기반의 네트워크, 보안, 관측 가능성 기능을 제공하는 Kubernetes 네트워킹 도구

eBPF는 애플리케이션보다 낮은 커널 수준에서 정보를 볼 수 있다는 점이 특징이다. Cilium은 이 기술을 활용해 Kubernetes 네트워크와 보안, 관측 가능성 기능을 제공한다.

## 4. Kubernetes 오토스케일링

### HPA

- HPA(Horizontal Pod Autoscaler): CPU, 메모리, 사용자 정의 메트릭 등을 기준으로 Pod replica 수를 자동 조정하는 Kubernetes 기능

HPA는 부하가 늘면 Pod 수를 늘리고, 부하가 줄면 Pod 수를 줄인다. 단순히 서버를 많이 띄워두는 방식보다 비용과 안정성 사이의 균형을 잡는 데 도움이 된다.

### KEDA

- KEDA(Kubernetes Event-driven Autoscaling): 이벤트 소스를 기준으로 Kubernetes 워크로드를 확장하거나 축소하는 오토스케일링 도구

KEDA는 CPU나 메모리뿐 아니라 큐 길이, 메시지 수, 외부 이벤트 같은 기준으로도 스케일링할 수 있게 해준다.

### 메트릭 관련 구성 요소

- 표준 메트릭: CPU, 메모리처럼 기본 리소스 사용량을 나타내는 메트릭
- 사용자 정의 메트릭: Prometheus Adapter 등을 통해 애플리케이션이나 비즈니스 특성에 맞게 HPA가 사용할 수 있는 메트릭
- Metrics Server: kubelet에서 리소스 메트릭을 수집해 Kubernetes Metrics API로 제공하는 컴포넌트
- kubelet: 각 Kubernetes 노드에서 Pod와 컨테이너 상태를 관리하는 에이전트
- Prometheus Adapter: Prometheus의 메트릭을 Kubernetes HPA가 사용할 수 있는 사용자 정의 메트릭으로 변환하는 어댑터

CPU, 메모리 같은 표준 메트릭만으로는 실제 서비스 부하를 정확히 설명하기 어려울 수 있다. 요청 수, 큐 길이, 처리 지연 시간 같은 사용자 정의 메트릭을 활용하면 더 서비스에 맞는 스케일링이 가능하다.

### 스케일링 관련 용어

- 스케일아웃(Scale-out): 인스턴스 또는 Pod 수를 늘리는 방식
- 스케일인(Scale-in): 인스턴스 또는 Pod 수를 줄이는 방식
- 스래싱(Thrashing): 작은 부하 변화에 따라 스케일링이 반복적으로 발생해 시스템이 불안정해지는 현상
- 지연 반응: 스케일링 판단에 의도적으로 지연을 두어 잦은 변동과 스래싱을 줄이는 방식

스케일링은 빠르게 반응하는 것도 중요하지만 너무 민감하면 오히려 불안정해질 수 있다. 그래서 안정화 구간이나 지연 반응을 두는 설계가 필요하다.

## 5. 관측 가능성 운영 프로세스

### 일반 운영 프로세스

- 일반 운영 프로세스: 메트릭, 트레이스, 로그를 순서대로 분석하며 서비스 상태를 확인하는 흐름

평상시에는 메트릭으로 전체 상태를 보고, 이상한 흐름이 보이면 트레이스와 로그로 더 자세히 들어가는 방식이 자연스럽다.

### 장애 발생 운영 프로세스

- 장애 발생 운영 프로세스: 알람 발생 후 트레이스와 로그를 통해 장애 원인을 좁혀가는 흐름

장애가 발생하면 알람을 시작점으로 삼고, 어떤 서비스에서 지연이나 오류가 시작됐는지 트레이스와 로그로 좁혀가야 한다.

### AIOps

- AIOps: 이상 탐지, 자동 복구, 알림, 운영자 개입을 자동화하거나 지능화하는 운영 방식

AIOps는 운영자가 모든 신호를 직접 확인하지 않아도 이상 징후를 찾고 대응을 자동화하는 방향이다. 다만 자동화가 잘 동작하려면 기본 메트릭과 로그, 트레이스 품질이 먼저 좋아야 한다.

## 6. 데이터 확장과 저장 구조

### 수평 샤딩

- 수평 샤딩(Horizontal Sharding): 데이터를 여러 샤드로 나누어 저장해 데이터베이스를 수평 확장하는 기술
- 샤드(Shard): 전체 데이터의 일부를 담당하는 분할 저장 단위
- 샤딩 키 또는 파티션 키: 데이터를 어느 샤드에 저장할지 결정하는 기준 값
- 스큐 현상(Skew): 데이터나 부하가 특정 샤드에 편향되는 현상
- 리밸런싱(Rebalancing): 샤드 간 데이터와 부하를 다시 균등하게 분배하는 작업
- 핫스팟 키(Hotspot Key): 특정 키에 요청이 집중되어 특정 샤드나 서버에 부하가 몰리는 문제
- 유명인사 문제: 특정 사용자나 객체에 트래픽이 집중되어 데이터 분산이 깨지는 핫스팟 문제

샤딩은 데이터를 나누어 저장하는 방법이지만, 키를 잘못 잡으면 특정 샤드만 바빠질 수 있다. 그래서 샤딩 키 설계가 중요하다.

### 비정규화와 객체 스토리지

- 비정규화(Denormalization): 조인 비용을 줄이기 위해 데이터를 중복 저장하거나 조회 중심으로 구조를 바꾸는 방식
- 객체 스토리지(Object Storage): 대량의 비정형 데이터나 파일 데이터를 객체 단위로 저장하는 스토리지 방식

관측 가능성 데이터는 조회 속도와 저장 비용이 중요하기 때문에 정규화된 관계형 구조만 고집하기 어렵다. 로그나 장기 보관 데이터는 객체 스토리지에 저장하는 방식도 사용할 수 있다.

### 마이크로서비스와 데이터 접근

- 마이크로서비스(Microservice): 하나의 시스템을 독립 배포 가능한 작은 서비스들로 나누는 아키텍처
- CQRS(Command Query Responsibility Segregation): 명령과 조회의 책임을 분리하는 패턴
- 읽기/쓰기 분리: 조회 부하와 변경 부하를 분리해 성능, 확장성, 데이터 모델을 각각 최적화하는 방식

마이크로서비스에서는 서비스별로 데이터 소유권이 나뉘기 때문에 조회와 쓰기 모델을 분리하는 방식이 자주 언급된다. CQRS는 이런 문제를 해결하는 패턴 중 하나다.

### 일관된 해시

- 일관된 해시(Consistent Hashing): 서버 수가 바뀌어도 전체 키가 아니라 일부 키만 재배치되도록 하는 해싱 방식
- 해시 링(Hash Ring): 해시 공간을 원형으로 배치해 키와 서버를 매핑하는 구조
- 해시 키(Hash Key): 해시 함수를 적용해 저장 위치를 결정하는 기준 값
- 해싱(Hashing): 키를 해시 함수에 넣어 나온 결과를 주소처럼 사용해 값을 찾는 방법

일관된 해시는 노드가 추가되거나 제거될 때 전체 데이터를 다시 배치하지 않아도 된다는 점에서 분산 저장소에 유용하다.

### 키-값 저장소

- 키-값 저장소(Key-Value Store): 키를 기준으로 값을 저장하고 조회하는 비관계형 데이터베이스
- DynamoDB: AWS에서 제공하는 관리형 키-값 및 문서 데이터베이스
- Redis: 인메모리 기반 키-값 저장소로 캐시, 세션, 큐 등에 사용되는 데이터 저장소
- Consul: 서비스 디스커버리와 키-값 저장 기능을 제공하는 도구
- etcd: Kubernetes 등에서 설정과 상태 정보를 저장하는 분산 키-값 저장소
- Memberlist: 분산 시스템에서 노드 멤버십과 장애 감지를 구현하는 라이브러리

키-값 저장소는 구조가 단순하고 빠른 조회에 유리하다. Kubernetes의 etcd처럼 시스템 상태를 저장하는 데 쓰이기도 하고, Redis처럼 캐시나 세션 저장소로 많이 쓰이기도 한다.

### 안정적인 데이터 관리

- 데이터 파티션: 데이터를 여러 구간이나 노드로 나누어 저장하는 방식
- 데이터 다중화: 장애에 대비해 데이터를 여러 위치에 복제하는 방식
- 데이터 일관성: 여러 복제본이나 노드가 같은 데이터 상태를 유지하는 성질
- 장애 감지: 노드나 서비스의 비정상 상태를 탐지하는 과정
- 일시적 장애 처리: 짧은 장애나 네트워크 흔들림을 재시도, 타임아웃, 복구 전략으로 흡수하는 방식

분산 시스템에서는 데이터를 나누는 것만큼 복제와 일관성도 중요하다. 장애가 발생했을 때 어느 노드가 실패했는지 감지하고, 일시적인 문제인지 실제 장애인지 구분하는 것도 필요하다.

### 시계열 데이터 집계

- 시계열 데이터(Time Series Data): 시간 순서대로 기록되는 메트릭, 로그, 이벤트 데이터
- 시계열 인덱스: 시간 기준으로 데이터를 빠르게 조회하고 집계하기 위한 인덱스 구조
- 열 지향 스토리지(Column-oriented Storage): 행이 아니라 열 단위로 데이터를 저장해 대량 집계와 분석에 유리한 저장 방식
- 시계열 테이블: 시간을 기준으로 분할하거나 파티셔닝한 테이블
- CSV 객체 스토리지: CSV 같은 파일 데이터를 객체 스토리지에 저장하는 형태
- 작은 파일 문제: 객체 스토리지에 너무 많은 작은 파일이 쌓여 쿼리 성능이 악화되는 문제

관측 가능성 데이터는 대부분 시간 기준으로 쌓인다. 그래서 시간 범위 조회, 집계, 보관 기간, 압축, 파티셔닝을 함께 생각해야 한다. 객체 스토리지에 파일을 저장할 때는 파일이 너무 잘게 쪼개지지 않도록 주의해야 한다.

## 7. 내 생각

이번 주 내용은 용어가 많아서 처음에는 각각 따로 외워야 하는 개념처럼 느껴졌다. 그런데 정리해보니 결국 하나의 흐름으로 이어졌다. 트래픽을 잘 분산하고, 장애를 격리하고, 부하에 맞춰 확장하고, 쌓이는 데이터를 잘 저장해야 관측 가능성도 제대로 동작한다.

특히 관측 가능성 시스템이 오히려 SPOF가 될 수 있다는 점이 인상적이었다. 장애를 보기 위한 시스템이 장애 상황에서 같이 무너지면 안 되기 때문에, 관측 가능성 도구 자체도 운영 대상이라고 봐야 할 것 같다.

궁금한 점은 실제 서비스에서는 서킷 브레이커나 폴백을 애플리케이션 코드에서 주로 처리하는지, 아니면 게이트웨이나 서비스 메시 계층에서 처리하는지다. 또 HPA 기준으로 CPU와 메모리 외에 어떤 메트릭을 많이 쓰는지도 궁금하다.

## 8. 발표할 때 말할 내용

이번 주는 개념이 많아서 큰 흐름만 잡고 발표하면 좋을 것 같다.

1. 관측 가능성은 데이터 수집만이 아니라 트래픽, 복원성, 확장, 저장 구조까지 함께 봐야 한다.
2. 장애가 났을 때 전체로 번지지 않게 하려면 벌크헤드, 서킷 브레이커, 폴백 같은 복원성 패턴이 필요하다.
3. Kubernetes에서는 HPA, KEDA, Metrics Server, Prometheus Adapter가 오토스케일링 흐름과 연결된다.
4. 관측 가능성 데이터는 양이 많기 때문에 샤딩, 일관된 해시, 시계열 저장 구조 같은 데이터 확장 방식도 중요하다.
