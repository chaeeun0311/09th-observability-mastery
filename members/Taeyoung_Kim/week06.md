### Chapter 5에서 실습에 참고해보고 싶은 데모/흐름 1개

**`메트릭에서 추적(metrics → traces)`**

- 메트릭과 추적 간 상관관계는 표준인 이그젬플러, 메트릭 생성기 등을 활용해서 구현
- 이그젬플러를 활용하면 메트릭 이상 징후를 관련 trace와 연결 가능
- 이그젬플러
    - 주어진 시간 간격에서 측정된 추적의 샘플링
    - 시계열 데이터 내의 특정 이벤트에서 높은 카디널리티를 가지는 메타데이터를 식별하는데 도움이 됨
- 메트릭
    - 시스템 전체의 상태를 집계된 형태로 보여주는 데 유용
- 추적
    - 하나의 요청이 여러 서비스나 컴포넌트를 거쳐 처리되는 과정을 세부적으로 보는데 유용
- 따라서, 이그젬플러를 통해 메트릭과 추적을 연결해야 함
    - 이그젬플러는 측정된 시간 내에서 높은 대기시간을 나타내는 거래를 추적하고, 문제가 있는 데이터 포인트를 식별하는데 도움을 줌 → 지연시간 문제를 추적과 연계하여 근본 원인 분석 수행해 빠르게 성능 문제 해결 가능

### 해당 내용을 왜 골랐는지

- 실제 서비스 모니터링과 장애 분석 과정에서 가장 기본적이고 중요한 방식이라고 생각했기 때문
- 메트릭
    - 시스템 전체의 상태를 한눈에 보여줌 → 문제 발생 빠르게 확인하는데 유용
        - ex. latency 증가, error rate 증가 상황 등등
    - 개별 요청이 어떤 경로에서 지연되었는지까지 자세한 확인이 불가
- 트레이스
    - 하나의 요청이 여러 서비스나 컴포넌트를 거쳐 처리되는 과정 확인 가능
    
     →  특정 요청에서 어느 span이 오래 걸렸는지, 어떤 서비스 호출구간이 병목이 되었는지 분석 가능
    
- 메트릭 → 추적
    - 문제가 있다는 사실 확인에서 끝이 아님
    - 문제의 원인을 단계적으로 찾아가는 관측 가능성의 핵심 흐름이라고 생각하여 선택

### 실습하면서 확인해보고 싶은 점이나 질문 1개

- 메트릭에서 발견한 이상 징후를 trace로 얼마나 자연스럽게 연결해서 분석할 수 있는지
    - ex. 
        - 특정 시간대에 latency가 증가하거나 error rate가 상승했을 때, Grafana에서 해당 구간과 관련된 trace를 바로 확인할 수 있는지?
        - 또한 trace를 통해 어떤 요청이 느렸는지, 그리고 어느 span이나 서비스 호출에서 가장 많은 시간이 소요되었는지?
        - 메트릭과 trace가 연결되었을 때 단순히 대시보드를 보는 것보다 장애 원인 분석이 얼마나 더 직관적으로 이루어지는지?
