### `로드밸런싱의 3가지 유형`

1. 플랫폼 로드 밸런싱
    - 대표 예시 : AWS ELB, Nginx
    - 4계층은 NLB, 7계층은 ALB
    - 온프레미스 환경에서는 정적인 로드밸런싱 기법이 널리쓰임
2. 게이트웨이 로드 밸런싱
    - 기본적인 라우팅 기능을 포함해서 비율 제한 A/B테스트, 카나리배포 제공, 인그레스 기능 제공
    - 대표 예시 : Netflix Zuul, Spring Cloud Gateway
3. 클라이언트 측 로드 밸런싱
    - 호출 주체가 로드 밸런싱 결정권을 가짐
    - 대표 예시 : Netflix Eureka, Istio

### `복원력을 높이는 방법`

- 애플리케이션 관점 : 소스 내 예외 처리와 재시도, 타임아웃 등을 정의
- 인프라 관점 : 서비스 메시와 메시징 사용

### `복원력을 높일 수 있는 4가지 패턴`

1. 재시도
- 가끔 발생하는 장애에 대응하는 효율적인 수단
- 하지만, 재시도 압력이 누적돼 시스템 과부하를 일으키면 안되므로 반복적이고 무차별적인 재시도 주의
- 재시도의 당위성과 서비스에 대한 도메인 지식이 있어야 재시도에 대한 판단 가능
- 최대 재시도 횟수와 시간 간격은 재시도 정책에 정의되어야 함
2. 비율 제한
- 비정상적인 상황에서 제한된 처리량이라도 요청에 대응하도록 서비스 지속시킬 수 있음
- Istio도 가능하지만 다양한 Ingress와 게이트웨이 로드밸런싱을 활용하는 것을 권장
    - why? ) Istio에서도 rate limiting은 가능하지만, 비율 제한은 과도한 요청이 내부 서비스까지 전달되기 전에 시스템 진입점에서 먼저 처리하는 것이 효율적이기 때문
    - 따라서 외부 트래픽에 대한 1차 제한은 Ingress나 API Gateway 계층에서 수행하고, Istio는 내부 서비스 간 트래픽 제어와 세부 정책 적용에 활용하는 것이 일반적으로 권장된다.
3. 벌크헤드
- 다운스트림 서비스를 서로 격리하고 각 서비스의 동시 처리 능력을 제한
- 일부 컴포넌트에 문제가 발생하더라도, 그것이 시스템 전체로의 장애로 전파되지 않게끔 만들어주는 패턴
4. 서킷 브레이커
- 벌크헤드 패턴을 확장한 개념
- 다수의 마이크로서비스는 분산되고 서로 호출하는 관계를 가지는데, 이때 마이크로서비스간 장애로 전파되지 않게끔 만들어주는 패턴

### `대표적인 기본 메트릭`

1. 요청 수
- 애플리케이션을 오토 스케일링 해야하는 지 판단하는 지표로 사용
- 총 요청수의 경우 부하 테스트에서 주목해야할 수치
2. 요청 기간
- 히스토그램에서 가장 잘 나타나며, 시각화를 통해 요청에 대한 분포를 이해할 수 있음
3. 동시 요청
- 비즈니스 요구사항이 변경됨에 따라 어떤 리소스가 필요할지 결정하는데 도움을 줌
4. 오류율
- 전체 요청 중 실패한 요청의 비율
- HTTP 기준으로는 4xx, 5xx 비율을 볼 수 있음
- 특히 5xx 오류율은 서버 내부 문제나 과부하를 판단하는 데 중요함
5. 응답 지연 시간
- p95 latency = 전체 요청 중 95%가 이 시간 안에 응답됨
- p99 latency = 전체 요청 중 99%가 이 시간 안에 응답됨
6. 처리량
- 초당 처리 가능한 요청 수
- 보통 RPS 또는 TPS로 표현함
    - RPS (Requests Per Second) : 초당 요청 수 (요청 단위)
    - TPS (Transactions Per Second) : 초당 트랜잭션 수 (비즈니스 처리 단위)
7. CPU 사용률
- 애플리케이션 인스턴스나 Pod가 CPU를 얼마나 사용하는지 나타내는 지표
- 오토스케일링에서 가장 자주 사용되는 기본 메트릭 중 하나
8. 메모리 사용률
- 애플리케이션이 사용하는 메모리 양
- 메모리 누수, 캐시 증가, OOM Kill 가능성을 판단하는 데 중요함

### `샤딩`

- 데이터베이스의 수평적 확장 (샤드로 분할)
- 이를 통해 시스템의 확장, 성능 향상, 높은 안정성을 갖춘 클러스터 구성
- 모든 샤드는 같은 스키마를 쓰고, 샤드에 보관되는 데이터는 중복이 없음

### `샤딩 전략 구현시 고려사항`

- 샤딩 키(파티션 키)를 어떻게 정하는가
- 만약 샤딩 키(파티션 키)가 잘못될 경우 발생할 수 있는 문제
    1. 데이터가 한쪽으로 편향되는 스큐가 발생할 수 있음
    2. 비효율적인 병렬 처리 → 처리 과정에서 성능 저하 및 비용 낭비 초래
    3. 불필요한 리밸런싱 작업 수행 → 시스템 불안정

### `샤딩 관련 다양한 문제`

1. 데이터의 재분배
- 샤드간 데이터 분포가 불균등하여 특정 샤드에 할당된 공간 소모가 다른 샤드에 비해 빠르게 진행될때 샤드 소진 현상 발생
- 이 때, 데이터 재배치를 위해 안정 해시 기법을 사용하여 해결 가능
- 노드의 증감에 따라 샤드는 영향을 받고 항상 변동 발생하는데, 이로 인해 샤드간 데이터 재분배 발생 (리밸런싱)
2. 유명 인사 문제 (핫스팟 키 문제)
- 특정 샤드에 질의가 집중되어 서버에 과부하가 걸리는 문제
- 조회수가 좋은 유명 인사가 전부 같은 샤드에 저장된 데이터베이스의 경우, 읽기 연산으로 인한 과부하 발생
3. 조인과 비정규화
- 하나의 데이터베이스를 여러 샤드 서버로 쪼개고나면  여러 샤드에 걸친 데이터를 조인하기 힘들어짐
- 이를 해결하기 위해, 데이터베이스를 비정규화하여 하나의 테이블에서 질의가 수행될 수 있도록 함

### `CQRS패턴`

- 읽기를 위한 데이터 모델과 쓰기를 위한 데이터 모델을 분리한 패턴
- 이점
    1. 독립적인 크기 조정
    2. 최적화된 데이터 스키마
    3. 관심사의 분리
    4. 보안 강화
    5. 단순한 쿼리
