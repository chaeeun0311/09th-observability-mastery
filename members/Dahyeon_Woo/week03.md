## 2. 책 내용 핵심 요약

2.1.1 트래픽 관리: 네트워크는 SPOF(single point of failure, 단일 장애점)의 특징을 갖고 있어 네트워크 장애가 생기면 많은 손해를 끼친다. 강력한 네트워크 관리 체계를 구축하고 정확한 원인 분석과 사전에 이상 현상을 탐지하고 예측하는 방안이 필요하다.

**2.1.2 로드밸런서: 플랫폼 로드밸런싱, 게이트웨이 로드밸런싱, 클라이언트 측 부하 분산**

**2.1.3. 복원성 패턴: 서비스 매시를 이용하여 네트워크 구간의 모니터링과 복원성 패턴을 구현 가능. 복원력을 높이는 네 가지 패턴: 재시도, 비율 제한, 벌크헤드, 서킷브레이커**

2.1.4. 가시성: 복잡한 마이크로서비스 내부에서 일어나는 통신과 상태를 실시간으로 모니터링하여 투명하게 들여다볼 수 있는 능력이다.

2.1.5. 서비스 매시: 이스티오(Istio) 등과 같이 애플리케이션 코드의 수정 없이 프록시를 통해 서비스 간 통신, 보안, 모니터링, 제어를 인프라 레이어에서 전담하는 기술이다.

2.2 쿠버네티스 오토스케일링: 리소스 사용량(CPU, Memory) 및 사용자 정의 메트릭에 따라 팟(Pod)의 개수를 동적으로 조절하는 HPA(Horizontal Pod Autoscaler) 등의 자동 확장 매커니즘이다.

**2.3.관측가능성 운영 프로세스**

2.3.1 운영 프로세스: SLI, SLO, 로그, 메트릭, 추적
2.3.2 장애 프로세스: 장애 발생, 알림 전송, 추적 분석, 로그 분석

2.4 수평 샤딩: 데이터베이스 서버를 여러 개로 쪼개어 데이터를 분산 저장함으로써 단일 DB의 용량 및 성능 한계를 극복하는 아키텍처이다.

2.5 마이크로서비스: 하나의 거대한 애플리케이션을 독립적으로 배포 및 운영이 가능한 작은 서비스 단위들로 쪼개어 비즈니스 민첩성을 극대화하는 설계 방식이다.

2.6 일관된 해시: 노드가 추가되거나 삭제될 때 오직 `K/N` 개의 키만 재배치하면 되도록 설계되어, 분산 캐시나 저장소의 확장성을 높이는 해싱 알고리즘이다.

2.7 관측 가능성 시각화: 히스토그램, 히트맵

2.8 키-값 저장소: NoSQL의 일종으로 고성능 읽기/쓰기를 지원하기 위해 데이터를 Key와 Value의 형태로 매핑하여 저장하는 고속 스토리지이다.

2.9 객체 스토리지: 파일 계층 구조 없이 비정형 데이터(이미지, 영상, 로그 파일 등)를 ID 기반으로 대용량 저장 및 관리하는 클라우드 지향형 스토리지이다.

2.10 안정적 데이터 관리: 데이터 파티션, 데이터 다중화, 데이터 일관성, 장애 감지, 일시적 장애 처리 등을 포함하여 분산 환경에서 데이터 훼손 없이 연속성을 보장하는 설계 기법이다.

2.11 시계열 데이터 집계: 대량으로 발생하는 메트릭 데이터를 효율적으로 저장 및 조회하기 위해 파티션을 생성하고 해당 날짜에 해당하는 파일을 폴더에 추가하는 방식이다.

## 3. 중요하다고 생각한 개념


### 개념1: 로드밸런서 -  플랫폼 로드밸런싱, 게이트웨이 로드밸런싱, 클라이언트 측 부하 분산

### 1. 개념 설명
- 대규모 트래픽을 여러 서버나 인스턴스로 분산하여 시스템의 가용성과 성능을 최적화하는 기술입니다. 분산 메커니즘의 위치와 역할에 따라 세 가지로 분류됩니다.

- 플랫폼 로드밸런싱: 인프라 전면(L4/L7 스위치, 클라우드 ALB 등)에서 외부 트래픽을 1차적으로 받아 내부 서버들로 나누어주는 정적/동적 부하 분산입니다.

- 게이트웨이 로드밸런싱: 마이크로서비스 아키텍처(MSA)의 진입점인 API 게이트웨이가 라우팅 규칙, 인증, 인가와 함께 내부 서비스 인스턴스들로 트래픽을 분산하는 방식입니다.

- 클라이언트 측 부하 분산: 서버나 하드웨어가 아닌, 서비스를 호출하는 클라이언트(또는 호출 측 애플리케이션 서비스)가 서비스 디스커버리를 통해 가용한 서버 목록을 직접 조회하고 self로 부하를 분산(예: Netflix Ribbon)하는 방식입니다.

### 2. 중요하다고 생각한 이유
- 가용성 확보와 단일 장애점(SPOF) 제거: 특정 서버가 다운되더라도 로드밸런서가 이를 감지(Health Check)하고 정상 서버로만 트래픽을 보내므로 서비스가 중단되지 않습니다.

- 아키텍처 유연성: 현대의 MSA 환경에서는 서비스 간 통신(East-West 트래픽)이 폭발적으로 증가합니다. 이때 모든 트래픽을 중앙 하드웨어 로드밸런서로만 처리하면 병목이 생기므로, 게이트웨이나 클라이언트 측 분산 같은 계층화된 부하 분산 전략이 필수적입니다.

### 3. 운영 관점에서의 의미
- 무중단 배포(Zero-downtime Deployment): Blue/Green 배포나 카나리(Canary) 배포 시, 로드밸런서의 가중치 조절이나 타겟 그룹 변경을 통해 사용자 유실 없는 안전한 릴리즈를 수행할 수 있습니다.

- 동적 스케일링과의 연동: 오토스케일링(Auto-scaling)에 의해 서버가 자동으로 늘어나거나 줄어들 때, 로드밸런서 타겟 그룹에 자동으로 등록/해제(Deregistration delay 관리 필요)되는 동적 제어가 운영 자동화의 핵심입니다.

## 2.1.3 복원성 패턴: 재시도, 비율 제한, 벌크헤드, 서킷브레이커
### 1. 개념 설명
분산 네트워크 환경에서 일부 서비스에 장애가 발생하더라도, 시스템 전체가 도미노처럼 무너지는 연쇄 장애(Cascading Failure)를 막고 스스로 회복하게 만드는 설계 패턴입니다.

- 재시도(Retry): 일시적인 네트워크 환경 문제나 일시적 부하로 실패한 요청을 순간적인 시차를 두고 다시 시도하는 기법입니다.

- 비율 제한(Rate Limiting): 특정 클라이언트나 서비스가 정해진 시간 동안 보낼 수 있는 요청의 총량을 제한(Throttling)하여 시스템 과부하를 막는 방안입니다.

- 벌크헤드(Bulkhead): 선박의 격벽(Bulkhead)에서 유래한 용어로, 서비스의 리소스(스레드 풀, 프로세스 등)를 격리하여 하나의 서비스가 리소스를 독점해 다른 서비스까지 마비시키는 것을 방지합니다.

- 서킷브레이커(Circuit Breaker): 전기 차단기처럼, 하위 서비스의 에러율이 임계치를 넘으면 호출을 즉시 차단(Open)하고 에러 응답이나 폴백(Fallback)을 반환하여 하위 시스템이 회복할 시간을 벌어주는 패턴입니다.

### 2. 중요하다고 생각한 이유
- 장애 격리(Fault Isolation): 분산 시스템에서 "장애는 반드시 발생한다"는 것을 전제로 합니다. 복원성 패턴은 장애의 발생 자체를 막는 것이 아니라, 장애가 다른 곳으로 번지는 것을 차단하여 시스템의 '부분적 작동'을 보장합니다.

- 사용자 경험(UX) 방어: 서킷브레이커와 폴백을 통해 에러가 발생하더라도 무한 대기(Timeout 지연)를 겪게 하는 대신 "잠시 후 다시 시도해주세요" 혹은 캐시된 데이터를 즉시 보여줌으로써 최악의 UX를 피할 수 있습니다.

### 3. 운영 관점에서의 의미
- 인프라와 애플리케이션의 분리 (서비스 매시 활용): 과거에는 이러한 복원성 로직을 개발자가 애플리케이션 코드에 직접 구현해야 했으나, 이스티오(Istio) 같은 서비스 매시를 활용하면 코드 수정 없이 인프라(Sidecar 록시) 수준에서 설정만으로 재시도, 서킷브레이커 등을 일관되게 제어할 수 있어 운영 효율성이 극대화됩니다.

- 임계치(Threshold) 튜닝의 지속성: 잘못 설정된 재시도는 오히려 '디도스(DDoS) 공격'처럼 시스템 부하를 가중시킵니다. 운영자는 평소 메트릭을 모니터링하며 재시도 횟수, 타임아웃 시간, 서킷브레이커 오픈 조건 등의 파라미터를 지속적으로 최적화해야 합니다.

## 2.3 관측가능성 운영 프로세스 (SLI, SLO / 장애 발생, 알림, 추적 및 로그 분석)
### 1. 개념 설명
- 시스템 내부의 상태를 외부로 출력되는 데이터(로그, 메트릭, 추적)를 통해 정확히 파악하고(관측가능성), 이를 기반으로 서비스 수준을 관리하며 장애에 대응하는 체계입니다.

- SLI/SLO: 서비스 수준 지표(SLI, 예: 성공률)와 이를 바탕으로 설정한 목표치(SLO, 예: 99.9% 성공)를 통해 정량적으로 시스템 건강 상태를 정의합니다.

- 장애 프로세스: 이상 현상이 감지되면 즉시 담당자에게 알림(Alerting)을 보내고, 분산 추적(Tracing)을 통해 어떤 서비스 구간에서 병목이나 에러가 났는지 파악한 뒤, 상세 로그(Log)를 분석해 근본 원인을 찾아 해결하는 일련의 흐름입니다.

### 2. 중요하다고 생각한 이유
- 데이터 기반의 의존성 탈피: "서버가 좀 느린 것 같다"는 주관적 판단이 아니라, 구체적인 지표(SLI)를 통해 객관적으로 시스템을 평가할 수 있습니다.

- MTTR(평균 장애 조치 시간) 단축: 현대의 복잡한 마이크로서비스 환경에서는 장애가 났을 때 원인 지점을 찾는 것조차 수 시간이 걸릴 수 있습니다. 관측가능성 프로세스가 잘 잡혀있어야 장애 구간을 몇 분 만에 좁히고 진화할 수 있습니다.

### 3. 운영 관점에서의 의미
- 에러 예산(Error Budget) 중심의 운영: SLO를 달성하고 남은 여유분을 '에러 예산'으로 정의하여, 이를 다 쓰기 전까지는 공격적인 신기능 배포를 진행하고, 예산을 소진하면 시스템 안정화 작업에 우선순위를 두는 등 개발팀과 운영팀 간의 합리적인 의사결정 기준으로 활용됩니다.

- 알림 피로(Alert Fatigue) 관리: 무분별한 알림은 운영자의 피로도를 높여 정작 중요한 장애 알림을 놓치게 만듭니다. SLO를 위협하는 실질적인 이상 징후에만 알림이 가도록 필터링하고 프로세스를 자동화하는 것이 운영 실무의 핵심입니다.

## 4. 이해가 어려웠던 부분

쿠버네티스 분산 환경이나 수평 샤딩, 해시 같은 고도화된 아키텍처 방법론들이 상세하게 써져 있어서 한 번에 이해하기는 어려웠습니다. 이 부분은 꼼꼼히 읽으며 이해해보아야 할 것 같습니다.

프로메테우스(Prometheus) 예시에서 히스토그램(Histogram), 분위수(Quantile), 시계열(Time Series)을 통해 지연 시간을 근사적으로 표현하는 부분도 한 번 읽는 것만서는 이해하기 어려웠습니다. 개념 자체는 알겠지만, 실제 쿼리(Query)나 대시보드(Dashboard)에서 어떻게 해석해야 하는지는 추가 학습이 필요하다고 느꼈습니다.

## 5. 추가로 찾아본 내용

### AWS Loadbalancer 및 계층별 로드밸런싱 매핑
책에 나온 로드밸런서의 세 가지 유형이 실제 클라우드 환경(AWS) 인프라에서 어떻게 구현되는지 아키텍처 관점으로 추가 탐색을 진행함.

1. **플랫폼 로드밸런싱 ➡️ AWS ALB(Application Load Balancer) / NLB(Network Load Balancer)**
   - 인프라 최전선에서 동작하며 L4(IP/Port) 레벨은 NLB가 고속 처리하고, L7(HTTP/HTTPS) 레벨의 정적/동적 라우팅은 ALB가 담당하여 백엔드 EC2나 ECS 컨테이너 타겟 그룹으로 부하를 분산함.
2. **게이트웨이 로드밸런싱 ➡️ AWS API Gateway & Spring Cloud Gateway**
   - 클라이언트의 단일 진입점이 되어 백엔드 마이크로서비스들로 트래픽을 라우팅(Routing)해 주는 역할을 함. 단순히 부하 분산뿐 아니라 호스트/경로 기반 라우팅, 인증/인가 토큰 검증, Rate Limiting 설정을 엔드포인트 단에서 통합 관리함.
3. **클라이언트 측 부하 분산 ➡️ Spring Cloud LoadBalancer (Ribbon의 후속)**
   - 서버 측 하드웨어 장비가 트래픽을 쪼개주는 것이 아니라, 쿠버네티스 CoreDNS나 AWS Cloud Map 같은 '서비스 디스커버리'를 통해 가용한 파드(Pod)/인스턴스의 IP 목록을 클라이언트(호출 가상머신 앱)가 획득한 뒤, 클라이언트 메모리 내에서 Round-Robin 등의 알고리즘으로 직접 타겟을 선정해 HTTP 요청을 날리는 방식임을 확인하여 개념을 명확히 잡음.

## 6. 내 생각, 경험, 질문

### [내 생각 및 경험]

인프라를 운영하는 관점에서는 장애를 완벽히 막는 것보다, 발생한 장애의 '원인 구간'을 얼마나 빠르게 찾아내어 서비스 복구 시간(MTTR)을 단축하느냐가 중요합니다.

그런 면에서 SLI/SLO를 기반으로 알림 기준을 명확히 세우고, 경고가 울렸을 때 분산 추적(Tracing)으로 수십 개의 서비스 중 에러가 발생한 단 하나의 지점을 짚어낸 뒤, 해당 구간의 로그를 핀포인트로 분석하는 유기적인 장애 프로세스가 중요하다고 생각합니다.

또한, 무분별한 알림으로 인한 피로감을 줄이고 '에러 예산'이라는 객관적인 지표를 통해 개발팀과 시스템 안정화 시점을 정량적으로 합의할 수 있다는 점에서, 관측 가능성은 단순한 모니터링 툴을 넘어 실무 운영의 표준 가이드라인이 되어야 한다고 생각합니다.

## 7. 발표용 요약

### 발표 흐름

1. **도입부(SPOF와 분산 환경의 위험성):** 분산 네트워크 환경 및 MSA 구조는 단일 장애점(SPOF)에 취약하며 연쇄 장애로 이어질 수 있음.
2. **부하 분산 제어(로드밸런서):** 이를 해결하기 위해 트래픽 진입 경로 및 주체에 따라 계층화된 로드밸런싱(플랫폼, 게이트웨이, 클라이언트 측) 전략 구조 소개.
3. **인프라 회복력 확보(복원성 패턴):** 장애 발생을 디폴트로 전제하고 시스템을 보호하는 4대 복원성 패턴(재시도, 비율 제한, 벌크헤드, 서킷브레이커)의 메커니즘 설명.
4. **정량적 측정과 대응(관측가능성):** 보이지 않는 분산 내부를 SLI/SLO 지표 및 로그·메트릭·추적의 프로세스로 정량화하여 MTTR을 단축하는 방법 제시.
5. **데이터 시각화와 결론:** 평균 지표에 속지 않고 꼬리 지연을 잡아내기 위해 히스토그램과 히트맵을 활용하는 시각화 분석의 필요성을 공유.

### 발표 핵심 메시지
**"현대의 분산 시스템 운영은 장애를 완벽히 막는 것이 불가능하므로, 다각도의 부하 분산과 복원성 패턴으로 장애를 격리하고, 고도화된 관측 가능성 프로세스 및 시각화 도구를 통해 시스템 내부 상태를 데이터 기반으로 투명하게 실시간 관제해야 한다."**
