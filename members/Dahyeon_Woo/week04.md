## 3. 중요하다고 생각한 개념

### 1. 프로메테우스 시계열 데이터베이스(TSDB)
### 개념 설명
프로메테우스는 메트릭 데이터를 시계열(Time Series) 형태로 저장한다.
데이터는 메트릭 이름 + 레이블 + 타임스탬프 + 값 조합으로 관리된다.
TSDB는 샘플 수집 → 블록 생성 → 블록 병합 과정을 통해 데이터를 효율적으로 저장한다.
데이터 흐름은 메모리 → WAL(로그 선행 기입) → 디스크 순서로 처리된다.
카디널리티는 특정 레이블 조합이 얼마나 많이 생성되는지를 의미하며, 높을수록 성능 부담이 커진다.

### 중요하다고 생각한 이유
프로메테우스의 핵심은 결국 “메트릭 저장과 조회”이기 때문이다.
TSDB 구조를 이해해야 성능 문제나 저장 공간 문제를 해결할 수 있다.
특히 카디널리티 증가는 운영 장애의 주요 원인이 되므로 중요하다.

### 운영 관점에서의 의미
불필요한 레이블 사용을 줄여 저장 비용과 메모리 사용량을 최적화해야 한다.
WAL, 블록 병합 구조를 이해하면 장애 복구와 성능 튜닝이 가능하다.
장기 보관이 필요한 경우 타노스 같은 외부 스토리지 연계가 필요하다.

### 2. 프로메테우스 오토스케일링(어댑터/KEDA)
### 개념 설명
프로메테우스 어댑터는 프로메테우스 메트릭을 쿠버네티스 HPA가 사용할 수 있는 형태로 변환한다.
discovery, association, naming, querying 규칙을 통해 메트릭과 쿠버네티스 리소스를 연결한다.
KEDA는 이벤트 기반 오토스케일링을 지원하며, 다양한 외부 메트릭을 기반으로 스케일링한다.

### 중요하다고 생각한 이유
단순 CPU 사용률이 아니라 실제 서비스 메트릭 기반으로 스케일링이 가능해진다.
트래픽 급증 상황에서 자동 확장이 가능해 서비스 안정성이 높아진다.
클라우드 네이티브 환경에서 핵심적인 자동화 요소라고 생각했다.

### 운영 관점에서의 의미
적절한 메트릭 선정이 중요하다. 잘못된 메트릭은 과도한 스케일링을 유발할 수 있다.
HPA/KEDA 설정 최적화를 통해 비용과 성능 균형을 맞출 수 있다.
운영자는 메트릭 지연이나 어댑터 장애까지 함께 모니터링해야 한다.

### 3. 타노스 운영 아키텍처
### 개념 설명
타노스는 프로메테우스의 확장 솔루션으로 장기 저장과 글로벌 조회를 지원한다.
사이드카는 프로메테우스 데이터를 외부에서 사용할 수 있도록 제공한다.
쿼리어는 여러 프로메테우스/스토리지에 분산된 데이터를 통합 조회한다.
스토어 게이트웨이, 콤팩터, 버킷 등을 통해 오브젝트 스토리지 기반 장기 보관을 수행한다.

### 중요하다고 생각한 이유
기본 프로메테우스는 장기 저장과 고가용성에 한계가 있기 때문이다.
대규모 환경에서는 여러 클러스터 데이터를 통합해서 보는 기능이 필수적이다.
다운 샘플링과 글로벌 뷰 기능이 운영 효율성을 크게 높인다.

### 운영 관점에서의 의미
장애 발생 시 여러 클러스터 메트릭을 통합 분석할 수 있다.
오브젝트 스토리지를 활용해 저장 비용을 절감할 수 있다.
다만 구성 요소가 많아져 운영 복잡도가 증가하므로 아키텍처 이해가 중요하다.

## 4. 이해가 어려웠던 부분

* 프로메테우스 TSDB의 블록 병합(Compaction) 과정이 처음에는 이해하기 어려웠다.
  특히 메모리 → WAL → 디스크 → 블록 병합 흐름이 어떻게 이어지는지 헷갈렸다.

* 카디널리티 개념도 어려웠다.
  레이블이 많아질수록 단순히 데이터 양만 늘어나는 것이 아니라, 메모리 사용량과 조회 성능까지 영향을 준다는 점이 인상적이었다.

* 타노스 구조는 구성 요소가 많아서 역할 구분이 복잡했다.
  사이드카, 쿼리어, 스토어 게이트웨이, 콤팩터 각각이 어떤 위치에서 동작하는지 정리하면서 이해했다.

## 5. 추가로 찾아본 내용

* WAL(Write Ahead Log)은 장애 발생 시 데이터 유실을 줄이기 위한 로그 저장 방식이라는 점을 추가로 학습했다.

* HPA는 기본적으로 CPU/메모리 기반 오토스케일링을 수행하지만, 프로메테우스 어댑터를 사용하면 사용자 정의 메트릭 기반 스케일링도 가능하다는 점을 확인했다.

* 타노스는 S3 같은 오브젝트 스토리지를 사용해 장기 보관을 수행하며, 여러 클러스터의 메트릭을 통합 조회할 수 있다는 점을 추가로 찾아봤다.

## 7. 발표용 요약

### 발표 흐름

1. 프로메테우스 기본 구조
   → 오퍼레이터, 익스포터, 어댑터 역할 설명

2. TSDB와 데이터 저장 방식
   → 시계열 데이터 구조와 블록 관리 설명

3. 쿠버네티스 연계와 오토스케일링
   → HPA, 프로메테우스 어댑터, KEDA 소개

4. 알람 및 운영 아키텍처
   → 규칙 관리, 샤딩, 페더레이션 설명

5. 타노스 확장 구조
   → 글로벌 뷰, 장기 저장, 고가용성 설명

### 발표 핵심 메시지

* 프로메테우스는 단순 모니터링 도구가 아니라 “클라우드 네이티브 운영 플랫폼”의 핵심 요소다.

* 프로메테우스는 메트릭 수집 → 저장 → 분석 → 오토스케일링 → 알람까지 전체 운영 흐름을 연결한다.

* 대규모 환경에서는 타노스를 통해 장기 저장과 글로벌 통합 모니터링을 구현할 수 있다.
