# 2장 정리

## 분산 클라우드 네이티브 환경

- 일반 클라우드: vm 중심
- 클라우드 네이티브: 컨테이너 + 분산시스템(마이크로서비스, k8s) 중심

#### 분산 클라우드 시스템에서는 네트워크 장애가 치명적이다

- 모놀리식 환경에서는 하나의 프로세스 내부 함수 호출이 많다
- but 마이크로서비스에서는 전부 네트워크 호출이 된다.
    - 서비스 a → 서비스 b
    - 서비스 b → 서비스 c
- 따라서 네트워크 문제가 곧 서비스 장애가 된다.

### 네트워크는 spof 특성이 있다

dns 장애, 로드밸런서 장애, 라우팅 문제 만 생겨도 전체 서비스가 무너질 수 있음

특히 쿠버네티스는

- Pod ↔ Pod
- Service ↔ Service
- Node ↔ Node

통신이 엄청 많아서 네트워크 의존도가 높다

#### 분산 마이크로서비스와 동적인 클라우드 환경에서의 장애 해결을 위해 Observability의 범위 자체가 확장되고 있다.

어플리케이션 상태(metrics, logs, traces)만 보면 부족하고 네트워크 흐름 자체도 같이 봐야 함

→ 네트워크 데이터(커널 인프라 레벨)가 중요

"누가 누구와 통신했는가"

"어디서 지연이 발생했는가"

"어디서 패킷이 끊겼는가"

## 로드밸런싱 3가지

```bash
사용자
  ↓
AWS ALB
  ↓
API Gateway / Ingress
  ↓
service-a
  ↓
service-b

# 그리고 내부 서비스 간 통신을 관리하는 게 Service Mesh
```

1. 플랫폼 로드밸런싱
    - 인프라 레벨 로드밸런싱
    - 트래픽을 여러 서버/파드에 분산
    - aws alb, nginx
2. 게이트웨이 로드밸런싱
    - api gateway, ingress, gateway api
    - 쿠버네티스 내의 무슨 서비스로 보낼지
    - 쿠버네티스는 api 서버의 기능이 떨어져 멀티 테넌트, 멀티 클러스터, 멀티 클라우드의 구성 시에 많은 어려움이 따름 다양한 방법으로 문제 해결이 가능하지만, 그중 실리움이 좋은 대안이라고 판단
3. 클라이언트 측 부하 분산
    - 로드밸런서가 서버 쪽에 있는 게 아니라, 요청 보내는 클라이언트가 직접 대상 선택
    - Netflix OSS 계열

## 서비스메시

- 내부 서비스 간 통신 관리 시스템
- 네트워크 기능을 어플리케이션 코드와 분리
- 대표적인 게 istio
- 원래 istio는 사이드카 방식
    
    ```bash
    Pod
     ├ app container
     └ envoy sidecar
    ```
    
    → 모든 pod 옆에 envoy 프록시 붙임
    
    → 모든 트래픽이 envoy를 지나니까 
    
    - 누가 누구 호출
    - 응답시간
    - 실패율
    
    를 envoy를 통해 자동 수집 가능
    
    → 네트워크 레벨 telemetry를 얻기 쉬워짐
    

#### cilium/ebpf의 등장

이때 cilium은 ebpf 기반으로 접근하므로 커널 레벨에서 네트워크 처리

즉 굳이 pod마다 프록시 넣지 않고도 커널에서 처리 가능

즉 sidecar 없이도:

- 네트워크 정책
- observability
- load balancing

등 가능하게 함.

## 복원력 높일 수 있는 네 가지 패턴

마이크로 서비스 환경에서는 

```bash
Service A
  ↓
Service B
  ↓
Service C
```

처럼 계속 호출

→ 하나 느려지면 연쇄적으로 다 느려질 수 있음

→ 장애가 전파됨

⇒ 장애의 전파를 막기 위한 패턴

1. 재시도
    - 시스템 에러의 경우 재시도하면 정상적인 처리가 가능하다
    - 가끔 발생하는 장애에는 효율적이지만,
    - 누적돼 시스템 과부하 일으키면(retry storm) 안되므로 반복적이고 무차별적인 재시도에 주의해야 함
2. 비율 제한
    - 비율 제한 기법 도입 → 제한된 처리량이나마 요청에 대응 가능
3. 벌크헤드
    - 마이크로서비스의 각 서비스별 전용 thread 풀을 분리
    - 마비된 마이크로 서비스가 스레드 풀을 다 사용하고 있으면 다른 서비스까지도 먹통 되므로
    - 서비스 간 호출에 사용하는 스레드 풀을 애초에 분리시킴
    
    → 한 서비스의 장애로 리소스가 고갈되더라도 다른 서비스는 독립된 자원을 통해 정상적으로 동작 가능
    
4. 서킷 브레이커
    - 벌크헤드 패턴을 확장한 개념
    - 각 마이크로 서비스 앞단에 서킷 브레이커를 둬서 정상이면 문제없이 통과시키고,
    - 문제 감지하면 호출 강제 종료하고 더이상 응답 기다리지 않게 함 → 실패 처리

## 오토스케일링

현재 부하에 따라 리소스 양을 자동으로 조정하는 시스템

#### hpa의 오토스케일링 과정

```bash
Metrics 수집
   ↓
HPA Controller
   ↓
현재 값 vs 목표 값 비교
   ↓
필요한 Pod 개수 계산
   ↓
Deployment replicas 수정
   ↓
ReplicaSet → Pod 생성/삭제
```

#### 메트릭 수집

hpa 정의에 따라 스케일링된 파드에 대한 메트릭을 가져온다.

- metrics server → 기본
- prometheus & prometheus adapter → custom metrics

파드에서 직접 메트릭을 읽지 않고, 집계된 메트릭을 제공하는 쿠버네티스 메트릭 api에서 메트릭을 읽는다.

### 오토스케일링 오픈소스

1. Kubernetes Metrics Server (기본)
    
    cpu, 메모리 같은 기본 리소스를 hpa에 제공
    
    ```bash
    Pod
     ↓ (kubelet /stats)
    Node kubelet
     ↓
    Metrics Server
     ↓
    Kubernetes API Server
     ↓
    HPA
    ```
    
    - 메트릭 서버
        - summary api를 통해 각 노드에 질의하여 클러스터의 노드와 파드를 추적하고 메트릭을 수집
        - 메트릭 서버는 kubelet으로부터 리소스 메트릭을 수집하고, 이를 vpa와 hpa가 활용할 수 있도록 메트릭 api를 통해 쿠버네티스 api 서버로 전달한다.
2. 프로메테우스 어댑터
    
    cpu, 메모리 같은 기본 메트릭이 아닌 사용자가 만든 지표를 custom metrics라고 함
    
    이때 어댑터가
    
    ```bash
    Prometheus → (데이터 저장)
            ↓
    Adapter → Kubernetes Custom Metrics API로 변환
    ```
    
    - 프로메테우스 어댑터에 대한 구성 작성하여 메트릭 수집하도록 프로메테우스 구성
