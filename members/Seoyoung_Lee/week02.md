# 모니터링, Observability, 가시성

---

### 모니터링과 observability(관측 가능성)의 차이점

- 모니터링은 블랙박스와 화이트박스로 구분되는데
    - 블랙박스: 밖에서 서비스 결과만 봄 → 사용자 입장에서 서비스가 정상인가
    - 화이트박스: 시스템 내부 상태를 직접 봄 → 시스템 내부 상태가 어떤가
- 옵저버빌리티는 화이트박스 모니터링 + 시스템에 대한 이해를 기반으로 미래에 발생할 이벤트 예측까지

### 모니터링

- 모든 것을 수집하고 모니터링하기보다는,
- 가능한 샘플링을 사용해 적은 데이터를 수집하더라도 의미 있고 예측 가능한 결과를 도출해내는 것이 좋음

### 가시성

- 안정적이고 신뢰할 수 있는 클라우드에서도 인프라와 관련한 다양한 에러를 접하기 마련이다.
- 특히, 클라우드 환경에서는 많은 장애와 에러가 네트워크로 인해서 발생한다.
- 따라서 네트워크 이벤트와 신호를 수집하고 원인 분석과 문제 해결에 활용하는 방안을 마련해야 한다.

인프라와 네트워크 레벨에서 문제 원인을 이해하고, 이를 해결하는 방법과 절차를 **가시성**이라고 지칭

그러나 옵저버빌리티와 가시성의 차이를 잘 모르겠다…. 범용적으로 쓰이는 건지

---

### 관측 가능성의 3가지 요소

1. 메트릭: 일정 시간 동안 측정된 데이터를 집계하고 이를 수치화 → 시계열과 히스토그램 차트
2. 로그: 어플리케이션 실행 시 생성되는 텍스트 라인 → 오픈텔레메트리
3. 추적: 하나의 요청(트랜잭션)이 여러 시스템을 거치며 처리된 전체 흐름
    - 트랜잭션이 시스템을 이동하는 경로
    - 트랜잭션을 처리하는 과정에서 발생하는 대기시간과 지연시간

# 데이터 수집

---

### SLI와 SLO

- SLI: 서비스 수준 지표
- SLO: 서비스 수준 목표

SLI는 SLO라고 불리는 임계 기준을 달성해야 하며, SLO는 SLI의 상한과 하한 범위를 지정한다.

### 구글의 골든 시그널

메트릭을 수집할 때 네 가지 골든 시그널에 집중해야 한다

1. 지연
    - 빈번하게 발생하는 지연은 에러가 아닌 경우가 많다.
    - 지연이 꼭 에러인 것은 아니댜.
    - 사용자가 지연을 자주 경험하면 시스템이 정상적이지 않다는 것을 추론할 수 있다.
    - 지연은 잠재적인 장애를 예측할 수 있는 유용한 신호로
    - 지연이 그니까 어느 정도 느려지긴 했는데 짧으면 OK, 길어지면 에러로 판단
2. 에러
    - 에러의 기준을 명확히 설정해야 함
    - 에러는 3가지 상태: 정상 → 에러 처음 발견 시 대기 → 활성 상태를 유지하면 파이어링
3. 트래픽
    - 발생하는 요청의 양
4. 포화
    - 리소스가 현재 얼마나 채워졌는지와 얼마나 가득 채울 수 있는지를 나타냄
    - 지연 증가는 종종 포화의 선행 지표
    - 포화 상태가 되지 않도록 시스템을 설계하는 것이 중요 → 동적인 트래픽에 맞추어 스케일하는 방법을 제공하는 것이 적합 LIKE 오토스케일링, db의 샤딩

신호가 에러, 지연, 정상, 오류(소스상의 버그)인지 정의하는 작업이 필요

### 시계열 데이터

시간에 따라 계속 변하는 데이터 → 보통 프로메테우스가 저장

- 그라파나 로키
    - 로그 저장/검색 시스템
    - 로그용 프로메테우스 느낌
    - 프로메테우스가 cpu 90% 찍혔다는 걸 보여준다면, 로키는 왜 90%가 됐는지 로그를 보여줌
    - 그라파나에서 프로메테우스로 받아보는 cpu 급등 발견 → 같은 pod 로그 바로 조회 가능
- 프로메테우스 이그젬플러
    - 메트릭 데이터에 해당 메트릭 샘플을 대표하는 실제 요청(trace)의 trace ID를 붙임
        
        ```bash
        http_request_duration_seconds_bucket
        value=5s
        
        exemplar:
        trace_id=abc123
        ```
        
        - 이 latency spike는 trace abc123때문이다 라고 연결해줌
        - 이때 모든 느린 요청 trace를 다 저장하는 것이 아니라 샘플링된 trace 중 일부, 대표 요청 하나를 연결 → 그래서 exempler
    - 즉 metrics ↔ traces 연결 담당

### 프로메테우스로 메트릭이 어떻게 제공될까

- Prometheus는 Push가 아니라 Pull 방식이다.
- 흐름
    1. application이 Prometheus client 또는 OTel을 통해 메트릭을 생성하고,
    2. http 엔드포인트로 메트릭(ex: /metrics)을 노출
    3. Prometheus의 ServiceMonitor를 통해 어떤 서비스의 어떤 엔드포인트를 Prometheus가 scrape할지 정의
    4. Operator가 ServiceMonitor를 감지하여 scrape target 리스트에 해당 엔드포인트를 추가
    5. Prometheus가 주기적으로 해당 엔드포인트에서 scrape해서 가지고 온다.
    6. Prometheus가 수집한 메트릭을 TSDB에 저장 후 Grafana에서 조회 가능

### 오픈트레이싱, 예거, 오픈텔레메트리

- OpenTracing: tracing 표준 시도
- Jaeger: tracing 시스템 구현체 → 저장소 + ui + tracing backend
- OpenTelemetry: tracing + metrics + logs 통합 표준 → 데이터 생성/수집 표준

마이크로서비스 환경에서 문제가 발생했을 때, 어느 서비스가 느린지, db때문인지 등을 확인하기 위해서는 분산 추적이 필수적

어플리케이션들이 trace_id=abc123 같은 trace 데이터를 생성하면,

Jaeger가 수집, 저장, 검색, 시각화

- ex
    
    ```bash
    POST /payment
    총 5초
    
    # 펼쳐보면
    Gateway      20ms
    Auth         30ms
    Payment      4.5s
    DB Query     4.3s
    ```
    
    - 어디서 시간이 오래 걸렸는지 한눈에 보임

그러나 초창기 tracing 시스템들(예거, datadog 등)은 각자 API가 달랐음 → tracing 시스템 간 이동이 어려움

그래서 Tracing API의 표준을 정의한 OpenTracing이 나오게 됨

허나 지금은 OpenTracing, OpenCensus 합쳐져서 OpenTelemetry로 통합됨

즉, OpenTelemetry는 어플리케이션 내부에서 observability 데이터를 생성하고 내보내 표준 sdk/수집 체계

⇒ 현대 구조

```bash
Application
   ↓
OTel SDK
   ↓
OTel Collector
   ↓
Jaeger / Prometheus / Loki / Tempo
```

### Prometheus Exporter와 Opentelemetry

- 이렇게 보면 Prometheus exporter와 opentelemetry의 기능이 겹치는 것 같다
- Prometheus exporter는 메트릭만을 대상으로 하고 otel은 메트릭, 로그, trace 모두를 대상으로 하는 것 같은데,
- 그렇다면 Promethues만의 이점이 무엇이지..

### 데이터의 상관관계

- 서로 다른 데이터(로그, 메트릭, 추적, 트래픽)가 결합이 되었을 때 비로소 데이터의 활용 가치가 증가
- 로그에 대한 통계와 집계 정보는 메트릭이 될 수 있으며, 로그를 발생 시점과 처리 시점을 계산해서 나열하면 추적이 될 수 있다.
