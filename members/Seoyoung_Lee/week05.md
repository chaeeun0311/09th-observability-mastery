### 로키 컴포넌트

로키의 컴포넌트는 크게 읽기, 쓰기, 저장 컴포넌트로 구성된다.

!image.png

**쓰기 경로** 

- 로그 소스 → 디스트리뷰터 → 인제스터 순서로 흐른다.
- Promtail, Fluent Bit, Grafana Agent 같은 로그 수집기가 이 로그에 레이블을 붙여서 Loki로 보낸다.
- 디스트리뷰터는 수신된 로그를 검증하고 일관된 해싱으로 적절한 인제스터에 분산시킨다.
- 인제스터는 실제 로그 내용을 메모리에 버퍼링하다가 청크(chunk) 단위로 묶어 오브젝트 스토리지에 flush하고, 레이블 인덱스는 인덱스 스토리지에 쓴다.

**저장소 레이어** 

- 오브젝트 스토리지(S3, GCS 등)는 압축된 로그 청크를, 인덱스 스토리지(BoltDB 등)는 레이블 인덱스를 각각 보관
- 인덱스 게이트웨이는 인덱스 조회를 캐싱해 읽기 부하를 줄이고, 콤팩터는 청크를 주기적으로 재압축하고 보존 정책(정책에 따라 오래된 로그 삭제)을 집행

**읽기 경로**

- 쿼리 프론트엔드 → 쿼리 스케줄러 → 쿼리어 순서로 흐른다.
- 프론트엔드는 큰 쿼리를 잘게 분할하고 캐싱·재시도를 처리하며, 스케줄러는 큐를 관리해 쿼리어에게 작업을 배분한다.
- 쿼리어는 실제로 로그를 찾아오는 역할을 한다. → 인덱스 게이트웨이로 인덱스를 조회하고, 오브젝트 스토리지에서 청크를, 인제스터의 메모리에서 최신 로그를 함께 가져와 결과를 병합한다.

**운영 컴포넌트** 

- 룰러는 내부적으로 쿼리어를 통해 LogQL 규칙을 주기적으로 평가
- 조건 만족 시(=이상 상황 발생 시) 알림 생성

## Mimir 메트릭 관리

---

**Prometheus 메트릭을 장기 저장하고 대규모로 조회하기 위한 분산 시계열 데이터베이스**

Prometheus는 기본적으로 로컬 디스크에 데이터를 저장하는 단일 서버 중심 구조이기 때문에 장기 보관과 대규모 조회에는 한계가 있다 → 오브젝트 스토리지 기반의 분산 장기 저장소가 필요

타노스는 Prometheus 메트릭을 장기 저장하기 위한 대표적인 오픈소스 솔루션인데, 타노스는 운영 관점에서는 구성 요소가 많고, 튜닝이 어렵고, 장애가 발생했을 때 원인을 파악하거나 성능을 최적화하는 데 시간이 오래 걸릴 수 있다

→ 복잡성을 줄이고 더 대규모 환경에서 안정적으로 운영하기 위한 대안이 미미르

### 미미르 컴포넌트

---

!image.png

**쓰기 경로**

- 프로메테우스가 여러 대상에서 /metrics scrape하고 원격 쓰기 api 사용해서 미미르로 푸시
- 이 요청은 디스트리뷰터가 받아서 처리하고 여러 인제스터에 복제 전송
- 인제스터로 들어온 샘플은 메모리에 보관되고, wal에 기록됨
- 메모리 내 샘플은 주기적으로(기본적으로 2시간) TSDB 블록으로 만들어져 디스크로 flush되고, wal은 해당 샘플이 새 TSDB 블록으로 생성될 때 삭제됨
- 콤팩터가 여러 블록을 단일 블록으로 병합하고, 중복 샘플을 제거 → 스토리지 절약

**읽기 경로**

- 쿼리는 쿼리 프런트엔드로 먼저 들어간다.
- 쿼리 프런트엔드는 쿼리를 여러 개의 더 작은 쿼리로 분할
- 프런트엔드는 결과 캐시를 확인하고, 캐시된 경우에는 캐시 결과 반환하고, 캐시에서 응답할 수 없는 쿼리는 프런트엔드 내부 또는 쿼리 스케줄러의 큐에 넣는다.
- 쿼리어는 큐에서 “쿼리 작업”을 가져와 실행
- 쿼리를 실행할 때 실제 메트릭 데이터는 인제스터(최근 메트릭 데이터)와 스토어 게이트웨이(장기 저장된 블록 데이터)에서 가져온다.

### 해시 링

---

미미르는 여러 컴포넌트로 구성되는데, 각 컴포넌트는 고가용성과 확장을 위해 여러 복제본으로 실행될 수 있음

이때 같은 종류의 컴포넌트 복제본들 사이에서 데이터나 작업을 분산하고, 필요한 경우 복제하기 위해 해시 링을 사용

**해시 링 방식이란?** 

아무 인제스터에 랜덤으로 보내는 게 아니라, 메트릭의 이름/레이블 같은 값을 해싱해서 정해진 규칙에 따라 담당 인제스터를 고른다는 뜻

**랜덤으로 보내면 불편한 점**

인제스터는 단순히 쓰기 요청을 받는 역할만 하는 것이 아니라, 들어온 시계열 샘플을 메모리에 보관하고 이후 TSDB 블록으로 만드는 역할도 한다. 따라서 같은 시계열 데이터가 매번 다른 인제스터로 흩어지면 조회, 블록 생성, 복제 관리가 복잡해진다. 이를 방지하기 위해 Mimir는 해시 링을 사용하여 같은 시계열이 항상 정해진 인제스터 그룹으로 전달되도록 한다.

## Tempo 추적 관리

---

분산 추적 데이터를 저장하고 조회하는 시스템

#### 특징

- 애플리케이션에서 OpenTelemetry SDK 등을 통해 생성된 trace/span 데이터가 OpenTelemetry Collector를 거쳐 Tempo로 전달된다.
- Tempo는 수신한 trace를 Distributor와 Ingester를 통해 처리하고, 블록 단위로 오브젝트 스토리지에 저장한다.
- Grafana에서는 Tempo 데이터 소스를 통해 traceID 또는 TraceQL을 사용해 특정 요청의 흐름을 조회할 수 있으며, 이를 통해 어떤 서비스에서 지연이나 오류가 발생했는지 파악할 수 있다.
- 또한 Tempo는 metrics-generator(trace 데이터를 기반으로 메트릭 만들어줌)를 통해 trace 데이터를 기반으로 서비스 그래프와 exemplar(메트릭 그래프의 특정 지점과 trace를 연결해주는 링크)를 생성

## Jaeger 추적 관리

---

Tempo처럼 **분산 추적을 저장하고 조회하는 도구**

**공통점**

- 목적은 비슷 → 분산 시스템에서 요청 하나가 어떤 서비스를 거쳐 처리되었는지 추적한다
- 둘 다 OpenTelemetry와 함께 사용 가능

**차이점**

- 템포의 강점: Grafana stack과의 통합, 멀티테넌트 운영, 오브젝트 스토리지 기반 장기 저장, span metrics 및 service graph 자동 생성
- 예거의 강점: Cassandra, Elasticsearch, OpenSearch 등 다양한 저장소 백엔드를 지원하고, 전통적인 trace 검색 및 분석 UI를 통한 병목과 오류의 근본 원인 추적
