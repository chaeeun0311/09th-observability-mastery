# NetworkDoctor: Kubernetes 네트워크 장애 진단 프로젝트

## 1. 프로젝트 소개

**NetworkDoctor**는 Kubernetes 환경에서 발생하는 네트워크 장애를 관측하고, 여러 메트릭을 조합해 장애 원인 후보를 좁히는 네트워크 진단 프로젝트입니다.

Kubernetes 환경에서는 여러 Pod와 Service가 네트워크를 통해 연결되어 있습니다. 이때 요청 지연, timeout, 연결 실패 같은 문제가 발생하면 단순히 “요청이 실패했다”는 사실만으로는 원인을 판단하기 어렵습니다.

예를 들어 같은 요청 실패라도 원인은 다양할 수 있습니다.

* 잘못된 포트로 요청한 경우
* Service 또는 Pod 연결이 잘못된 경우
* 네트워크 중간에서 패킷 손실이 발생한 경우
* TCP 레벨에서 재전송이나 혼잡이 발생한 경우
* 애플리케이션 자체의 처리 지연이 발생한 경우

이 프로젝트는 이러한 상황에서 Prometheus 기반 메트릭과 eBPF 기반 TCP 메트릭을 함께 확인하여, 네트워크 장애 원인을 더 설명 가능하게 진단하는 것을 목표로 합니다.

---

## 2. 프로젝트 목표

NetworkDoctor의 목표는 단순히 장애 발생 여부를 감지하는 것이 아니라, 장애가 발생했을 때 다음 질문에 답하는 것입니다.

* 어떤 Pod 또는 Service 간 통신에서 문제가 발생했는가?
* 요청 실패나 지연 증가가 실제 네트워크 문제와 관련이 있는가?
* TCP 레벨에서 재전송, RTT 증가, CWND 감소 같은 변화가 나타났는가?
* 관측된 메트릭 조합을 바탕으로 어떤 원인 후보를 제시할 수 있는가?

즉, 이 프로젝트의 핵심은 단일 메트릭만 보는 것이 아니라 **여러 지표를 조합해 장애 원인 후보를 좁히는 것**입니다.

---

## 3. eBPF가 필요한 이유

NetworkDoctor는 먼저 Prometheus 기반 메트릭을 통해 Kubernetes 네트워크 상태를 관측합니다.

Prometheus Exporter를 사용하면 다음과 같은 지표를 확인할 수 있습니다.

| 구분             | 예시 지표                               | 확인할 수 있는 내용             |
| -------------- | ----------------------------------- | ----------------------- |
| Node 메트릭       | CPU, Memory, Network RX/TX          | 노드 리소스 사용량과 네트워크 송수신량   |
| Network 메트릭    | packet drop, TCP retransmit         | 노드 단위의 패킷 드랍, 재전송 증가 여부 |
| CoreDNS 메트릭    | DNS request, DNS error, DNS latency | DNS 요청량, 실패율, 응답 지연     |
| Kubernetes 메트릭 | Pod 상태, Service 상태                  | 워크로드 상태와 배포 상태          |

이러한 메트릭만으로도 “특정 노드에서 네트워크 사용량이 증가했다”, “패킷 드랍이나 재전송이 늘었다”, “DNS 에러가 증가했다” 같은 이상 신호는 확인할 수 있습니다.

하지만 일반적인 Exporter 메트릭은 대부분 노드 단위나 컴포넌트 단위로 집계됩니다. 그래서 특정 노드에서 재전송이나 드랍이 증가했다는 사실은 알 수 있어도, 그 문제가 어떤 Pod의 어떤 통신에서 발생했는지까지는 바로 확인하기 어렵습니다.

또한 네트워크 장애 원인을 구분하려면 단순 송수신량이나 집계 지표뿐만 아니라, 커널 내부에서 발생하는 연결 실패, 재전송, reset, socket drop, RTT 변화 같은 세부 이벤트를 함께 볼 필요가 있습니다. 이러한 정보는 일반 Exporter만으로는 충분히 수집하기 어렵습니다.

그래서 eBPF를 함께 사용했습니다 !!

eBPF를 사용하면 커널 네트워크 경로에서 발생하는 이벤트를 직접 관측할 수 있고, 이를 Pod, Namespace, Node, 목적지 IP, 목적지 Port 정보와 연결할 수 있습니다.

즉, eBPF는 Prometheus Exporter가 제공하는 노드/컴포넌트 단위 메트릭을 보완하여, 커널 네트워크 경로에서 발생하는 세부 네트워크 이벤트를 확인하기 위해 필요합니다.

## 4. 전체 시스템 구조

NetworkDoctor는 Kubernetes 노드에 eBPF 기반 Node Agent를 배포하고, 이 Agent가 수집한 네트워크 메트릭을 Prometheus가 수집하도록 구성합니다.

수집된 메트릭은 Grafana 대시보드에서 시각화하여, 장애 시나리오별로 어떤 지표가 변화하는지 확인합니다.

전체 구조는 다음과 같습니다.

```text
[Client Pod]
    |
    | HTTP Request
    v
[Backend Service]
    |
    v
[Backend Pod]


[Node Agent - eBPF]
    |
    | TCP RTT / Retransmit / CWND / Connection Fail 등 수집
    v
[Prometheus]
    |
    v
[Grafana Dashboard]
```

위 구조는 크게 두 부분으로 나눌 수 있습니다.

```text
1. 트래픽 경로
   client Pod → backend Service → backend Pod

2. 관측 경로
   Node Agent - eBPF → Prometheus → Grafana
```

`client Pod → backend Service → backend Pod`는 실제 실험 트래픽이 흐르는 경로입니다.

`Node Agent → Prometheus → Grafana`는 해당 트래픽에서 발생하는 네트워크 상태를 관측하는 경로입니다.

## 5. 실험 구조

실험은 `client Pod`에서 `backend Service`로 지속적으로 요청을 보내는 방식으로 진행합니다.

```text
client Pod  →  backend Service  →  backend Pod
```

이후 다음 세 가지 트래픽을 발생시켜 메트릭 변화를 비교합니다.

1. 정상 트래픽
2. 연결 실패 트래픽
3. 패킷 손실 트래픽

정상 트래픽에서는 baseline 메트릭을 확인하고, 연결 실패와 패킷 손실 상황에서는 어떤 지표가 다르게 변화하는지 비교합니다.

이 실험의 목적은 단순히 요청이 성공했는지 실패했는지를 보는 것이 아니라, **실패 원인에 따라 eBPF 메트릭이 어떻게 다르게 나타나는지 확인하는 것**입니다.

---

## 6. 시나리오 1: 연결 실패 트래픽

### 6.1 상황

첫 번째 시나리오는 `client Pod`에서 잘못된 포트 또는 잘못된 엔드포인트로 요청을 보내는 상황입니다.

예를 들어 backend Service는 8080 포트로 열려 있는데, client가 존재하지 않는 9999 포트로 요청을 보내는 상황을 만들 수 있습니다.

```text
client Pod  →  backend Service:9999
```

### 6.2 기대되는 현상

이 경우 정상적인 HTTP 응답을 받지 못하고 연결 실패가 발생합니다.

관측 포인트는 다음과 같습니다.

| 관측 지표           | 기대 변화                       |
| --------------- | --------------------------- |
| 요청 실패율          | 증가                          |
| Connection Fail | 증가                          |
| Timeout         | 상황에 따라 증가                   |
| TCP Retransmit  | 패킷 손실 시나리오만큼 핵심 지표는 아닐 수 있음 |
| RTT / CWND      | 큰 변화가 없을 수 있음               |

### 6.3 이 시나리오의 목적

이 시나리오는 요청 실패가 항상 네트워크 혼잡이나 패킷 손실 때문에 발생하는 것은 아니라는 점을 보여주기 위한 시나리오입니다.

같은 “요청 실패”라도 잘못된 포트, 잘못된 주소, Service 설정 문제, NetworkPolicy 차단 등으로 발생할 수 있습니다.

따라서 이 경우에는 RTT 증가나 CWND 감소보다, 연결 실패 이벤트나 timeout 발생 여부를 중심으로 확인하는 것이 더 적절합니다.

---

## 7. 시나리오 2: 패킷 손실 트래픽

### 7.1 상황

두 번째 시나리오는 `tc netem` 등을 사용해 네트워크에 패킷 손실을 주입하는 상황입니다.

```text
client Pod  →  backend Service
          중간에 packet loss 발생
```

예를 들어 특정 인터페이스에 10% 패킷 손실을 주입하면, 일부 요청은 정상 처리되지만 일부 요청은 지연되거나 실패할 수 있습니다.

### 7.2 기대되는 현상

패킷 손실이 발생하면 TCP는 손실된 패킷을 다시 전송하려고 합니다.

따라서 다음과 같은 변화가 나타날 수 있습니다.

| 관측 지표          | 기대 변화     |
| -------------- | --------- |
| 요청 지연 시간       | 증가        |
| 요청 실패율         | 상황에 따라 증가 |
| TCP Retransmit | 증가        |
| RTT            | 증가        |
| CWND           | 감소 가능     |
| Timeout        | 증가 가능     |

### 7.3 이 시나리오의 목적

이 시나리오는 단순히 HTTP 요청이 느려졌다는 사실뿐만 아니라, TCP 레벨에서도 실제 손실과 혼잡 반응이 나타나는지 확인하기 위한 시나리오입니다.

특히 다음 조합을 확인하는 것이 중요합니다.

```text
Latency 증가
+ TCP Retransmit 증가
+ RTT 증가
+ CWND 감소
```

이 조합이 함께 나타나면, 애플리케이션 코드 문제보다는 네트워크 손실 또는 혼잡 가능성을 더 강하게 의심할 수 있습니다.

## 8. Grafana 대시보드 구성

발표용 Grafana 대시보드는 복잡한 운영 대시보드보다는, 시나리오별 차이를 확인할 수 있는 핵심 지표 중심으로 구성합니다.

### 8.1 공통 패널

| 패널                    | 설명                   |
| --------------------- | -------------------- |
| Request Rate          | client → backend 요청량 |
| Error Rate            | 실패한 요청 비율            |
| p95 / p99 Latency     | 요청 지연 시간             |
| TCP Retransmit        | TCP 재전송 발생량          |
| RTT                   | TCP 왕복 지연 시간         |
| CWND                  | TCP 혼잡 윈도우 변화        |
| Connection Fail Count | 연결 실패 이벤트 수          |
| Timeout Count         | 요청 timeout 발생 수      |

### 8.2 시나리오별 관찰 포인트

#### 정상 트래픽

정상 트래픽에서는 기준 상태를 확인합니다.

* 요청 성공률이 높음
* latency가 안정적임
* retransmit이 거의 없음
* RTT가 안정적임
* CWND가 급격히 감소하지 않음

#### 연결 실패 트래픽

연결 실패 트래픽에서는 잘못된 연결 시도에 따른 실패 지표를 확인합니다.

* 요청 실패율 증가
* connection fail 이벤트 증가
* timeout 증가 가능
* RTT / CWND 변화는 핵심 증거가 아닐 수 있음

#### 패킷 손실 트래픽

패킷 손실 트래픽에서는 TCP 레벨 손실 반응을 확인합니다.

* latency 증가
* retransmit 증가
* RTT 증가
* CWND 감소 가능
* timeout 증가 가능

## 9. 결론

이번 실험의 핵심은 다음과 같습니다.

```text
요청 실패나 지연 증가는 단순 증상이다.
원인을 판단하려면 여러 메트릭을 함께 봐야 한다.
```

정상 트래픽, 연결 실패 트래픽, 패킷 손실 트래픽을 비교하면 같은 “요청 실패”라도 원인이 다르게 나타날 수 있습니다.

연결 실패 트래픽에서는 잘못된 포트나 엔드포인트로 인해 connection fail 관련 지표가 증가할 수 있습니다.

반면 패킷 손실 트래픽에서는 TCP retransmit 증가, RTT 증가, CWND 감소 같은 TCP 레벨 변화가 함께 나타날 수 있습니다.

따라서 NetworkDoctor는 Prometheus 기반 모니터링과 eBPF 기반 커널 레벨 관측을 결합하여, Kubernetes 네트워크 장애를 더 설명 가능하게 진단하는 것을 목표로 합니다.
