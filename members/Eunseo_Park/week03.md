## 2. 책 내용 핵심 요약

1. **트래픽 관리는 관측 가능성의 출발점이다.**

   서비스가 커질수록 장애는 특정 서버 하나의 문제가 아니라 네트워크, 로드 밸런서, 게이트웨이, 서비스 간 호출 흐름 전체에서 발생한다. 따라서 단일 장애점, 로드 밸런싱, API 게이트웨이, 클라이언트 측 부하 분산, 서비스 메시를 이해해야 장애 위치를 추적할 수 있다. Istio 같은 서비스 메시는 트래픽 관리, 보안, 관측 가능성을 애플리케이션 코드 변경 없이 제공하는 인프라 레이어로 설명된다.

2. **복원성 패턴은 장애 전파를 막는 운영 장치다.**

   분산 시스템에서는 일부 서비스가 느려지거나 실패했을 때 그 장애가 전체 시스템으로 번지지 않도록 해야 한다. 이를 위해 재시도, 타임아웃, 비율 제한, 벌크헤드, 서킷 브레이커 같은 패턴을 사용한다. 재시도는 일시적 장애에 대응하지만 과도하면 장애를 키울 수 있고, 서킷 브레이커는 실패 가능성이 높은 호출을 차단해 장애 의존성을 보호한다. 벌크헤드는 장애 영향 범위를 특정 구획으로 제한하는 방식이다.

3. **오토스케일링은 “자동으로 늘린다”가 아니라 “어떤 신호를 믿고 늘릴지 결정한다”의 문제다.**

   쿠버네티스 HPA는 Deployment나 StatefulSet 같은 워크로드의 파드 수를 관측된 메트릭에 맞춰 조정한다. 기본적으로 CPU, 메모리 같은 리소스 메트릭을 사용할 수 있고, 사용자 정의 메트릭이나 외부 메트릭도 사용할 수 있다. KEDA는 큐 길이, 이벤트 수, 요청량 같은 실제 워크로드 이벤트를 기준으로 스케일링할 수 있게 해준다.

4. **관측 가능성 운영 프로세스는 메트릭 → 추적 → 로그 흐름으로 이해할 수 있다.**

   메트릭은 전체 상태와 이상 여부를 빠르게 알려주고, 추적은 문제가 발생한 요청 흐름과 병목 구간을 보여주며, 로그는 마지막 단계에서 구체적인 에러 메시지와 디버깅 정보를 제공한다. 학습 블로그에서도 운영 흐름을 `메트릭 분석 → 추적 분석 → 로그 분석`으로 정리하고, 장애 상황에서는 알람 이후 추적과 로그 분석으로 원인을 좁혀가는 방식으로 설명한다.

5. **대규모 관측 가능성 시스템은 분산 데이터 시스템이다.**

   로그, 메트릭, 트레이스 데이터는 양이 많고 계속 증가하기 때문에 단일 서버나 단일 저장소로는 감당하기 어렵다. 그래서 수평 샤딩, 일관된 해시, 읽기/쓰기 분리, 키-값 저장소, 객체 스토리지, 복제, 쿼럼, 장애 감지 같은 분산 시스템 기술이 필요하다. Grafana Loki와 Mimir도 컴포넌트를 나누어 독립적으로 확장하고, 해시 링과 객체 스토리지를 활용하는 구조를 가진다.

## 3. 중요하다고 생각한 개념

### 개념 1. 트래픽 관리와 서비스 메시

### 개념 설명

트래픽 관리는 사용자의 요청이나 서비스 간 요청이 어느 서버, 어느 파드, 어느 버전의 애플리케이션으로 전달될지 제어하는 기술이다. 단순히 요청을 나누는 로드 밸런싱뿐 아니라, 장애가 있는 인스턴스를 제외하고, 특정 버전으로 일부 트래픽만 보내고, 호출 실패 시 재시도하거나 차단하는 것까지 포함한다.

서비스 메시는 이런 기능을 애플리케이션 코드 내부가 아니라 인프라 계층에서 처리하는 방식이다. Istio는 프록시를 사용해 애플리케이션 인지형 트래픽 관리, 관측 가능성, 보안 기능을 네트워크에 추가한다고 설명한다. 또한 Istio의 관측 가능성 기능은 서비스 간 통신에서 메트릭, 분산 추적, 접근 로그를 생성해 운영자가 서비스 동작을 파악할 수 있게 한다.

### 중요하다고 생각한 이유

마이크로서비스 구조에서는 장애가 하나의 서버에서만 끝나지 않는다. A 서비스가 B 서비스를 호출하고, B 서비스가 DB나 외부 API를 호출하는 식으로 연결되어 있기 때문에 어느 구간에서 지연이 발생했는지 알기 어렵다. 트래픽 관리가 제대로 되어 있지 않으면 장애가 있는 인스턴스에도 계속 요청이 전달되고, 재시도가 폭증하면서 전체 시스템이 더 느려질 수 있다.

### 운영 관점에서의 의미

운영 관점에서 트래픽 관리는 장애 대응의 첫 번째 방어선이다. 로드 밸런서의 헬스 체크, API 게이트웨이의 라우팅과 인증, 서비스 메시의 트래픽 분산과 추적 데이터가 함께 있어야 “어느 서비스가 느린가”, “어느 버전에서 에러가 늘었는가”, “특정 인스턴스에 트래픽이 몰리는가”를 판단할 수 있다. 즉, 관측 가능성을 잘하려면 요청 흐름 자체를 제어하고 관찰할 수 있어야 한다.

### 개념 2. 쿠버네티스 오토스케일링과 메트릭 선정

### 개념 설명

쿠버네티스 오토스케일링은 부하에 따라 파드 수나 리소스를 자동으로 조정하는 기능이다. 대표적으로 HPA는 현재 메트릭 값과 목표 메트릭 값의 비율을 바탕으로 필요한 파드 수를 계산한다. CPU 사용률이 목표보다 높으면 파드를 늘리고, 낮으면 줄이는 식이다. 다만 실제 운영에서는 CPU만으로는 부족할 수 있어 요청 수, 지연 시간, 큐 길이, 에러율 같은 서비스 특화 메트릭도 고려해야 한다.

### 중요하다고 생각한 이유

오토스케일링은 자동화처럼 보이지만, 잘못 설정하면 오히려 장애를 키울 수 있다. 예를 들어 CPU 사용률만 보고 파드를 늘렸는데 새 파드가 뜨는 과정에서 CPU가 더 올라가고, 그 값을 보고 또 파드를 늘리면 스케일링이 반복될 수 있다. 학습 블로그에서도 이런 현상을 스레싱으로 정리하며, 적절한 지표 선택, 쿨다운 기간, 스케일 인/아웃 정책 분리가 중요하다고 설명한다.

### 운영 관점에서의 의미

운영자는 “어떤 메트릭이 실제 부하를 가장 잘 나타내는가”를 먼저 결정해야 한다. 웹 API라면 초당 요청 수, p95/p99 지연 시간, 에러율이 중요할 수 있고, 메시지 큐 기반 서비스라면 큐 길이나 소비 지연 시간이 더 적합할 수 있다. KEDA처럼 이벤트 기반 오토스케일링을 사용하면 Kafka, RabbitMQ, SQS 같은 외부 이벤트 소스를 기준으로 파드를 늘릴 수 있다.

### 개념 3. 수평 샤딩, 일관된 해시, 안정적인 데이터 관리

### 개념 설명

수평 샤딩은 데이터를 여러 서버나 파티션으로 나누어 저장하는 방식이다. 데이터가 많아질수록 하나의 서버가 모든 데이터를 처리하기 어렵기 때문에 데이터를 여러 조각으로 나누어 병렬로 처리한다. 일관된 해시는 샤딩을 구현할 때 자주 쓰이는 방식으로, 노드가 추가되거나 제거될 때 전체 데이터를 다시 옮기지 않고 일부 데이터만 재배치되도록 한다.

Grafana Loki 문서에서도 해시 링은 로그 라인 샤딩, 고가용성, 클러스터 수평 확장과 축소를 돕는 구조로 설명된다. 각 컴포넌트 인스턴스는 링에 참여하고, 상태 정보는 key-value store와 gossip 방식으로 공유될 수 있다.

### 중요하다고 생각한 이유

관측 가능성 시스템은 로그, 메트릭, 트레이스처럼 양이 매우 많은 데이터를 계속 받아야 한다. 장애 상황에서는 오히려 로그와 메트릭이 폭증할 수 있기 때문에, 관측 가능성 도구 자체가 확장성과 안정성을 갖추지 못하면 정작 장애를 분석해야 할 때 관측 시스템이 먼저 무너질 수 있다.

### 운영 관점에서의 의미

운영 관점에서는 관측 가능성 저장소도 하나의 핵심 서비스로 봐야 한다. Loki, Mimir, Tempo 같은 시스템은 단순한 대시보드 도구가 아니라 대량의 데이터를 수집, 저장, 조회하는 분산 시스템이다. 따라서 복제 수, 쿼럼, WAL, 객체 스토리지, 파티션, 쿼리 성능, 보존 기간까지 함께 관리해야 한다. Prometheus도 시계열 데이터를 메트릭 이름과 라벨 조합으로 식별하고, 로컬 TSDB에 블록과 WAL 형태로 저장한다.

## 4. 이해가 어려웠던 부분

### 어려웠던 부분

- 서비스 메시, API 게이트웨이, 로드 밸런서의 역할 차이
- HPA에서 어떤 메트릭을 기준으로 스케일링해야 하는지 판단하는 기준
- 수평 샤딩과 일관된 해시가 실제 데이터 저장 구조에서 어떻게 동작하는지
- 쿼럼, 복제, 가십 프로토콜이 데이터 안정성과 어떤 관계를 가지는지
- 메트릭, 추적, 로그를 장애 상황에서 어떤 순서로 봐야 하는지

### 왜 어려웠는지

> 로드 밸런서, 서비스 메시, 오토스케일링은 각각 따로 보면 이해되지만 실제 운영에서는 서로 연결되어 동작하기 때문에 구조가 복잡하게 느껴졌다. 예를 들어 파드가 오토스케일링으로 늘어나면 트래픽도 새 파드로 분산되어야 하고, 서비스 메시나 로드 밸런서는 새 인스턴스의 상태를 알아야 하며, 관측 가능성 시스템은 이 변화까지 메트릭과 추적으로 보여줘야 한다.
>
>
> 또한 HPA에서 CPU 사용률을 기준으로 스케일링하는 것은 이해하기 쉽지만, 실제 서비스에서는 CPU보다 요청 지연 시간, 큐 길이, 에러율이 더 중요한 경우도 있을 수 있다. 어떤 메트릭이 “좋은 스케일링 기준”인지 판단하려면 애플리케이션의 동작 방식과 병목 지점을 알아야 해서 어렵게 느껴졌다.
>

## 5. 추가로 찾아본 내용

### 추가로 찾아본 키워드

- Kubernetes HPA
- KEDA Event-driven Autoscaling
- Istio Service Mesh
- Resilience Pattern
- Circuit Breaker / Retry / Bulkhead / Rate Limiting
- Grafana Loki, Mimir 간단 Architecture
- Prometheus TSDB

### 찾아본 내용 요약

쿠버네티스 HPA는 Deployment나 StatefulSet 같은 워크로드의 파드 수를 자동으로 조정한다. HPA는 관측된 메트릭과 목표 메트릭의 비율을 기준으로 원하는 replica 수를 계산하며, CPU/메모리 외에도 custom metric을 사용할 수 있다. KEDA는 HPA를 대체하기보다는 외부 이벤트 소스와 연결해 큐 길이, 요청량, 이벤트 수 같은 실제 워크로드 기반 스케일링을 가능하게 해주는 도구로 이해할 수 있다.

Istio 서비스 메시는 서비스 간 통신을 제어하는 인프라 계층으로, 트래픽 관리, 보안, 관측 가능성을 제공한다. Istio 문서에서는 서비스 간 통신에서 메트릭, 분산 추적, 접근 로그를 생성해 운영자가 서비스 동작을 이해하고 문제를 해결할 수 있도록 한다고 설명한다. 이 내용은 책의 2.1 트래픽 관리, 2.1.4 가시성, 2.1.5 서비스 메시와 직접 연결된다.

복원성 패턴에서는 Retry, Circuit Breaker, Bulkhead, Rate Limiting이 특히 중요했다. Retry는 짧은 일시적 장애에 대응할 수 있지만, 무작정 반복하면 장애 서비스를 더 압박할 수 있다. Circuit Breaker는 실패 가능성이 높은 호출을 일정 기간 막아 장애 전파를 줄이고, Bulkhead는 시스템을 구획화해 한 부분의 장애가 전체로 번지지 않게 한다. Rate Limiting은 요청 비율을 제한해 과부하와 무제한 재시도를 막는다.

Grafana Loki는 단일 바이너리, simple scalable mode, microservice mode처럼 여러 배포 모드를 지원하며, Distributor, Ingester, Querier, Query Frontend 같은 컴포넌트로 나누어 동작한다. Loki는 로그 데이터를 객체 스토리지에 저장하고, 해시 링을 통해 샤딩과 고가용성, 수평 확장을 구현한다. Mimir도 여러 컴포넌트가 병렬로 실행되는 마이크로서비스 기반 아키텍처를 가진다.

Prometheus는 메트릭을 시계열 데이터로 저장하며, 각 시계열은 메트릭 이름과 라벨 조합으로 식별된다. 라벨은 같은 메트릭을 다양한 차원으로 나누어 볼 수 있게 해주지만, 라벨 값이 너무 많아지면 시계열 수가 폭증할 수 있다. Prometheus의 로컬 저장소는 블록, chunk, index, WAL 구조를 사용하며, 장기 저장이나 고가용성에는 외부 저장소 또는 Thanos, Mimir 같은 확장 구조가 필요하다는 점을 연결해서 이해할 수 있다.

> 책 2장은 이후 장에서 다룰 Prometheus, Grafana Loki/Mimir/Tempo, OpenTelemetry를 이해하기 위한 기반 기술을 먼저 설명하는 장으로 보인다. 3장에서 Prometheus를 배우려면 시계열 데이터와 메트릭 라벨 구조를 알아야 하고, 4장에서 Grafana 계열 도구를 이해하려면 마이크로서비스 구조, 객체 스토리지, 샤딩, 해시 링, 읽기/쓰기 분리 개념이 필요하다.

## 6. 내 생각, 경험, 질문

### 자유 정리

> 이번 장을 보면서 관측 가능성은 단순히 Grafana 대시보드를 만들거나 Prometheus 메트릭을 수집하는 일이 아니라는 점을 느꼈다. 실제 운영에서는 트래픽이 어디로 흐르는지, 장애가 발생했을 때 어느 구간에서 끊기는지, 부하가 늘었을 때 파드를 어떤 기준으로 늘릴지, 수집된 로그와 메트릭을 어떻게 안정적으로 저장할지가 모두 연결되어 있었다.
>
>
> 개인 프로젝트나 학교 프로젝트에서는 보통 서버가 느려지면 CPU 사용률, 메모리 사용량, 로그 정도만 확인했을 것 같다. 하지만 이번 장을 공부하면서 CPU가 높다는 사실만으로는 원인을 알 수 없고, 요청 수가 갑자기 늘었는지, 특정 API의 지연 시간이 증가했는지, 외부 API 호출이 느려졌는지, DB 응답이 느린지까지 함께 봐야 한다는 점을 알게 되었다.
>
> 특히 오토스케일링 부분이 인상적이었다. 자동으로 파드를 늘려준다는 점만 보면 편리해 보이지만, 잘못된 메트릭을 기준으로 잡으면 스케일링이 반복되거나 비용이 급증할 수 있다. 예를 들어 CPU 사용률만 기준으로 잡으면 애플리케이션 시작 시 CPU가 높아지는 특성 때문에 파드가 계속 늘어날 수 있다. 그래서 오토스케일링은 단순 설정이 아니라 부하 테스트와 운영 경험을 바탕으로 조정해야 하는 영역이라고 생각했다.
>
> 또 하나 중요하다고 느낀 점은 관측 가능성 시스템 자체도 장애에 강해야 한다는 것이다. 장애가 발생했을 때 로그와 메트릭이 폭증할 수 있는데, 이때 관측 가능성 시스템이 감당하지 못하면 원인 분석이 불가능해진다. 그래서 Loki나 Mimir 같은 도구가 마이크로서비스 구조, 객체 스토리지, 해시 링, 복제, 쿼럼 같은 분산 시스템 기술을 사용하는 이유를 이해할 수 있었다.
>
