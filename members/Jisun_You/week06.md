## 읽은 범위
Chapter 5. 그라파나 관측 가능성 데모 (p.259 - )

### 상관관계

#### 메트릭 -> 추적 
- 이그젬플러
- 메트릭 차트에서 특정 시점 클릭 -> 해당 요청 추적 화면으로 이동
 
#### 추적 -> 메트릭 
- 스팬 메트릭
- 템포가 추적 스팬에서 요청 수 / 에러 / 지연시간 메트릭 자동 생성 -> 프로메테우스로 전달
  
#### 로그 <-> 추적 
- Trace ID
- 로그에 Trace ID 포함 시 그라파나에서 로그 -> 추적 화면 이동 가능

#### 메트릭 <-> 로그 
- 표준 x
- 레이블 / 타임스탬프 / 네이밍 규칙으로 직접 구현 필요

### 데모 세 가지

- TNS : 그라파나 공식 데모. 상관관계 전체 흐름 확인
- HotROD : 예거 공식 데모. OTel 점진적 도입 + 카오스 엔지니어링으로 장애 발생 후 원인 파악
- 그라파나 관측 가능성 : 커스텀 메트릭 / 로그 / 추적 직접 개발하는 심화 실습


## 프로젝트 적용 계획

### 프로젝트 개요

- 홈서버에서 운영 중인 Spring Boot + MySQL 기반 CRUD 웹 서비스
- 프로메테우스 / 그라파나 / 로키 연결되어 있으나 제대로 활용 못하는 상태
- Promtail 제거하고 OTel Collector로 로그 / 메트릭 / 추적을 한꺼번에 수집하는 방향으로 변경 예정

### Observability가 필요한 이유

- 도구는 붙여놨는데 뭘 어떻게 봐야 하는지 몰라서 방치 중
- 간헐적 에러 발생해도 원인 파악 방법 없음
- 책에서 배운 내용 실제 서비스에 적용해보고 싶음
