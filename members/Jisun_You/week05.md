## 1. Chapter 4 한 문단 요약

Chapter 4는 그라파나 관측 가능성 스택(LGTM)인 로키·미미르·템포·예거를 실제로 구성하는 챕터이다. 로키는 로그, 미미르는 메트릭, 템포는 추적을 담당하며, 세 도구 모두 프로메테우스 기반으로 디스트리뷰터 + 인제스터 + 콤팩터라는 유사한 마이크로서비스 구조를 공유하고 쓰기/읽기 경로가 분리되어 독립적인 오토스케일링이 가능하다. 예거는 추적 데이터를 시각화하는 역할을 하며, 그라파나가 이 모든 도구를 하나의 대시보드로 연결해 메트릭, 로그, 추적의 상관관계를 시각화한다.

---

## 2. Grafana, Loki, Mimir, Tempo, Jaeger 역할

- **Grafana** — 로키/미미르/템포의 데이터를 하나의 대시보드로 시각화하고, 메트릭·로그·추적의 상관관계를 연결하는 통합 UI
- **Loki** — 로그 데이터를 수집·저장하고 LogQL로 쿼리할 수 있는 로그 관리 도구
- **Mimir** — 프로메테우스 메트릭을 장기 저장하고 PromQL로 쿼리할 수 있는 대용량 메트릭 저장소
- **Tempo** — 분산 추적 데이터를 저장하고, Jaeger/Zipkin/OTLP 등 다양한 추적 포맷을 수신하는 추적 백엔드
- **Jaeger** — 스팬과 트레이스를 시각화해서 요청이 어느 서비스를 거쳐 얼마나 걸렸는지 보여주는 추적 시각화 도구

---

## 3. 관심 있는 주제 — Logs

- 로그는 메트릭과 추적만으로는 알 수 없는 상세한 맥락을 제공하는 신호
- 메트릭으로 이상을 감지하고 추적으로 어느 구간인지 좁힌 뒤, 로그에서 정확히 어떤 에러가 왜 발생했는지 확인하는 것이 관측 가능성의 핵심 흐름
- 이를 위해 로그에는 타임스탬프/서비스명/로그 레벨/Trace ID가 포함되어 있어야 추적과 연결 가능
- 그라파나에서는 로키를 로그 저장소로 사용하며, LogQL로 특정 서비스의 에러 로그나 특정 Trace ID에 해당하는 로그 검색 가능 
- Trace ID를 공통 식별자로 활용하면 메트릭/추적/로그를 하나의 흐름으로 연결해서 장애 원인을 빠르게 찾을 수 있음

## 4. 장애 상황 예시

**장애 상황:** 백엔드 API 서버에서 특정 시간대에 간헐적으로 500 에러가 발생하지만 재현이 되지 않는 상황

1. **그라파나 대시보드 (메트릭)**
   - 미미르에 저장된 API 에러율 메트릭을 그라파나에서 확인
   - 특정 시간대에 에러율이 순간적으로 튀었다가 정상으로 돌아오는 패턴 발견
   - 평소에는 정상이라 재현이 안 됐던 이유가 메트릭으로 처음 보일 것이라고 예상 

2. **그라파나 -> 템포 (추적)**
   - 에러 발생 시점의 추적 화면으로 이동
   - 스팬 목록에서 어느 구간에서 응답 지연이나 에러가 발생했는지 확인
   - 정상 요청과 에러 요청의 스팬 시간을 비교해서 병목 구간 파악

3. **그라파나 -> 로키 (로그)**
   - 템포에서 확인한 Trace ID로 로키에서 해당 요청의 로그 검색
   - `{service="api"} |= "ERROR" | json` 같은 쿼리로 에러 메시지 확인 및 정확한 원인 파악

4. **검증**
   - 원인에 맞게 조치한 뒤 그라파나 대시보드에서 미미르 메트릭을 다시 확인
   - 에러율이 정상으로 유지되는지 검증

## 5. 헷갈린 지점 / 추가로 찾아본 내용

### 헷갈린 지점

- 템포와 예거의 역할이 명확하게 구분이 되지 않음  
  - 둘 다 추적 관련 도구인데 템포가 저장이고 예거가 시각화라는 건 이해했는데, 그라파나도 추적 시각화를 지원한다면 굳이 예거가 필요한 상황이 언제인지 책에 명확히 나오지 않아서 궁금함

### 추가로 찾아본 내용 - 로키/미미르/템포가 왜 같은 구조를 가지는가

#### 읽기 구조
- Distributor : 검증 및 분산  
  - 무상태로 CPU 빠른 네트워크만 필요
    -> 로그 트래픽이 폭주하는 경우  

- Injester : 메모리 버퍼링 및 주입  
    - 메모리 필요 (상태성)  
        -> 보관해야 할 실시간 데이터가 많아지는 경우

- Compactor : 스토리지 저일 및 압축
    - 디스크 정리 (백그라운드 워커)
        -> 저장 공간을 아끼고 싶은 경우 

#### 쓰기 구조
- Query Frontend : 쿼리 작성 인터페이스  
- Querier : 쿼리 처리 

#### 전체 구조
- 그라파나 - 리소스에 따른 도구 - 캐시 - 오브젝트 스토리지 

#### 같은 구조를 가지는 이유 

- 프로메테우스는 단일 서버 설계 -> 규모가 커지면 고가용성·장기 저장에 한계
- 위브웍스(Weaveworks)가 이를 해결하기 위해 코텍스(Cortex) 개발 -> 프로메테우스를 마이크로서비스로 쪼개서 각각 독립 확장 가능하게 설계
- 2018년 CNCF 프로젝트로 채택 -> 그라파나랩이 최대 기여자로 활동
- 그라파나랩이 코텍스에서 검증된 구조를 로그(로키)/추적(템포)에도 그대로 적용 -> 세 도구가 같은 컴포넌트 이름을 쓰는 이유
- 5년간 기여하면서 일부 클라우드 벤더들의 무임승차 문제 발생 -> 2022년 그라파나랩이 미미르(Mimir)를 별도 오픈소스로 분리
- 현재 LGTM 스택(로키/그라파나/템포/미미르)이 이 흐름의 결과물
