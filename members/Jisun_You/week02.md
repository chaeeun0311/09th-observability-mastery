## 2. 책 내용 핵심 요약

1. 관측 가능성은 모니터링과 다름 
   - 모니터링이 "지금 뭔가 이상하다"를 알려준다면, 관측 가능성은 "왜 이상한지, 어디서 문제가 생겼는지"까지 시스템 내부 흐름을 보여줄 수 있음 

2. 메트릭 (Metric) : 이상을 가장 먼저 감지하는 역할  
     - 일정 시간 동안 시스템 상태를 수치로 집계한 것
    - CPU 사용률, 응답시간, 에러율 등 — 이상 징후의 신호를 수치로 가장 먼저 감지

3. 추적 (Tracing) : 어느 서비스가 문제인지 찾는 역할
    - 요청이 어느 서비스를 거쳐 얼마나 걸렸는지 경로를 기록한 것
    - 어디서 느려졌는지, 어디서 에러가 났는지 경로를 좁혀줌

4. 로그 (Log) : 정확한 원인을 확인하는 역할
    - 애플리케이션이 실행되면서 남기는 상세 기록

5. 세 가지 신호를 연결(상관관계)하는 게 진짜 목표 
   - 메트릭, 추적, 로그가 Trace ID 같은 공통 식별자로 연결될 때 관측 가능성의 가치가 생김

## 3. 중요하다고 생각한 개념

### 개념 1. 모니터링과 관측 가능성의 차이

#### 개념 설명

- 모니터링은 시스템 바깥에서 "지금 CPU가 몇 %다, 응답이 왔다 안 왔다" 정도를 확인하는 것
- 관측 가능성은 한 발 더 들어가서, 요청이 시스템 내부에서 어떤 경로로 흘렀는지, 어디서 느려졌는지까지 볼 수 있게 해주는 것  
=> 핵심 차이는 **내부 흐름을 볼 수 있느냐 없느냐**

#### 중요하다고 생각한 이유
 
- 모니터링만으로는 "이상하다"는 건 알아도 "왜 이상한지"는 모른다는 걸 알고 나서, 왜 관측 가능성이라는 개념이 따로 존재하는지 이해하게 되었음

#### 운영 관점에서의 의미

- 관측 가능성이 갖춰지면 추적(Tracing)으로 어느 서비스에서 병목이 생겼는지 바로 찾아갈 수 있음

---

### 개념 2. 세 가지 신호의 상관관계

#### 개념 설명

- 메트릭, 추적, 로그는 각각 다른 역할 수행
- 메트릭 : 수치로 이상을 감지
- 추적 : 어느 서비스가 문제인지 경로를 추적
- 로그 : 정확히 어떤 에러가 왜 났는지 상세 내용을 확인
  
=> 이 세 가지가 Trace ID 같은 공통 식별자로 연결될 때, 메트릭에서 추적으로, 추적에서 로그로 이동하며 원인을 추적하는 흐름이 가능해짐

#### 중요하다고 생각한 이유
 
- 세 가지를 따로 구축하는 건 기존 모니터링이랑 다를 게 없어서 연결이 핵심이라는 점이 이 챕터의 결론이라고 판단

#### 운영 관점에서의 의미

-  장애 대응 시간 감소
-  메트릭에서 이상 감지 -> 추적으로 원인 서비스 파악 -> 로그로 정확한 에러 확인, 이 흐름이 자동화되면 사람이 직접 로그를 뒤지는 시간이 없어짐

## 5. 추가로 찾아본 내용
 
### 추가로 찾아본 키워드
 
#### SLI / SLO / SLA의 실제 적용
 
### 찾아본 내용 요약
 
- LINE 테크 블로그에서 실제 SLI/SLO 도입 사례를 찾아봤다 (신뢰성 향상을 위한 SLI/SLO 도입 1편)

**SLI/SLO 설정 원칙 (LINE 기준)**
- 사용자 관점에서 바라볼 것 : 측정할 수 있는 값이 아니라, 사용자가 중요하게 생각하는 것을 먼저 찾는다
- 최대한 단순할 것 : SLI를 복잡하게 집계하면 시스템 상태를 명확히 반영하지 못한다
- 가능한 적은 수의 SLO를 설정 : 서비스 특성을 잘 확인할 수 있는 최소한의 SLO를 선택
- 100%와 같은 지나친 목표는 설정하지 않는다

**SLO 수치 설정 기준**
- 99.9% → 한 달 기준 장애 허용 시간 40분 (사람이 감지하고 수동 대응 가능한 수준)
- 99.99% → 한 달 기준 장애 허용 시간 4분 (사람이 감지하기엔 짧아서 시스템이 자동 감지·대응해야 하는 수준)
- 99.999% → 한 달 기준 장애 허용 시간 24초 (사람이 감지할 수 없어 시스템으로만 감지 가능한 수준)
- Google에 따르면 99.9%를 99.99%로 올리려면 기존 대비 10배의 노력이 필요하다. 그래서 서비스 특성에 맞게 적절히 설정하는 게 중요하다.

**오류 예산 (Error Budget)**
- SLO를 기준으로 어느 정도의 오류를 허용할 수 있는지 정의한 값
- 예: SLO 99.9% → 한 달에 0.1% = 40분의 오류를 허용
- 오류 예산이 남아있으면 → 신규 기능 개발에 집중
- 오류 예산이 초과되면 → 모든 신규 기능 개발을 멈추고 안정성 회복에 집중
- 개발 팀과 SRE 팀이 서비스 신뢰성에 대해 같은 기준으로 이야기할 수 있게 해주는 도구

**사용자 여정 (Critical User Journey, CUJ)**
- SLI/SLO를 설정하기 전에 먼저 사용자가 서비스를 사용하는 핵심 흐름을 정의해야 한다
- 예: 메시징 서비스라면 "메시지를 전송하는 흐름", 동영상 서비스라면 "영상을 재생하는 흐름"
- CUJ를 정의한 후 각 여정에서 사용하는 API를 찾아 SLI를 측정하는 게 시작점
 
### 책 내용과 연결되는 부분
 
- 책에서 SLO가 관측의 기준이 된다고 했는데, 실제로도 SLO를 먼저 정의하고 그걸 기준으로 모니터링 체계를 구성한다는 걸 확인했다.
- 책에서 "조직·협업 문제가 관측 가능성 실패의 이유"라고 했는데, LINE에서도 SLI/SLO를 팀 전체가 같은 언어로 신뢰성을 이야기하기 위한 도구로 활용한다는 점이 연결됐다.
