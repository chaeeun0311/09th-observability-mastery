## 0.5 이전 주 질문

### APM과 Observability는 어떻게 다른가?
- **APM** : 알려진 또는 예상되는 오류에 대해 대시보드와 알림을 사용하는 관행  
  ->  알고 있는 문제를 감지하고 알림 / 애플리케이션 성능 위주 / 미리 정해놓은 대시보드와 알람 중심  
  =>  Observability 관행의 일부  
- **Observability** : 복잡한 시스템을 이해하고 생각지도 못했던 문제에 대한 답을 발견하는 데 도움이 되는 총체적인 관행  
  ->  예상치 못한 문제까지 찾아낼 수 있음 / 메트릭-트레이스-로그 전체를 연결하여 시스템 전체 관측 

*참고 : https://newrelic.com/kr/blog/apm/apm-vs-observability


### Prometheus exporter와 OpenTelemetry collector는 어떻게 다른가?
- **Prometheus exporter** : 애플리케이션 안에 붙어서 메트릭을 /metrics 엔드포인트로 노출시키는 것. Prometheus가 주기적으로 와서 긁어가는(pull) 방식

- **OpenTelemetry Collector** : 애플리케이션 밖에 있는 독립 서비스. 여러 소스에서 메트릭·추적·로그를 받아서 가공하고 여러 백엔드로 보내주는 중간 허브 역할

*참고 : https://last9.io/blog/opentelemetry-collector-vs-exporter/


### OpenTelemetry가 Metrics/Logs/Traces를 다룬다면 Prometheus는 왜 쓰는가?
- **OpenTelemetry** : 데이터를 수집하고 보내는 표준 (저장소x) ->  수집 표준

- **Promtetheus** : 데이터를 저장하고 PromQL로 쿼리하고 알람을 보내는 것 ->  저장 / 쿼리 / 알림 

=>  OpenTelemetry로 계측하고 추적을 수집하고, Prometheus(또는 Prometheus 호환 백엔드)로 메트릭을 저장하고 알람을 보내는 방식 사용 

*참고 : https://last9.io/blog/opentelemetry-vs-prometheus/

5. 읽기와 쓰기 분리(CQRS)   
   - 그라파나 관측 가능성은 쓰기 경로(디스트리뷰터·인제스터)와 읽기 경로(쿼리어·쿼리 프런트엔드)가 분리
   - 대용량 데이터 처리 시 디스크 IO 병목을 막고 독립적인 오토스케일링이 가능해짐

## 3. 중요하다고 생각한 개념

### 개념 1. 관측 가능성 장애 프로세스

개념 설명

- 장애가 발생하면 먼저 메트릭 알람으로 문제를 감지하고, Trace ID로 추적 화면에서 어느 서비스가 문제인지 좁히고, 로그로 구체적인 에러 원인을 확인하는 순서로 대응
- Chapter 1에서 "메트릭→추적→로그 순으로 범위를 좁힌다"고 배운 것이 실제 장애 대응 절차로 구체화

중요하다고 생각한 이유

- Chapter 1에서 세 가지를 연결해야 한다고 배웠는데, 실제로 장애가 났을 때 어떤 순서로 어떻게 쓰는지가 여기서 처음으로 등장
- 개념이 실무 절차로 연결되는 지점이라 챕터 전체에서 가장 핵심적인 부분이라고 생각함

운영 관점에서의 의미

- 이 순서를 미리 정해두지 않으면 장애가 났을 때 사람마다 다른 방법으로 대응
- 메트릭 알람 규칙, 추적 태그 설계, 로그 표준화가 모두 이 프로세스를 위한 사전 작업이라는 것을 학습


### 개념 2. 서킷 브레이커

#### 개념 설명

- 마이크로서비스 A가 B를 호출할 때, B에 장애가 생기면 A도 함께 응답 불가 상태가 되는 문제를 막는 패턴  
- B로의 호출을 감지하고 문제가 있으면 강제로 차단해서 장애가 전체로 번지지 않게 함  
- 넷플릭스에서 추천 목록 서비스가 장애일 때 일반 목록으로 대체(폴백)하는 것이 대표 사례

#### 중요하다고 생각한 이유

- 마이크로서비스가 많아질수록 서비스 간 의존성도 늘어나는데, 하나의 장애가 전체로 번지는 걸 막는 것이 관측 가능성만큼 중요하다는 걸 알게 됨  
- 관측 가능성은 장애를 감지하고 원인을 찾는 것이고, 서킷 브레이커는 장애가 커지기 전에 차단하는 것이라 둘이 함께 필요

#### 운영 관점에서의 의미

- 서킷 브레이커가 작동했다는 것 자체가 하나의 신호
- 메트릭으로 서킷 브레이커 상태(OPEN/CLOSED)를 모니터링하면 어느 서비스에 문제가 생겼는지 빠르게 감지 가능
- 장애 대응 프로세스의 첫 단계인 메트릭 알람과 자연스럽게 연결


### 개념 3. 메트릭 선정과 오토스케일링

#### 개념 설명

- 오토스케일링이 제대로 작동하려면 파드 수와 직접 연관된 메트릭을 선택 필요
-  CPU는 파드를 늘리면 줄어들기 때문에 적합하지만, 메모리는 파드를 늘려도 줄어들지 않는 경우가 있어서 주의해야 함
-  히스토그램으로 지연시간 분포를 보고 부하가 늘었는지 판단하는 것이 핵심

#### 중요하다고 생각한 이유

-  단순히 "메트릭 보고 파드 늘린다"는 개념은 알겠는데, 어떤 메트릭을 써야 하는지가 전혀 생각해보지 않은 부분이었기에 잘못된 메트릭을 선택하면 오토스케일링이 오히려 장애를 만들 수 있다는 점이 인상적

#### 운영 관점에서의 의미

-  Chapter 1에서 배운 히스토그램이 오토스케일링 메트릭 선정에도 직접 사용
-   지연시간 분포가 특정 버킷으로 쏠리기 시작하면 파드를 늘려야 한다는 신호가 됨
-  메트릭을 단순히 보는 것이 아니라 시스템 자동화의 트리거로 활용한다는 점에서 관측 가능성의 실제 가치가 드러남

## 5. 추가로 찾아본 내용

### (1) 서킷 브레이커와 관측 가능성의 연계
책 2.1.3에서 서킷 브레이커가 장애 전파를 막는 패턴으로 나오는데, 서킷 브레이커 상태(OPEN/CLOSED)가 어떻게 메트릭과 연결되는지 구체적인 설명이 없어서 검색

URL: [우아한형제들 기술블로그 — 개발자 의식의 흐름대로 적용해보는 서킷브레이커](https://techblog.woowahan.com/15694/)

배민에서 Resilience4j로 서킷 브레이커를 적용한 실제 사례

- 서킷 브레이커는 CLOSED → OPEN → HALF_OPEN 세 가지 상태를 가지며, 실패율이 설정한 임계값을 초과하면 자동으로 OPEN 상태로 전환
- OPEN 상태에서는 연결된 서비스 호출을 차단하고 폴백(fallback) 응답을 반환함
우아한형제들은 전사적으로 그라파나를 사용하는데, Resilience4j가 서킷 브레이커 상태 전환 이벤트를 메트릭으로 발행하고 이를 그라파나 대시보드에서 시각화  

=>  즉, 서킷 브레이커 상태 자체가 관측 가능성의 메트릭 신호가 됨

URL: [Grafana Labs 공식 — Spring Boot Resilience4j Circuit Breaker Dashboard](https://grafana.com/grafana/dashboards/21307-circuit-breaker/)

Resilience4j 서킷 브레이커를 그라파나로 모니터링하는 공식 대시보드 설명

- `resilience4j_circuitbreaker_calls_seconds_max` : 호출 실패율  
  ->  높으면 다운스트림 서비스에 문제가 있음을 의미
- 느린 호출 비율 : 성능 저하를 조기에 감지하는 데 활용
- 분당 실패 호출 수 : 최근 장애 급증 여부를 확인

=>  서킷 브레이커는 장애를 차단하는 동시에 메트릭을 발행해서 관측 가능성의 알람 시스템과 직접 연결


### (2) 그라파나에서의 일관된 해시 실제 적용

책 2.6에서 일관된 해시가 샤드 추가·제거 시 데이터 재배치를 최소화한다고 설명하는데, 실제로 그라파나가 어떻게 쓰는지 구체적으로 나오지 않아서 검색

URL: [Grafana Labs 공식 블로그 — How we improved ingester load balancing in Grafana Mimir](https://grafana.com/blog/how-we-improved-ingester-load-balancing-in-grafana-mimir-with-spread-minimizing-tokens/)

그라파나 미미르가 일관된 해시를 실제로 어떻게 쓰는지, 그리고 어떤 문제를 겪었는지 공식적으로 설명한 글

- 미미르의 인제스터(Ingester)는 해시 링에 등록되어 있으며, 각 시계열 데이터가 어느 인제스터에 저장될지 일관된 해시로 결정
- 그런데 기존 일관된 해시 알고리즘이 인제스터 간 부하를 불균등하게 분배하는 문제가 발생  
  - 최대 25%까지 부하 차이가 나서 일부 인제스터는 과부하, 일부는 유휴 상태  

- 이를 해결하기 위해 "spread-minimizing token generation strategy"라는 새 알고리즘을 개발
- 책에서 "유명인사 문제(핫스폿)"로 설명한 내용이 실제 그라파나 운영에서도 발생한 사례

URL: [Grafana Labs 공식 문서 — Grafana Loki consistent hash rings](https://grafana.com/docs/loki/latest/get-started/hash-rings/)

로키가 해시 링을 어떻게 구성하는지 공식 문서로 설명

- 로키의 디스트리뷰터, 인제스터 등 컴포넌트들이 각각 독립적인 해시 링을 구성
    ```
    로그 데이터 유입
        ↓
    디스트리뷰터가 받음
        ↓
    해시 링으로 어느 인제스터가 담당인지 계산
        ↓
    해당 인제스터로 전달
        ↓
    인제스터가 메모리에 저장하다가 객체 스토리지로 영구 저장
    ```
- 각 노드는 키-값 저장소를 가지고 있으며 주기적으로 업데이트해서 모든 노드 간 정보를 일관되게 유지
- 인스턴스가 추가되거나 제거될 때 일관된 해시 덕분에 최소한의 데이터만 이동
- 책에서 로키 / 미미르 / 템포가 모두 샤딩과 일관된 해시를 쓴다고 나오는 내용이 여기서 구체적으로 확인
