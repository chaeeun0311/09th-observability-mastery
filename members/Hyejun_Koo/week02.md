## 2. 책 내용 핵심 요약

이번 주 책 범위에서 중요하다고 생각한 내용을 3~5개로 정리해주세요.

1. **관측 가능성은 시스템 외부로 드러나는 신호를 통해 내부 상태를 이해하는 능력이다.**  
   관측 가능성은 단순히 서버가 살아 있는지 확인하는 수준을 넘어, 메트릭, 로그, 추적 등의 신호를 활용해 시스템 내부에서 어떤 일이 일어나고 있는지 추론하고 문제의 원인을 파악하는 개념이다. 모니터링이 이미 정의된 지표를 관찰하는 활동에 가깝다면, 관측 가능성은 예상하지 못한 문제까지 분석할 수 있도록 시스템을 설계하는 데 초점이 있다.

2. **메트릭, 로그, 추적은 각각 다른 역할을 가지며 함께 사용해야 한다.**  
   메트릭은 시스템 상태를 수치로 빠르게 파악하게 해주고, 로그는 특정 시점에 어떤 일이 발생했는지 세부 맥락을 제공한다. 추적은 하나의 요청이 여러 서비스와 컴포넌트를 거치는 과정을 시간 순서대로 보여준다. 따라서 하나의 신호만 보는 것보다 여러 신호를 연결해서 해석하는 것이 중요하다.

3. **운영에서는 모든 정보를 보는 것보다 중요한 지표에 집중하는 것이 더 중요하다.**  
   시스템에서 발생하는 모든 데이터를 대시보드에 넣는다고 좋은 모니터링이 되는 것은 아니다. 응답 시간, 에러율, 트래픽, 리소스 사용률처럼 서비스 품질과 장애 판단에 직접 연결되는 핵심 지표를 우선적으로 봐야 한다. 즉, 많이 수집하는 것보다 무엇을 기준으로 볼 것인지가 중요하다.

4. **좋은 대시보드는 계층적으로 설계되어야 한다.**  
   한 화면에 너무 많은 정보를 넣기보다는, 전체 상태를 빠르게 파악할 수 있는 상위 수준의 대시보드와 세부 원인을 분석하기 위한 특화 대시보드를 구분해야 한다. 상위 대시보드는 서비스의 건강 상태를 보여주고, 세부 대시보드는 CPU, 메모리, API별 응답 시간, 에러 로그 등 구체적인 원인 분석에 활용될 수 있다.

5. **운영자는 도구 사용법뿐만 아니라 문제를 해석하는 기준을 가져야 한다.**  
   Prometheus, Grafana, CloudWatch 같은 도구를 사용하는 것도 중요하지만, 더 중요한 것은 어떤 지표가 장애를 의미하는지, 어떤 기준으로 알람을 설정해야 하는지, 여러 신호를 어떻게 연결해서 원인을 찾을 것인지를 판단하는 능력이다.

## 3. 중요하다고 생각한 개념

이번 주 내용 중 중요하다고 생각한 개념 2~3개를 골라 정리해주세요.

### 개념 1. 메트릭과 로그

#### 개념 설명

> 메트릭은 일정 시간 동안 측정된 시스템 상태나 이벤트를 수치화한 데이터이다. CPU 사용률, 메모리 사용량, 요청 수, 응답 시간, 에러율 등이 대표적인 예시이다. 메트릭은 주로 시간의 흐름에 따라 변화하는 값을 그래프로 시각화하여 시스템의 전반적인 상태를 빠르게 파악하는 데 사용된다. 메트릭의 유형에는 카운터, 게이지, 요약, 히스토그램 등이 있다.
>
> 로그는 애플리케이션이나 시스템이 실행되는 과정에서 발생한 이벤트를 텍스트 형태로 기록한 것이다. 메트릭이 전체적인 상태를 수치로 보여준다면, 로그는 특정 시점에 어떤 요청이 들어왔고, 어떤 에러가 발생했으며, 어떤 동작이 수행되었는지에 대한 세부적인 맥락을 제공한다.

#### 중요하다고 생각한 이유

> 메트릭과 로그는 모니터링을 이해하기 위한 가장 기본적인 신호라고 생각했다. 메트릭을 통해 문제가 발생했는지 빠르게 확인할 수 있고, 로그를 통해 왜 문제가 발생했는지 더 자세히 확인할 수 있기 때문이다. 따라서 두 개념을 제대로 이해해야 이후 알람 설정, 대시보드 구성, 장애 분석을 할 수 있다고 느꼈다.

#### 운영 관점에서의 의미

> 운영 관점에서 메트릭은 시스템 이상 징후를 빠르게 감지하는 데 중요하다. 예를 들어 CPU 사용률이 급격히 증가하거나 API 응답 시간이 길어지면 장애 가능성을 조기에 파악할 수 있다. 반면 로그는 장애가 발생한 뒤 원인을 분석하는 데 중요하다. 특정 API에서 에러가 발생했을 때 로그를 확인하면 어떤 요청에서 문제가 생겼는지, 어떤 예외가 발생했는지 추적할 수 있다.
>
> 즉, 메트릭은 장애를 발견하는 데 강점이 있고, 로그는 장애의 원인을 파악하는 데 강점이 있다. 실제 운영에서는 두 신호를 함께 보면서 문제를 판단해야 한다.

---

### 개념 2. 추적과 스팬

#### 개념 설명

> 추적은 하나의 요청이나 트랜잭션이 시스템 내부에서 어떤 경로를 거쳐 처리되는지 확인하는 개념이다. 특히 마이크로서비스 환경에서는 하나의 사용자 요청이 여러 서비스, 데이터베이스, 외부 API를 거칠 수 있기 때문에 단순 로그만으로 전체 흐름을 파악하기 어렵다. 이때 추적을 사용하면 요청이 어떤 서비스들을 거쳤고, 각 단계에서 얼마나 시간이 걸렸는지 확인할 수 있다.
>
> 스팬은 추적을 구성하는 기본 단위이다. 하나의 스팬은 특정 작업 구간을 의미하며, 시작 시간, 종료 시간, 걸린 시간, 관련 태그, 로그, 부모 스팬과의 관계 등을 포함할 수 있다. 여러 스팬이 연결되면 하나의 전체 추적이 된다.

#### 중요하다고 생각한 이유

> 추적과 스팬은 기존에 익숙했던 메트릭이나 로그보다 생소한 개념이었다. 하지만 마이크로서비스처럼 여러 컴포넌트가 연결된 시스템에서는 요청 하나가 어디서 지연되었는지, 어느 서비스에서 문제가 생겼는지를 파악하는 것이 매우 중요하다. 따라서 추적은 복잡한 시스템을 운영할수록 반드시 이해해야 하는 개념이라고 생각했다.

#### 운영 관점에서의 의미

> 운영 관점에서 추적은 장애 원인을 좁혀가는 데 큰 도움이 된다. 예를 들어 사용자는 단순히 서비스가 느리다고 느끼지만, 실제 원인은 API 서버, 인증 서버, 데이터베이스, 외부 결제 API 중 하나일 수 있다. 추적을 사용하면 요청이 각 구간에서 얼마나 지연되었는지 확인할 수 있기 때문에 병목 지점을 더 빠르게 찾을 수 있다.
>
> 특히 마이크로서비스 환경에서는 서비스 간 의존성이 복잡하기 때문에, 추적이 없으면 로그를 하나씩 찾아보며 원인을 추측해야 한다. 반면 추적이 있으면 요청 흐름을 시간 순서대로 볼 수 있어 장애 분석 시간을 줄일 수 있다.

---

### 개념 3. 대시보드와 알람 설계

#### 개념 설명

> 대시보드는 메트릭, 로그, 추적 등에서 얻은 정보를 운영자가 한눈에 볼 수 있도록 시각화한 화면이다. 그러나 좋은 대시보드는 단순히 많은 그래프를 모아둔 것이 아니라, 운영자가 현재 시스템 상태를 빠르게 판단할 수 있도록 중요한 정보를 구조화한 것이다.
>
> 알람은 특정 지표가 정해진 기준을 넘었을 때 운영자에게 문제 가능성을 알려주는 장치이다. 알람 기준은 너무 민감하면 불필요한 알람이 많아지고, 너무 느슨하면 실제 장애를 놓칠 수 있기 때문에 적절한 기준 설정이 중요하다.

#### 중요하다고 생각한 이유

> 실제 운영에서는 데이터를 수집하는 것만큼이나 데이터를 어떻게 보여주고, 어떤 기준으로 위험하다고 판단할지가 중요하다고 느꼈다. 특히 대시보드와 알람은 운영자가 시스템 상태를 판단하는 첫 번째 도구이기 때문에, 잘못 설계하면 오히려 문제를 늦게 발견하거나 불필요한 알람에 피로해질 수 있다.

#### 운영 관점에서의 의미

> 운영 관점에서 대시보드는 장애 상황에서 가장 먼저 확인하는 화면이 될 수 있다. 따라서 상위 대시보드에서는 서비스 전체 상태를 빠르게 보여주고, 세부 대시보드에서는 원인 분석에 필요한 정보를 제공해야 한다.
>
> 또한 알람은 단순히 CPU 사용률 같은 리소스 지표만 기준으로 삼기보다, 사용자 경험과 직접 연결되는 응답 시간, 에러율, 요청 실패율 등을 함께 고려해야 한다. 결국 좋은 모니터링은 단순히 시스템 내부 상태를 보는 것이 아니라, 사용자가 실제로 서비스를 정상적으로 이용하고 있는지를 판단할 수 있어야 한다.

## 4. 이해가 어려웠던 부분

책을 읽으면서 이해가 어려웠던 개념, 단어, 문장, 구조를 적어주세요.

### 어려웠던 부분

- Prometheus의 PromQL 계산식
- 스팬과 추적 구조
- 메트릭, 로그, 추적을 실제 장애 분석에서 어떻게 연결하는지

### 왜 어려웠는지

> PromQL은 단순히 지표 이름을 조회하는 것이 아니라, rate, sum, avg 같은 함수와 조건식을 조합해서 원하는 지표를 계산하는 방식이기 때문에 처음 볼 때는 수식처럼 느껴졌다. 특히 실제 Prometheus 환경에서 쿼리를 작성해본 경험이 많지 않아, 각 계산식이 어떤 상황에서 필요한지 바로 와닿지 않았다.
>
> 스팬도 이번에 처음 접한 개념이라 이해하는 데 시간이 걸렸다. 하나의 요청이 여러 서비스로 나뉘어 처리되고, 각 처리 구간이 스팬으로 기록되며, 이 스팬들이 부모-자식 관계로 연결된다는 구조가 처음에는 복잡하게 느껴졌다. 단순히 로그처럼 한 줄씩 남는 것이 아니라, 요청의 흐름과 시간 정보, 의존 관계를 함께 표현한다는 점이 생소했다.
>
> 또한 메트릭, 로그, 추적을 각각 따로 이해하는 것은 가능했지만, 실제 장애 상황에서 이 세 가지를 어떤 순서로 확인하고 어떻게 연결해서 원인을 찾는지는 아직 더 연습이 필요하다고 느꼈다.

## 5. 내 생각, 경험, 질문

이번 주 개념을 내 경험, 프로젝트, 실무, 인턴, 학교 프로젝트, 개인 프로젝트와 연결해본 내용

### 자유 정리

> 인턴 생활 중 숙박세일페스타라는 행사를 경험한 적이 있다. 오전 10시에 쿠폰이 발급되면서 사용자가 급격히 몰리는 이벤트였기 때문에, 평소보다 이른 시간에 출근하여 AWS 콘솔에서 Auto Scaling이 설정된 EC2 인스턴스가 정상적으로 scale out되었는지 확인했다. 또한 CloudWatch를 통해 API의 CPU 사용률, 트래픽, 사용자 유입량 등을 확인하고, 필요한 경우 로그도 함께 살펴보았다.
>
> 당시에는 서버가 잘 늘어났는지, CPU 사용률이 너무 높지는 않은지, 요청이 몰리고 있지는 않은지를 확인하는 정도로만 모니터링을 이해하고 있었다. 그런데 이번 책 범위를 읽으면서 CloudWatch 대시보드에 보이는 값들이 결국 메트릭을 기반으로 구성된 것이며, 운영자가 어떤 지표를 중요하게 볼 것인지 기준을 정해야 의미 있는 모니터링이 된다는 점을 알게 되었다.
>
> 특히 모든 지표를 무작정 보는 것이 아니라, 서비스 품질과 연결되는 핵심 지표를 SLI로 삼고 이를 기준으로 대시보드와 알람을 설계할 수 있다는 점이 인상 깊었다. 예를 들어 단순히 CPU 사용률만 보는 것이 아니라, 쿠폰 발급 API의 응답 시간, 실패율, 요청량, 에러 로그 등을 함께 봐야 실제 사용자 경험에 가까운 모니터링이 가능하다는 것을 이해했다.
>
> 앞으로 직접 모니터링 관련 프로젝트를 진행할 때는 단순히 Grafana나 CloudWatch 같은 도구를 붙이는 것에 그치지 않고, 어떤 지표를 핵심 지표로 삼을지, 어떤 기준에서 알람을 발생시킬지, 장애 상황에서 어떤 순서로 대시보드를 확인할지까지 함께 설계해야겠다고 느꼈다.

## 7. 발표용 요약

발표할 때 아래 순서대로 말하면 됩니다.

### 발표 흐름

1. 이번 주 책 범위 소개

   > 이번 주에는 모니터링과 관측 가능성의 기본 개념, 그리고 메트릭, 로그, 추적과 같은 핵심 신호에 대해 학습했다.

2. 핵심 내용 3가지 요약

   > 첫 번째로, 관측 가능성은 시스템 외부로 드러나는 신호를 통해 내부 상태를 이해하는 능력이다.
   >
   > 두 번째로, 메트릭, 로그, 추적은 각각 다른 역할을 가지며 함께 사용해야 한다.
   >
   > 세 번째로, 좋은 모니터링은 많은 정보를 보여주는 것이 아니라 중요한 지표를 기준에 맞게 구성하는 것이다.

3. 중요하다고 생각한 개념 1~2개 설명

   > 중요하다고 생각한 개념은 메트릭과 추적이다. 메트릭은 시스템 상태를 수치로 빠르게 확인하게 해주고, 추적은 하나의 요청이 여러 서비스와 컴포넌트를 거치는 과정을 보여준다. 특히 복잡한 시스템에서는 단순히 CPU 사용률이나 로그만 보는 것보다, 요청 흐름 전체를 파악하는 것이 중요하다고 느꼈다.

4. 이해가 어려웠던 부분 또는 추가로 찾아본 내용 공유

   > 어려웠던 부분은 PromQL 계산식과 스팬 개념이었다. PromQL은 단순 조회가 아니라 여러 함수와 조건을 조합해서 지표를 계산하는 방식이라 처음에는 수식처럼 느껴졌다. 스팬은 하나의 요청 안에서 발생하는 작업 단위이고, 여러 스팬이 연결되어 전체 추적을 만든다는 점이 생소했다.

5. 내 생각, 경험, 질문 중 하나를 골라 공유

   > 인턴 중 숙박세일페스타 이벤트를 모니터링한 경험이 있다. 당시에는 AWS Auto Scaling이 정상적으로 되었는지, CloudWatch에서 CPU 사용률과 트래픽이 어떤지 확인하는 정도였다. 하지만 이번 내용을 통해 이런 대시보드가 단순한 그래프 모음이 아니라, 어떤 지표를 중요하게 볼 것인지에 따라 운영 품질이 달라진다는 점을 이해했다.

6. 같이 이야기해보고 싶은 지점 제시

   > 다양한 지표를 수집할 수 있는 상황에서, 어떤 지표를 핵심 지표로 삼고 어떤 방식으로 조합해야 실제 장애 대응에 가장 효과적일지 이야기해보고 싶다.

## 발표 핵심 메시지

> 모니터링에서 중요한 것은 단순히 많은 정보를 수집하는 것이 아니라, 시스템 상태와 사용자 경험을 판단할 수 있는 핵심 신호를 정하고 이를 적절히 조합해 해석하는 것이다. 정보가 아무리 많아도 기준 없이 나열되면 운영에 도움이 되지 않으며, 좋은 관측 가능성은 문제를 빠르게 발견하고 원인을 추적할 수 있도록 정보를 구조화하는 데서 시작된다.
