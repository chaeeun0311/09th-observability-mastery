## 2. Chapter 4 한 문단 요약

Chapter 4는 오픈소스 기반 관측 가능성 도구인 Grafana와 LGTM 스택을 중심으로, 서비스 운영에서 로그, 메트릭, 트레이스를 어떻게 수집하고 연결해 볼 수 있는지 설명한다. 오픈소스 관측 가능성의 장점은 비용을 줄일 수 있고, 필요에 따라 소스를 수정하거나 확장할 수 있다는 점이다. Grafana는 대시보드를 통해 여러 데이터를 시각화하고, Loki는 로그, Mimir는 메트릭, Tempo는 트레이스를 담당한다. 이 조합을 LGTM 스택이라고 하며, 운영자는 이를 통해 단순 모니터링을 넘어 이상 탐지, 상관관계 분석, 근본 원인 분석까지 이어갈 수 있다. 특히 Kubernetes 환경처럼 Pod가 계속 생성되고 사라지는 구조에서는 개별 서버를 직접 확인하는 방식보다, Grafana 기반으로 전체 흐름을 보는 것이 중요하다고 느꼈다.

## 4. 책 내용 핵심 요약

1. **Grafana는 단순 대시보드가 아니라 관측 가능성의 중심이다.**
   Grafana는 여러 데이터를 그래프로 보여주는 도구이지만, 핵심은 시각화 자체가 아니라 운영자가 서비스 상태를 판단할 수 있게 돕는 데 있다. 책의 실습에서는 Helm을 이용해 Kubernetes 환경에 Grafana를 설치하고, 내부적으로 Docker 이미지를 참조해 Pod를 생성하는 방식으로 구성한다.

2. **LGTM 스택은 운영 데이터를 역할별로 나누어 수집한다.**
   Loki는 로그, Grafana는 대시보드, Tempo는 트레이스, Mimir는 메트릭을 담당한다. Metrics는 “문제가 발생했는가”, Logs는 “무슨 일이 있었는가”, Traces는 “어디에서 지연되었는가”를 확인하는 데 가깝다. 중요한 것은 도구 이름을 외우는 것이 아니라, 장애 상황에서 이 데이터를 연결해 보는 것이다.

3. **Loki는 Kubernetes 환경에 맞는 로그 관리 도구이다.**
   기존 로그 관리에서는 Elasticsearch가 많이 사용되었지만, Loki는 로그 본문 전체를 색인하기보다 label을 중심으로 조회한다. 이 방식은 저장 비용과 운영 부담을 줄이는 데 유리하다. 또한 레플리케이션 팩터와 리밸런싱 기능을 제공해 더 신뢰성 있는 로그 관리 구성을 지원한다.

4. **Prometheus의 한계를 Mimir가 보완한다.**
   Prometheus는 메트릭 수집과 조회에는 강력하지만, 단독으로는 장기 저장, 고가용성, 대규모 확장에 한계가 있다. 디스크 용량이나 메모리 부족으로 데이터 유실이나 장애가 발생할 수 있기 때문이다. Mimir는 마이크로서비스 아키텍처 기반으로 병렬 실행과 수평 확장을 지원해 이러한 한계를 보완한다.

## 5. 중요하다고 생각한 개념

### 개념 1. LGTM 스택

#### 개념 설명

LGTM 스택은 Loki, Grafana, Tempo, Mimir를 함께 사용하는 오픈소스 관측 가능성 구성이다. Loki는 Logs, Grafana는 Dashboard, Tempo는 Traces, Mimir는 Metrics를 담당한다. Metrics는 시스템 상태를 수치로 보여주고, Logs는 특정 시점에 어떤 일이 발생했는지 보여주며, Traces는 하나의 요청이 여러 서비스 사이를 지나가며 어디에서 지연되었는지 보여준다.

#### 중요하다고 생각한 이유

서비스 장애는 하나의 데이터만 보고 원인을 찾기 어렵다. CPU 사용률이 높다고 해서 CPU 자체가 원인이라고 단정할 수 없고, 요청 증가, 메모리 부족, DB 지연, 외부 API 장애, 애플리케이션 오류가 함께 연결되어 있을 수 있다. LGTM 스택은 이런 데이터를 따로 보는 것이 아니라 하나의 흐름으로 연결해 볼 수 있게 해준다는 점에서 중요하다.

#### 운영 관점에서의 의미

운영자는 먼저 Metrics로 이상 징후를 확인하고, Logs로 구체적인 에러를 찾고, Traces로 요청 흐름의 병목을 확인할 수 있다. 즉, LGTM 스택은 단순 모니터링 도구가 아니라 장애 대응과 근본 원인 분석을 위한 운영 기반이라고 볼 수 있다.

### 개념 2. Loki 아키텍처와 해시 링

#### 개념 설명

Loki는 로그 데이터를 안정적으로 처리하기 위해 Distributor, Ingester, Query Frontend 같은 컴포넌트로 나누어져 있다. Distributor는 들어오는 로그 스트림을 받아 적절한 Ingester로 보내고, Ingester는 로그 데이터를 메모리에 보관하다가 장기 스토리지에 저장한다. Query Frontend는 사용자의 조회 요청을 조정하고 대기열에 넣어 Queryer에게 분배한다.

일관된 해시 링은 데이터를 여러 서버에 나누어 배치하기 위한 방식이다. 쉽게 말하면 여러 Ingester를 하나의 원형 공간에 배치하고, 로그 스트림이 어느 Ingester로 갈지 해시값으로 결정하는 구조이다. 노드가 추가되거나 빠져도 전체 데이터를 다시 나누지 않고 일부만 이동하면 되기 때문에 수평 확장과 장애 대응에 유리하다.

#### 중요하다고 생각한 이유

Loki는 단순히 로그를 저장하는 도구가 아니라, 대량의 로그를 계속 받아들이고 동시에 많은 조회 요청을 처리해야 하는 분산 시스템이다. 이때 쓰기 경로와 읽기 경로가 분리되어 있고, 해시 링을 통해 데이터를 분산하기 때문에 확장성과 고가용성을 확보할 수 있다.

#### 운영 관점에서의 의미

운영 중 특정 Ingester에 문제가 생기거나 노드를 추가해야 할 때 해시 링과 리밸런싱 구조가 있으면 로그 데이터를 더 안정적으로 분산할 수 있다. 또한 Query Frontend가 큰 쿼리를 바로 실행하지 않고 큐에 넣어 분배하면, 특정 Queryer에 부하가 몰리거나 OOM이 발생하는 위험을 줄일 수 있다. OOM은 Out Of Memory의 약자로, 메모리 부족으로 프로세스나 컨테이너가 종료되는 상황이다.

| Memcached | 멀티스레드를 지원하는 메모리 기반 캐시이다. Redis보다 단순한 캐시 용도에 집중한다.                                             |

## 7. 장애 상황 예시와 확인 순서

### 장애 상황 예시

실제 서비스 운영 환경에서 특정 Pod의 CPU 사용률이 90% 이상이라는 알림이 왔다. Grafana에서 확인해보니 CPU와 메모리 사용량이 순간적으로 튀었고, 해당 Pod가 계속 재시작되는 현상이 발생했다. 이 상황에서 시니어는 당장 원인을 완전히 분석하기보다, 우선 메모리 리소스를 높여 재배포하는 것이 낫다고 판단했다.

이 사례가 인상 깊었던 이유는 장애 대응에서는 원인을 찾는 것만큼 서비스 영향을 줄이는 임시 완화 조치도 중요하다는 점을 알 수 있었기 때문이다. Pod 재시작이 반복되면 사용자는 계속 장애를 겪을 수 있고, 로그를 충분히 확인하기 전에 컨테이너가 다시 죽을 수도 있다. 따라서 우선 리소스를 높여 서비스를 안정화한 뒤, 이후 Metrics와 Logs를 바탕으로 원인을 분석하는 접근이 현실적이라고 느꼈다.

### 확인 순서

1. **Alert와 Dashboard로 장애 범위 확인**
   Grafana Alert에서 CPU 90% 이상 알림이 발생한 시점과 대상 Pod를 확인한다. 이후 Dashboard에서 해당 Pod만 문제인지, 같은 Deployment의 다른 Pod도 영향을 받는지 확인한다.

2. **Metrics로 CPU, Memory, Restart Count 확인**
   CPU 알림이 왔다고 해서 CPU만 보면 안 된다. Pod가 재시작되고 있다면 메모리 부족으로 인한 OOM, Health Check 실패, 애플리케이션 오류 가능성도 함께 봐야 한다.

3. **Kubernetes 이벤트와 Pod 상태 확인**
   Pod가 반복적으로 재시작된다면 `OOMKilled`, `CrashLoopBackOff`, `Liveness probe failed` 같은 이벤트가 있는지 확인해야 한다. Metrics만으로는 재시작 원인을 확정하기 어렵기 때문에 Pod 상태 확인이 중요하다.

4. **Logs로 재시작 직전 에러 확인**
   Loki를 통해 재시작 직전 시간대의 로그를 확인한다. 애플리케이션 예외, 메모리 부족 메시지, 특정 요청 폭증, DB 연결 지연, 외부 API 호출 실패 로그가 있었는지 확인한다.

5. **Traces로 특정 요청 병목 여부 확인**
   특정 API 호출 이후 CPU와 메모리가 급증했다면 Tempo나 Jaeger로 요청 흐름을 확인한다. 특정 API에서 DB 조회가 오래 걸렸는지, 외부 API 응답 지연으로 요청이 쌓였는지, 한 요청이 과도한 연산을 수행했는지 확인할 수 있다.

6. **임시 완화와 근본 원인 분석 분리**
   메모리를 높여 재배포하는 것은 근본 해결이라기보다 우선 서비스 영향을 줄이기 위한 완화 조치에 가깝다. 이후에는 요청량 증가 때문인지, 특정 API의 비효율적인 로직 때문인지, 리소스 request와 limit 설정이 낮았는지 추가로 분석해야 한다.

## 8. 이해가 어려웠던 부분

### 어려웠던 부분

* 일관된 해시 링 구조


### 왜 어려웠는지

일관된 해시 링은 단순히 데이터를 나누어 저장한다는 개념은 이해할 수 있었지만, 실제 Loki 클러스터에서 로그 스트림이 어떤 기준으로 특정 Ingester에 배치되는지는 바로 와닿지 않았다. 한두 줄로 정리하면, 해시 링은 데이터를 여러 노드에 안정적으로 나누기 위한 분산 방식이고, 노드가 추가되거나 빠져도 전체 데이터를 다시 옮기지 않고 일부만 이동하게 해주는 구조이다.

## 9. 내 생각, 경험, 질문

최근 공통 클러스터 구축 새벽 작업을 진행하면서 새벽 2시에 도메인에 연결된 ALB 값을 바꾸고, Grafana 모니터링을 확인하는 업무를 맡았다. 이미 EKS 환경에는 LGTM 기반 모니터링 구성이 되어 있었고, 책에서 배운 Metrics, Traces, Logs를 실제 화면에서 확인할 수 있었다.

당시에는 Grafana를 완벽히 다루는 수준은 아니었고, 트래픽이 정상적으로 들어오는지, 로그가 잘 수집되는지, 서비스 상태가 정상으로 보이는지를 중심으로 확인했다. 그래도 책에서 배운 개념이 실제 운영 화면과 연결되니, 관측 가능성이 단순 이론이 아니라 배포 이후 서비스 안정성을 확인하는 데 필요한 기반이라는 것을 체감했다.

또한 실제 서비스 운영 환경에서 CPU 사용률 90% 이상 알림이 오고, CPU와 메모리 사용량이 튀면서 Pod가 반복적으로 재시작되는 상황을 보게 되었다. 이 경험을 통해 CPU 알림이 왔다고 CPU만 보면 안 되고, Memory, Restart Count, Kubernetes 이벤트, Logs를 함께 봐야 실제 원인에 가까워질 수 있다는 점을 느꼈다.

앞으로는 졸업프로젝트에도 모니터링을 직접 붙여보면서, 단순히 “그래프가 보인다”에서 끝나는 것이 아니라 “이 그래프를 보고 어떤 운영 판단을 할 수 있는가”까지 연습해보고 싶다.

---



### 발표 핵심 메시지

Grafana와 LGTM 스택은 단순히 그래프를 보여주는 도구가 아니라, 장애 상황에서 데이터를 연결해 원인을 좁히고 운영 판단을 내리게 해주는 기반이다.
