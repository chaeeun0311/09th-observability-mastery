# handDoc 프로젝트에 옵저버빌리티 적용하기

## 1. 적용해보고 싶은 프로젝트 개요

handDoc은 청각장애인을 위한 대면·비대면 진료 플랫폼이다. 환자의 수어를 실시간으로 텍스트로 변환하고, 의사의 음성은 자막으로 제공한다. 또한 구음을 사용하는 환자의 음성은 교정 후보 문장으로 제시하며, WebRTC 기반 화상 진료와 진료 후 GPT 요약 기능까지 제공한다.

인프라 구조를 보면, 단일 AWS EC2 위에서 Docker로 세 개의 애플리케이션 컨테이너가 실행된다. Spring Boot 백엔드, 수어 인식 FastAPI 서버, 음성 인식 FastAPI 서버가 같은 호스트에 올라가 있고, Nginx 리버스 프록시가 `/api`, `/ws`, `/ai-sign`, `/ai-voice` 경로로 트래픽을 라우팅한다. 데이터는 AWS RDS(MySQL)와 MongoDB Atlas를 함께 사용하며, OpenAI와 Naver CLOVA 같은 외부 API에도 의존한다.

즉 handDoc은 여러 서비스가 하나의 호스트에서 함께 동작하고, 실시간 스트림 처리와 외부 API 호출까지 얽혀 있는 구조다. 이런 특성 때문에 단순히 서버가 켜져 있는지만 확인하는 수준을 넘어, 서비스 내부에서 어떤 일이 일어나는지 관측할 수 있는 구조가 필요하다고 판단했다.

## 2. 이 프로젝트에 옵저버빌리티가 필요한 이유

현재 handDoc은 운영 상태를 확인할 수 있는 수단이 거의 없다. 문제가 생겼을 때 어디서, 왜, 얼마나 자주 발생하는지 파악하기 어렵다는 점에서 옵저버빌리티 적용이 필요하다.

첫째, 리소스 경합을 확인하기 어렵다. PyTorch와 Whisper 기반 AI 추론 서버가 Spring 백엔드와 같은 EC2에서 실행되고 있기 때문에, AI 추론 부하가 증가하면 다른 서비스에도 영향을 줄 수 있다. 하지만 현재 구조에서는 어떤 컨테이너가 CPU나 메모리를 많이 사용하는지, 그 영향이 다른 서비스의 지연으로 이어지는지 분리해서 보기 어렵다.

둘째, 실시간 품질을 측정할 수 없다. 의료 서비스에서 수어 번역이나 음성 변환이 몇 초만 늦어져도 사용자는 서비스가 제대로 동작하지 않는다고 느낄 수 있다. 그런데 현재는 환자가 실제로 응답을 얼마나 기다리는지, 수어·음성 모델의 추론 시간이 어느 정도인지 측정하는 지표가 없다.

셋째, 장애 위치를 추적하기 어렵다. 하나의 요청은 프론트엔드, Nginx, Spring 백엔드, AI 서버, 외부 API를 거칠 수 있다. 이 중 어느 구간에서 지연이나 오류가 발생했는지 확인할 수 없다면, 장애가 발생해도 원인을 추측하는 데 그칠 가능성이 크다.

넷째, 외부 의존성이 사각지대에 있다. handDoc은 OpenAI, Naver CLOVA 같은 외부 API를 사용한다. 외부 API의 응답 지연이나 실패가 서비스 장애로 이어질 수 있지만, 현재는 해당 호출의 지연 시간이나 실패율을 따로 확인할 수 없다.

## 3. 만들고 싶은 최종 산출물

옵저버빌리티는 한 번에 모든 것을 완성하기보다, 메트릭, 로그/알림, 트레이싱 순서로 단계적으로 적용해보고자 한다.

먼저 Prometheus, Grafana, Loki를 기반으로 대시보드를 구축한다. 서비스별 요청률, 에러율, 지연 시간, 리소스 포화도 같은 골든 시그널을 확인할 수 있는 대시보드를 만들고, 호스트와 컨테이너 단위의 인프라 상태도 함께 볼 수 있게 한다. 여기에 handDoc만의 커스텀 메트릭인 수어 모델 추론 지연, 음성 모델 추론 지연, 외부 API 호출 지연 및 실패율을 추가한다.

다음으로 SLI와 SLO를 정의한다. 예를 들어 “수어 번역 p99 지연 시간이 N초 미만이어야 한다”, “에러율은 1% 미만이어야 한다”와 같은 기준을 세운다. 이후 Alertmanager를 통해 에러 버짓이 빠르게 소진되기 전에 알림이 오도록 설정하고, 장애 상황에서 어떤 순서로 확인해야 하는지 runbook도 작성한다.

이후 OpenTelemetry를 적용해 분산 트레이싱을 구축한다. 목표는 `프론트엔드 → Nginx → Spring → AI 서버 → 외부 API`로 이어지는 하나의 요청 흐름을 끝까지 추적하는 것이다. 이를 통해 병목이 Spring 로직에 있는지, AI 추론에 있는지, 외부 API 호출에 있는지 구체적으로 확인할 수 있게 한다.

실제 운영 트래픽이 많지 않다는 점을 고려해, 검증은 부하 테스트와 고장 주입 방식으로 진행한다. k6와 WebSocket 부하 클라이언트를 사용해 트래픽을 만들고, 컨테이너 강제 종료나 외부 API 타임아웃 시뮬레이션을 통해 의도적으로 장애를 발생시킨다. 이후 “고장 발생 → 탐지 → 원인 진단 → 회고”의 흐름을 시연한다.

마지막으로 전체 관측 스택을 docker-compose 또는 IaC 형태로 코드화하고, 장애 시뮬레이션 결과는 postmortem 문서로 정리한다. 단순히 도구를 설치하는 데서 끝나는 것이 아니라, 어떤 문제가 있었고 어떤 지표와 로그, 트레이스로 원인을 찾았는지 기록으로 남기는 것을 최종 목표로 한다.

## 4. 책에서 참고하고 싶은 데모

『모니터링의 새로운 미래 관측 가능성』에서 소개하는 네 번째 데모인 “커스텀 메트릭·로그·추적을 직접 개발해 관측 가능성을 구축하는 흐름”을 참고하고자 한다.

이 데모는 쿠버네티스 환경이 아니라 바이너리 프로세스로 실행된다. 애플리케이션 코드에 직접 계측 코드를 넣고, 메트릭, 로그, 추적 데이터를 생성한 뒤 Grafana, Prometheus, Loki, Tempo 스택과 연결하는 방식으로 진행된다. 또한 에러 비율 조정, 클라이언트·서버 지연 시간 조절 같은 카오스 엔지니어링 기능도 포함되어 있어, 의도적으로 장애를 발생시키고 관측 결과를 확인할 수 있다.

## 5. 해당 데모를 고른 이유

이 데모를 고른 이유는 크게 세 가지다.

첫째, 현재 handDoc의 구조와 잘 맞는다. handDoc은 쿠버네티스 기반 서비스가 아니라 단일 EC2와 Docker 중심으로 구성되어 있다. 따라서 쿠버네티스 기반 데모보다 바이너리 프로세스 또는 단일 호스트 환경에서 시작하는 데모가 적용하기 쉽다. 클러스터 운영 부담이나 추가 비용 없이 관측 가능성의 핵심 개념에 집중할 수 있다는 점도 장점이다.

둘째, handDoc에 가장 필요한 작업이 커스텀 메트릭 계측이기 때문이다. 일반적인 CPU, 메모리, 요청 수 지표만으로는 handDoc의 품질을 충분히 설명하기 어렵다. 수어 모델 추론 시간, 음성 인식 지연, 외부 API 호출 실패율처럼 서비스 도메인에 맞는 지표를 직접 정의해야 한다. 책의 네 번째 데모는 바로 이런 커스텀 지표를 애플리케이션에 직접 심는 과정을 다룬다는 점에서 참고 가치가 크다.

셋째, 메트릭, 로그, 추적을 함께 설계해볼 수 있다. 자동으로 수집되는 기본 지표만 보는 것이 아니라, 애플리케이션 코드 수준에서 어떤 데이터를 남길지 직접 결정해야 한다. 이 과정은 클라우드 엔지니어링 관점에서도 의미가 있다. 단순 모니터링 도구 사용을 넘어, 관측 가능성의 세 가지 신호를 어떻게 연결해 장애 원인을 찾는지 경험할 수 있기 때문이다.

## 6. 실습하면서 확인해보고 싶은 점

가장 확인해보고 싶은 점은 직접 만든 커스텀 메트릭, 로그, 추적이 Grafana 안에서 실제로 잘 연결되는지이다.

예를 들어 데모의 카오스 엔지니어링 기능으로 의도적으로 지연이나 에러를 발생시켰을 때, 먼저 메트릭 대시보드에서 이상 징후를 발견한다. 그다음 exemplar나 trace ID를 통해 해당 요청의 트레이스로 이동하고, 다시 관련 로그를 확인해 원인을 좁혀가는 흐름이 가능한지 확인하고 싶다.

이 흐름이 자연스럽게 이어진다면 handDoc에도 같은 방식을 적용할 수 있다. 예를 들어 수어 추론 지연 시간이 SLO를 넘었을 때, 단순히 “느리다”는 사실만 아는 것이 아니라 해당 요청의 trace와 로그로 바로 이동해 원인이 AI 서버 부하인지, Spring 백엔드 처리 지연인지, 외부 API 응답 지연인지 구분할 수 있다.

결국 이번 실습의 핵심 질문은 다음과 같다.

> handDoc에서 수어·음성 처리 지연이나 외부 API 장애가 발생했을 때, 메트릭에서 이상을 발견하고 로그와 트레이스로 곧바로 연결해 실제 원인까지 진단할 수 있는가?
