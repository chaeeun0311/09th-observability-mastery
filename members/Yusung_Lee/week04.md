## 2. 책 내용 핵심 요약

- **프로메테우스 역할**: 메트릭 수집 / 서비스 디스커버리 / 오토스케일링 / 알람 — 쿠버네티스와 함께 클라우드 운영의 두 축
- **TSDB 저장 흐름**: 데이터 수집 → 메모리 → WAL → 2h 후 디스크 블록 → 블록 병합
- **오퍼레이터**: 쿠버네티스에서 파드·서비스가 동적으로 바뀔 때 설정을 자동 업데이트해줌
- **알람 상태**: Inactive → Pending → Firing — 조건이 일정 시간 유지될 때만 알람 전송 (오탐 방지)
- **확장 방법**: 샤딩(수직/수평) → 페더레이션 → Thanos 순으로 규모에 따라 선택

## 3. 중요하다고 생각한 개념

### 개념 1. TSDB 블록 구조

**한 줄 설명**: 수집된 메트릭은 즉시 디스크에 저장되지 않고, **메모리 → WAL → 디스크 블록** 순서로 저장된다.

**키워드 정리**

- `WAL` = 장애 복구용 로그. 프로메테우스가 갑자기 죽어도 메모리 데이터를 여기서 복구
- `블록(Block)` = 2시간치 데이터 묶음. chunks(데이터) + index(색인) + tombstones(삭제표시)
- `컴팩션(Compaction)` = 작은 블록들을 더 큰 블록으로 합치는 작업 → 조회 성능 향상
- `retention` = 데이터 보관 기간. 기본 15일, 이후 자동 삭제

**왜 중요한가**: 운영 중 데이터가 사라거나 디스크가 꽉 찼다 같은 장애의 원인이 모두 여기서 시작됨

---

### 개념 2. 카디널리티 (Cardinality)

**한 줄 설명**: 메트릭 하나가 만들어내는 **시계열의 수**. 레이블 조합이 많아질수록 폭발적으로 증가함.

**키워드 정리**

- `레이블(Label)` = 메트릭을 구분하는 key-value 쌍. 예: `{status="200", method="GET"}`
- `카디널리티 폭발` = 레이블 값이 무한히 늘어나는 상황 (예: user_id, session_id를 레이블로 쓸 때)
- **공식**: 레이블 A의 값 수 × 레이블 B의 값 수 = 시계열 수

**왜 중요한가**: 레이블 설계를 잘못하면 프로메테우스 메모리/디스크가 폭발함. 가장 흔한 운영 실수 중 하나

---

### 개념 3. Thanos - 사이드카 vs 리시버

**한 줄 설명**: 프로메테우스 혼자는 클러스터링도 안 되고 장기 보관도 어려운데, Thanos가 이걸 해결해줌.

**키워드 정리**

|  | 사이드카(Sidecar) | 리시버(Receiver) |
| --- | --- | --- |
| 방식 | Pull (원격 읽기) | Push (원격 쓰기) |
| 특징 | 프로메테우스 파드에 붙어서 블록을 객체 스토리지에 업로드 | 프로메테우스가 직접 데이터를 밀어넣음 |
| 단점 | 프로메테우스 종속, 멀티 테넌트 어려움 | 상대적으로 설정 복잡 |
| 추천 | 소규모 단일 클러스터 | 멀티 테넌트/멀티 클러스터 |

**Thanos가 주는 것**: 글로벌 뷰 (여러 프로메테우스를 하나처럼 조회) + 장기 보관 (S3, MinIO 등)

**왜 중요한가**: 현업에서 프로메테우스를 쓰다 보면 반드시 부딪히는 한계. 미리 알고 있어야 확장 설계를 제대로 할 수 있음
