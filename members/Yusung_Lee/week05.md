## 4.2 로키 로그 관리

#### 로키 마이크로 서비스 모드 컴포넌트

- Ingester
- 실제 로그를 저장하는 컴포넌트
- 일단 메모리(WAL 포함)에 버퍼링하다가 일정량이 차면 Object Storage에 블록으로 플러시
- Distributor
    - 로그 수신의 진입점
    - 들어온 로그의 유효성을 검사하고, replication factor에 따라 여러 Ingester에 분배
- Query Frontend
    - 쿼리 요청의 진입점
    - 큰 쿼리를 잘게 쪼개 병렬 처리하고, 결과를 캐싱해서 동일 쿼리의 반복 비용을 줄임
- Query Scheduler
    - Query Frontend에서 쪼갠 쿼리들을 큐에 쌓아두고 Querier에게 순서대로 분배하는 역할
    - Frontend와 Querier 사이의 부하 조절을 담당
- Querier
    - 실제로 데이터를 조회하는 컴포넌트.
    - 캐시 → Ingester 메모리(WAL) → Object Storage 순서로 찾음
- Index Gateway
    - 로그 블록의 인덱스 조회를 전담하는 컴포넌트
    - Querier가 직접 스토리지 인덱스를 뒤지지 않아도 되게 분리해 성능을 높임
- Ruler
    - Recording Rule(쿼리 결과를 메트릭으로 저장)과 Alert Rule(알림 조건 평가)을 주기적으로 실행함
    - Grafana 알림과 연동됨
- Compactor
    - Object Storage에 쌓인 블록들을 압축하고 중복을 제거하며, 보존 기간이 지난 데이터를 삭제해 스토리지를 효율적으로 관리

- Distirbutor는 Ingester로 스트림을 전달
    - 이 과정에서 replication factor 개수를 참고
    - replication factor란?
    - repilcation factor가 3이라면 Ingester 내에 3개의 복제본을 생성하도록 3개의 스트림을 전달
    - prometheus는 고가용성과 클러스터 구성이 어려워 디스크 용량과 메모리 부족으로 데이터가 유실되거나 장애가 자주 발생하는데
    Loki는 replication factor와 rebalancing 기능을 제공하여 신뢰성있는 시스템 구성을 지원
    - replication factor가 3이라는 것은 복제본 3개가 존재한다는 의미
        - Ingester가 Stream을 저장하거나 쿼리어가 스트림을 조회 시 중복이 발생할 수 있지만 Loki 내부적으로 중복 제거 과정을 거쳐 중복 발생 X
- Ingester에서 장애 발생 시 handoff를 사용하여 새로운 Ingester를 생성할 수 있음
    - 하지만 이 경우에는 rebalancing에 따른 지연이 발생해서 이를 방지하기 위해
    WAL(Write Ahead Logging)을 사용하는 것을 권장
- Distributor와 Ingester 간에는 Quorum을 사용해서 안정적으로 스트림을 전달
    - 각 샤드 간에는 Gossip을 사용해서 헬스 체크 수행
- Grafana Observability에서는 WAL, Quorum, Gossip등 다양한 기능을 제공
    - WAL?
        - 데이터를 실제로 메모리나 디스크에 쓰기 **전에**, 먼저 로그 파일(WAL)에 기록하는 방식입니다. Ingester가 갑자기 죽더라도 WAL에 남아 있는 기록을 보고 데이터를 복구할 수 있습니다.
        - Ingester 장애 시 기존에는 **handoff** 방식(살아있는 Ingester에게 데이터를 넘겨주는 방식)을 사용했는데, 이 방식은 새 Ingester를 만들고 rebalancing하는 과정에서 지연이 생깁니다. WAL을 쓰면 handoff 없이 재시작만으로 복구가 가능해서 훨씬 빠르고 안정적입니다.
    - Handoff
        - Ingester 장애 시 다른 Ingester에 데이터를 넘기는 방식 (WAL로 대체 권장)
    - Quorum?
        - 분산 시스템에서 **과반수 이상의 노드가 동의**해야 작업이 성공한 것으로 간주하는 방식입니다.
        - replication factor가 3이라면, 최소 2개 (과반수) Ingester에서 쓰기가 성공해야 전체 쓰기가 성공으로 처리됩니다. 하나가 느리거나 죽어도 나머지 둘이 살아있으면 정상 동작합니다. 읽기도 마찬가지로, 2개에서 결과를 받아 일치 여부를 확인합니다.
    - Gossip?
        - 분산된 노드들이 서로를 직접 모니터링하지 않고, **소문처럼 퍼지는 방식**으로 서로의 상태를 공유하는 프로토콜입니다.
        - A가 B에게 자신의 상태를 알려주고, B는 C에게 A와 자신의 상태를 전달하는 식입니다. 중앙 관리자 없이도 전체 클러스터가 서로의 헬스 상태를 알 수 있고, 장애 노드를 감지할 수 있습니다. Loki에서는 Ingester(샤드) 간에 Gossip으로 헬스 체크를 수행합니다.
    - Hash Ring?
        - 각 Ingester(또는 Querier 등)를 **가상의 링 위에 배치**해서 요청을 분산하는 구조입니다.
        - 들어오는 스트림의 레이블을 해싱해서 링의 특정 위치를 정하고, 그 위치에 해당하는 Ingester에 데이터를 전달합니다. Ingester가 추가되거나 제거돼도 링 전체를 재구성하지 않고 **인접한 샤드만 재조정(rebalancing)** 하면 되므로 효율적입니다.


- 읽기 요청을 받은 쿼리어는 캐시 → Ingester WAL 메모리, 블록 스토리지 순으로 결과 조회
- Loki component는 다수의 해시 링에 샤드를 구성한 다음 WAL, Quorum, Gossip, Cache, replication facotr를 사용해서 안정성을 향상시키고, 병령 처리와 중복 제거를 기본으로 처리
- 위 방식은 미미르, 템포에도 적용됨 → prometheus 기반으로 개발되어 유사하게 동작하기 때문

#### 로키 컴포넌트

- Distributor
    - 들어오는 스트림을 처리하는 역할
    - log 데이터에 대한 쓰기 경로의 첫 번째 단계
    - 정확섬 검증
    - 유효한 청크를 배치로 분할하고 병렬로 여러 Ingester에 보낸다
    - Distrbutor로 가는 트래픽 균형을 적절하게 맞추기 위해 LB는 Distrbutor 앞에 위치해야 함
    - 상태 비저장 컴포넌트
        - Distributor가 자체적으로 데이터를 저장하거나 기억하지 않는다는 뜻, 요청을 받아서 검증하고 넘겨주는 것만 합니다.
        - 상태가 없으니까 **Distributor를 여러 개 띄워도 아무 문제가 없다.** 
        어떤 Distributor가 요청을 받든 결과가 동일. 
        반면 Ingester는 자기가 현재 어떤 데이터를 메모리에 들고 있는지 상태가 있기 때문에 함부로 늘리거나 줄이기가 까다롭습니다.
    - 즉, 쓰기 경로에서 중요한 컴포넌트인 인제스터에서 가능한 많은 작업을 쉽게 확장 가능
    이러한 검증 작업을 독립적으로 확장할 수 있다는 것은 Loki가 수집기에 과부하를 줄 수 있는 서비스 거부 공격으로부터 스스로를 보호할 수 있다는 것을 의미
        - 트래픽이 갑자기 폭증하거나 악의적인 요청이 대량으로 들어오는 상황(DoS 공격)을 생각해보면, 만약 Ingester가 이 요청을 직접 다 받는다면 메모리가 터지거나 데이터가 유실될 수 있습니다. 하지만 Distributor가 앞에서 먼저 받아서 **유효하지 않은 요청은 걸러내고**, 정상 요청만 Ingester로 전달합니다. 그리고 부하가 커지면 Distributor만 추가로 늘리면 되니까 Ingester는 보호됩니다.
    - 유효성
        - Distrbutor가 처리하는 첫 번째 단계는 들어오는 모든 데이터가 사양에 맞는지 확인하는 유효성 검사
        - 레이블이 유효한 prometheus 레이블인지, 타임스템프가 넘 오래되거나 새롭지 않은지, 로그 줄이 너무 길지 않은지 확인하는 것 등이 포함
    - 비율 제한, Rate Limiting
        - Distributor는 테넌트당 최대 비트 전송률을 기반으로 수신 로그의 속도를 제한 가능
        - 비율 제한은 로키 클러스터를 간단하고 안전하게 작동시키는 방법
    - 포워딩
        - Distributor가 모든 유효성 작업을 수행하고 최정적으로 쓰기를 담당하는 인제스터 컴포넌트에 데이터를 전달
    - replication factor
        - 단일 Ingester에서 데이터 손실 가능성을 줄이기 위해 Distributor는 쓰기를 해당 데이터의 replication factor에 전달
        - 일반적인 값은 3
        - 주된 목적은 데이터 손실 방지보다 롤아웃 및 다시 시작하는 동안 중단 없이 쓰기를 계속할 수 있도록 하는 것
    - 해싱
        1. Hash Ring?
            - 모든 Ingester는 시작할 때 해시 링에 자신을 등록합니다. 이때 여러 개의 토큰(32비트 숫자)을 가져가서 링의 여러 위치를 차지합니다. 링 위에 Ingester들이 촘촘하게 박혀있다고 생각하면 됩니다.
        2. 스트림을 어디로 보낼지 결정하는 방법
            - 로그가 들어오면 Distributor는 그 스트림의 **테넌트 ID + 레이블 집합**을 해싱합니다. 그러면 하나의 숫자가 나오고, 그 숫자에 해당하는 링 위치에 있는 Ingester에게 보냅니다. 같은 스트림은 항상 같은 해시값 → 항상 같은 Ingester로 가게 됩니다.
        3. Replication Factor가 1보다 크면?
            - replication factor가 3이면, 해당 위치에서 시작해서 링을 따라 돌면서 3개의 Ingester를 선택해 모두 보냅니다.
        4.  Ingester 상태(JOINING / ACTIVE / LEAVING)
            
            Ingester가 항상 정상인 건 아니라서 상태가 있습니다.
            
            - **JOINING / ACTIVE** → 쓰기 요청 받을 수 있음
            - **ACTIVE / LEAVING** → 읽기 요청 받을 수 있음
            
            Distributor는 해시 룩업 시 적절한 상태의 Ingester에게만 보냅니다. 
            죽어가는(LEAVING) Ingester에게 새 쓰기를 보내지 않는 거죠.
            
        
        > 스트림을 해싱해서 번호를 뽑고 → 링에서 그 번호 담당 Ingester를 찾고 → replication factor 수만큼 복제해서 전송
        > 
    - 쿼럼 일관성
        - 모든 Distributor가 동일한 해시 링에 대한 액세스를 공유하므로 쓰기 요청을 모든 Distributor에게 보낼 수 있다
        - Loki는 일관된 쿼리 결과를 보장하기 위해 읽기와 쓰기에 쿼럼 일관성을 사용
- Ingester
    - Ingester 서비스는 쓰기 경로의 장기 스토리지 백엔드(Dynamo DB, S3, Casandra 등)에 로그 데이터를 쓰고, 읽기 경로의 메모리 내 쿼리의 로그 데이터를 반환하는 역할을 함
    - Ingeter에는 hash ring에서 관리하는 Ingester 수명 주기를 관리하는 수명주기가 포함
    - 상태
        - PENDING
            - LEAVING 상태인 다른 Ingester로부터 handoff를 기다리고 있을 때 Ingester의 상태
        - JOINING
            - 현재 토큰을 링에 삽입하고 초기화 중일 때 인제스터의 상태
        - ACTIVE
            - 완전히 초기화되었을 때 Ingester의 상태
            - 소유한 토큰에 대한 쓰기 요청을 받을 수 있음
        - LEAVING
            - 종료될 때 Ingester의 상태
            - 메모리에 있는 데이터에 대한 읽기 요청 가능
        - UNHEALTHY
            - 콘술의 하트비트에 실패했을 때 Ingester의 상태
    - Ingester가 수신하는 각 log stream은 메모리 내 다수의 청크 세트로 구성되고, 구성 가능한 주기로 백엔 스토리지 백엔드에 플러시 됨
    아래 경우 청크가 압축되고 읽기 전용으로 표시
        - 현재 청크가 용량(구성 가능한 값)에 도달했을 때
        - 현재 청크가 업데이트되지 않은 채로 많은 시간이 경과했을 때
        - 플러시 발생시
    - Ingester 프로세스가 충돌하거나 갑자기 종료되면 아직 플러시 되지 않은 모든 데이터가 손실
        - 이러한 위험을 완화하기 ㅜ이해 각 로그의 여러 레플리카를 복제하도록 구성
        - 영구 저장소에 플러시가 발생하면 청크는 테넌트, 레이블, 콘텐츠를 기반으로 해싱
        - 즉, 동일한 데이터 레플리카를 가진 여러 인제스터가 동일한 데이터를 영구 저장소에 두 번 쓰지 않는다.
    - 타임스탬프 정렬
        - Loki는 비순차적 쓰기를 허용하는 구성 가능
        - 그렇지 않은 경우 순서를 확인
    - 캐싱
        - 쿼리 프런트엔드는 메크릭 쿼리 결과 캐싱을 지원하고 후속 쿼리에서 이를 재사용
- 쿼리어, Querier
    - LogQL 쿼리 언어를 사용하여 쿼리를 처리하고 Ingester와 장기 저장소 모두에서 로그를 가져옴

#### 일관된 해시 링

일관된 해시 링은 로키 클러스터 아키텍처와 통합하여 다음과 같은 장점을 제공

- 고가용성
- 클러스터 수평 확장 축소

모든 로키 컴포넌트가 해시 링으로 자동 연결되는 것은 아님, 아래 컴포넌트 들을 수동으로 연결

- Distributor
- Ingester
- Query Scheduler
- Compactor
- Ruler

링의 각 노드는 컴포넌트의 인스턴스를 나타냄

- 노드는 모든 노드에 대한 콘텐츠 일관성을 유지하기 위해 주기적으로 키-값 저장소 업데이트
- 키-값 저장소는 다음을 보유
    - 컴포넌트 노드의 ID
    - 다른 노드에서 통신 채널로 사용하는 컴포넌트 주소
    - 컴포넌트 노드의 상태 표시

#### 그라파나 로키 레이블

레이블은 키-값으로 이루어진 쌍이며 무엇이든 정의 가능

- 로키가 레이블을 사용하는 방법
    - 로키의 레이블은 스트림을 정의하는 매우 중요한 작업을 수행
    - 모든 레이블 키와 값의 조합이 스트림을 정의하고 하나의 레이블값만 변경되어도 새 스트림이 생성
- 병렬 처리를 통한 로키 성능 향상

## 4.3 미미르 메트릭 관리

### 4.3.1 미미르 기능

타노스는 현시점에서 가장 대중적인 장기 저장소 솔루션이지만 튜닝이 어렵고, 문제가 발생하면 해결이 어렵다는 단점이 존재

타노스를 대체할 수 있는 그라파나 미미르를 소개

- 미미르는 MSA로 구현
- 병렬 실행과 수평으로 확장 가능한 마이크로서비스 다수 존재
- 프로메테우스와 오픈메트릭 등을 지원

미미르는 프로메테우스를 통합할 수 있는 메트릭 서버

#### 미미르 컴포넌트

미미르는 다양한 컴포넌트로 구성

- 대부분의 컴포넌트는 stateless
- 일부 state 컴포넌트는 프로세스 재시작 사이에서 데이터 손실 방지를 위해 비휘발성 저장소에 저장

#### 쓰기

- promethues
    - 원격쓰기 API를 사용하여 그라파나 미미르로 푸시
    - 원격 쓰기 API는 HTTP PU 요청 본문 내에서 일괄 처리된 스내피 압축 프로토콜 버퍼 메시지를 내보냄
    - 미미르에는 각 HTTP 요청에 대한 테넌트 ID를 지정하는 헤더가 있어야 함
    - 들어오는 샘플(프로메테우스의 쓰기)은 Distributor가 처리하고, 들어오는 읽기(PromQL 쿼리)는 쿼리 프런트엔드에서 처리
- Ingester
    - 수신된 샘플을 테넌트별 저장소에 추가
- compactor
    - 시리즈 샤딩과 복제를 위해 기본적으로 각 시계열은 3개의 Ingester로 복제되고, 각 Ingester는 자체 블록을 장기 저장소에 씀
    - compactor는 여러 수집기의 블록을 단일 블록으로 병합하고 중복 샘플을 제거

#### 읽기

- Query Frontend
    - 미미르로 들어오는 쿼리를 입수하고 더 작은 쿼리로 분할
    - 결과 캐시를 확인하고 캐시된 경우 결과를 반환
    - 캐시 미스 시 큐에 넣음
- 쿼리어
    - 쿼리 프론트엔드 내 메모리 큐에서 쿼리를 가져오는 작업자 역할

#### 스토리지 방식

그라파나 미미르는 각 테넌트의 시계열을 자체 TSDB에 저장함으로써 블록의 시리즈를 유지

#### 인덱스 헤더

stateful인 스토어 게이트웨이는 장기 저장소에서 블록을 쿼리

필요한 정보를 얻기 위해 스토어 게이트웨이는 블록에 대한 인덱스 헤더를 만들고 로컬 디스크에 저장

#### 해시 링의 작동 방식

Grafana Hash Ring은 컴포넌트의 복제본 간의 작업을 공유하는 데 사용되므로 다른 컴포넌트가 통신할 주소를 결정할 수 있다. 

해시 값을 사용하여 이를 소유한 링 구성원을 찾는다

#### 멤버리스트와 가십 프로토콜

MemberList는 Gossip 프로토콜을 사용하여 클러스터 맴버십과 구성원 실패 감지를 관리하는 Go 라이브러리

#### 쿼리 샤딩

- 미미르에는 여러 시스템에서 단일 쿼리를 실행할 수 있는 기능이 포함되어 있다. 이것은 데이터셋을 더 작은 조각으로 쪼개는데, 이 작은 조각을 `샤드`라고 한다.
- 샤드는 부분 쿼리에서 쿼리됨
- 부분 쿼리는 쿼리 프론트엔드에서 배포되어 다린 쿼리어에서 병렬로 실행
- 결과는 쿼리 프론트엔드가 집계하여 전체 쿼리 결과를 반환
