### 복원성 패턴

#### 1. 재시도

- 비즈니스 에러: 호출자가 잘못된 데이터를 입력, 프로그램의 버그 → 재시도 불필요
- 시스템 에러: 네트워크의 일시적인 지연, 서버와 백엔드의 장애 → 재시도 필요
- 가끔 발생하는 장애에 효율적, but 재시도 압력이 누적돼 시스템에 과부하를 일으키면 안 되므로 주의
- 다운스트림 서비스: 순간적인 장애가 잦음, thread pool 포화, 네트워크 접속 지연으로 인한 시간 초과 등 가용성 저해 요인이 많음 → 단시간 내에 자동적으로 비활성화, 호출자는 일시적 오류에 대처
- 최대 재시도 횟수와 시간 간격은 재시도 정책에 정의 필요

> **재시도 압력(Retry Storm)이란?**
서비스A가 서비스B를 호출했을 때 실패하여 3번 재시도한다고 가정하자. 서비스A의 인스턴스가 100개라면, 서비스B에는 실제 요청의 최대 300배에 달하는 재시도 요청이 밀릴 수 있다. 이미 장애 상태인 서비스B에 재시도가 쏟아지면 복구를 더 늦추게 된다. 이를 방지하기 위해 지수 백오프(Exponential Backoff) — 재시도 간격을 점점 늘리는 방식 — 와 최대 재시도 횟수를 정책으로 제한한다.
> 

> **다운스트림 서비스(Downstream Service)란?**
현재 서비스가 호출하는 방향에 있는 서비스. 서비스A → 서비스B를 호출하면, 서비스B가 서비스A의 다운스트림이다. 장애는 업스트림(호출하는 쪽)보다 다운스트림(호출받는 쪽)에서 먼저 감지된다.
>

- 카나리아, A/B 테스트와 유사
- Istio, 인그레스, 게이트웨이 로드밸런싱으로 구현 가능

#### 3. 벌크헤드

- 마이크로서비스는 한 서비스의 가용성만 저하되어도 의존하는 다른 서비스까지 응답 불가 상태에 놓일 위험 존재
- 다운스트림 서비스를 서로 격리, 각 서비스의 동시 처리 능력을 제한
- 장애가 생긴 서비스 호출을 동반한 요청은 배제, 그 외의 다른 요청만 처리

> **"동시 처리 능력을 제한"하는 방식**
서비스A가 결제 서비스와 배송 서비스 두 곳을 동시에 호출한다고 할 때, 벌크헤드 패턴을 적용하면 결제 서비스 호출 전용 스레드 풀과 배송 서비스 호출 전용 스레드 풀을 분리한다. 결제 서비스가 느려져 그 스레드 풀이 가득 차도, 배송 서비스 쪽 스레드 풀은 독립적으로 정상 동작한다. 두 풀을 공유했다면 결제 서비스의 장애가 배송 서비스 호출까지 막는다.
> 

---

#### 4. 서킷 브레이커

- 벌크헤드의 확장 개념
- 마이크로서비스 장애 전파 방지
- 서비스1 → 서킷 브레이커 → 서비스2
    - 서킷 브레이커가 강제로 호출을 종료하는 방식
    - 폴백(Fallback): 서비스2가 장애 상황에서 대안이 되는 로직 반환

> **서킷 브레이커의 세 가지 상태**
서킷 브레이커는 전기 회로 차단기처럼 세 상태로 동작한다:
> 
> - Closed(닫힘): 정상 상태. 요청을 서비스2로 통과시킨다.
> - Open(열림): 장애 감지 상태. 서비스2로의 요청을 차단하고 즉시 폴백을 반환한다. 서비스2를 실제로 호출하지 않으므로 타임아웃 대기 없이 빠르게 응답한다.
> - Half-Open(반열림): 일정 시간 후 소량의 요청을 시험적으로 서비스2에 보내 복구 여부를 확인한다.

### 쿠버네티스 오토스케일링

- HPA(Horizontal Pod Autoscaler, 수평적 파드 오토스케일러)
    - 지정한 메트릭을 컨트롤러가 체크하여 부하에 따라 필요한 레플리카 수가 되도록 자동으로 파드를 늘리거나 줄임
- 측정된 메트릭을 기반으로 파드를 증가시키는 흐름
    - 애플리케이션 메트릭의 수집, 관리를 위해 프로메테우스 어댑터 등 사용
- 유저의 트래픽을 각 파드로 분산하는 흐름
    - 부하가 파드로 균등하게 분산
- 파드 레벨의 오토스케일링과 노드 레벨의 오토스케일링 함께 사용
- 표준 메트릭(CPU, 메모리) / 사용자 정의 메트릭을 HPA 평가 기준으로 사용 가능

**오토스케일링 오픈소스**

1. 메트릭 서버
    - 메트릭 정보로 접근하는 쿠버네티스 API
2. 프로메테우스 어댑터
3. KEDA
    - Kubernetes Event-Driven Autoscailing
    - 이벤트 소스를 모니터링 하고 해당 데이터를 HPA에 공급
    - 이벤트 소스와 다양한 연결을 유지하면서 이벤트를 기반으로 확장 가능
    - 레디스, 카프카, 프로메테우스 등 다양한 자원의 메트릭을 직접 측정

**메트릭 측정 항목**

- 요청 수
- 요청 기간
- 동시 요청

### CQRS

- 마이크로서비스의 읽기와 쓰기는 분리됨

**CQRS 패턴의 이점**

- 독립적인 크기 조정
    - 오토스케일링을 통해 수평/수직 확장을 지원 ex)HPA, VPA

    - 질의 쪽에서는 쿼리에 최적화된 스키마를 사용, 쓰기 쪽에서는 업데이트에 최적화된 스키마 사용

    - 유지가능, 유연한 모델
- 보안 강화
    - 올바른 도메인 엔티티만 쓰기를 수행할 수 있는 쉽게 확인
- 단순한 쿼리
    - 읽기 데이터베이스에서 구체화된 뷰를 저장, 쿼리를 실행할 때 애프리케이션에서 복잡한 조인이 일어나는 것을 방지

### 안정적인 데이터 관리

**데이터 파티션**

- 데이터를 작은 파티션으로 분할 후 여러 대의 서버에 저장
- 안정 해시를 사용하여 데이터 파티션
- 샤딩: 메모리에 적용, 고유한 파티션 키를 통해 재분배
- 파티셔닝: 디스크에 적용, 시간 순서에 따라 적재

**데이터 다중화**

- 프라이머리와 레플리카로 구성

**데이터 일관성**

- 여러 노드에 다중화된 데이터는 적절히 동기화
- 정족수 합의 프로토콜을 사용하여 읽기/쓰기 연산 모두 일관성 보장

> **정족수 합의(Quorum Consensus)란?**
데이터를 N개의 노드에 복제할 때, 쓰기는 W개 이상의 노드에 성공해야 완료로 처리하고, 읽기는 R개 이상의 노드에서 값을 읽는 방식이다. `W + R > N` 조건을 만족하면 최소 하나의 노드는 최신 값을 갖고 있음이 보장된다. 예: N=3, W=2, R=2이면 항상 최신 데이터를 읽을 수 있다.
> 

**장애 감지**

- **가십 프로토콜(Gossip Protocol)**
- 시스템 내부적으로 문제점 진단

> **가십 프로토콜이란?**
각 노드가 주기적으로 랜덤하게 선택한 이웃 노드와 자신의 상태(생존 여부, 버전 정보)를 교환하는 방식이다. 중앙 관리자 없이 분산된 노드들이 서로 상태를 전파해, 특정 노드의 장애를 클러스터 전체가 자연스럽게 감지하게 된다.
> 

**일시적 장애 처리**

- 장애 서버로 가는 요청을 다른 서버가 잠시 맡아 처리 → 그동안 변경 사항은 서버 복구 때 일괄적으로 반영하여 데이터 일관성 보존
    
    ⇒ 임시 위탁 기법 (Hinted Handoff)
    

> **임시 위탁 기법(Hinted Handoff)의 동작 방식**
노드A가 다운된 상태에서 노드A에 저장되어야 할 쓰기 요청이 들어오면, 대신 노드B가 그 데이터를 "힌트"와 함께 임시 저장한다. 노드A가 복구되면 노드B는 힌트를 참고하여 해당 데이터를 노드A로 전달한다. 이렇게 하면 노드 장애 중에도 쓰기 작업이 실패하지 않고 최종적으로 일관성이 복원된다.
>
