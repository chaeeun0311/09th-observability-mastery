## 모니터링 vs 관측가능성 (Observability)

- **관측가능성:** 시스템에서 외부로 출력되는 값만 이용해서, 시스템의 내부 상태를 이해하고 예측하는 것
- **모니터링:** 블랙박스 + 화이트박스
- **관측가능성:** 화이트박스 + 예측
- 상세한 디버깅은 어려움 - 목적이 다름
- 상태 정보 제공 → 장애 발생 시 신속한 대응 가능
- 원하는 계측을 추가 → 발생 가능성이 높은 장애 예측
- 클라우드 네이티브처럼 분산되고 복잡한 시스템에서 발생하는 이벤트에 대한 문맥(context) 정보 제공
- 가시성: 인프라와 네트워크 레벨에서 문제 원인을 이해, 해결방법과 절차

## 메트릭 (Metrics)

- 측정 데이터 집계 및 수치화
    - ex) 큐의 대기 메시지 개수, 사용 중인 CPU와 메모리의 크기, 서비스에서 초당 처리하는 개수
- 전체적인 시스템 상태 보고에 유용 → 히스토그램, 게이지 등 차트로 시각적 표현
- SLI (Service Level Indicator / 서비스 수준 지표) 로 정량화
    - SLO (Service Level Objective / 서비스 수준 목표) 라는 임계 기준을 달성해야 함, 상한과 하한 범위 지정

---

### 골든 시그널 (Golden Signals)

**1. 지연 (Latency)**

- 에러가 아님, 잠재적 장애를 예측할 수 있는 신호
- 서비스 요청 후 완료까지 걸리는 시간
- 그라파나에서 제공하는 히스토그램을 사용 → PromQL로 지연 계산

**2. 에러 (Errors)**

- 에러의 상태 변화를 시계열로 출력
    - 정상 → 대기(pending) 로 변환, 에러가 계속 활성 상태면 firing 상태로 변하고 알림 전송

> 
> 
> 
> **firing 상태란:**
> 알림 시스템에서, 에러 조건이 일정 시간 이상 지속될 때 실제로 알림(슬랙, 이메일 등)을 보내는 상태. "대기(pending)" 는 조건이 막 충족되기 시작한 상태고, "firing" 은 그 상태가 설정한 시간을 넘어 지속되어 알림이 발송된 상태다.
> 

**3. 트래픽 (Traffic)**

- 발생하는 요청량
    - 시스템의 유형, 초당 요청수(RPS), 네트워크 입출력 등
- 양이 급격하게 많아진다면 지연과 에러 발생 → 장애로 연결 방지

**4. 포화 (Saturation)**

- 전체 총대역폭과 처리 가능한 트래픽에 대비해서 현재 트래픽
    - 얼마나 채워졌는지, 얼마나 채울 수 있는지
- 지연 증가는 포화의 선행 지표일 때도 있음
- 오토스케일링, DB 샤딩

---

### 메트릭 유형

- **카운터 (Counter):** 모니터링하는 이벤트의 누적 개수 혹은 크기
- **게이지 (Gauge):** 현재 상태를 표현
- **요약 (Summary):** 응답의 크기와 대기시간을 추적하는 데 사용, 측정한 이벤트의 합계와 카운트
- **히스토그램 (Histogram):** 시간 경과에 따른 추세와 데이터가 단일 범주에서 어떻게 분포되어 있는지를 표현

---

### 메트릭 관리

- 대시보드에 점진적으로 정보 설정
- 응답시간, 에러, 트래픽에 집중
- 태그 사용 - 메트릭에 상황 정보 포함
- 명명 규칙 유지

## 추적 (Tracing)

- 마이크로서비스가 트랜잭션을 처리하는 과정에서 발생하는 세부적인 정보
- 시스템을 이동하는 경로, 트랜잭션을 처리하는 과정에서 발생하는 대기시간, 지연시간, 병목현상이나 에러를 일으키는 원인을 메타데이터로 출력

---

### 스팬 (Span)

- 수행 시간 정보, 하위 동작의 시작과 소요 시간 정보
    - 동작 정보, 타임라인, 부모 스팬과의 의존성 포함
    - 태그, 로그, 스팬 컨텍스트, 배기지 정보 포함

**스팬 컨텍스트 (Span Context)**

- 새로운 스팬을 생성하기 위해서는 다른 스팬을 참조해야 함
- 스팬 컨텍스트로 표현된 메타데이터를 쓸 수 있는 `inject`와 읽을 수 있는 `extract` 메서드 제공
    - 주입과 추출을 통해 헤더에 전달

**스팬 레퍼런스 (Span Reference)**

- 두 스팬 사이의 인과관계
- 스팬들이 동일한 추적에 속하는 것을 추적자가 알 수 있게 함
- `child of` / `following from` 둘 중 하나의 관계를 가짐
    - **child of:** 루트 스팬이 작업을 완료하기 전에 하위 범위에 대한 종속성을 가짐
    - **following from:** 루트 스팬이 DAG(방향성 비순환 그래프)의 조상, 의존성 없이 독립적으로 처리
- 예거를 통해 한눈에 관찰 가능

> 📌 **child of vs following from 차이:**
> 
> - **child of:** 부모 스팬이 결과를 기다린다. 예: 결제 승인 스팬이 끝나야 주문 완료 스팬이 끝날 수 있다 — 종속 관계
> - **following from:** 부모 스팬이 결과를 기다리지 않는다. 예: 주문 완료 후 "구매 확인 이메일 발송"은 비동기로 처리 — 주문 스팬은 이미 끝났고, 이메일 스팬은 독립적으로 따라간다
> 
> 📌 **DAG (방향성 비순환 그래프) 란:**
> 화살표가 있고(방향성), 한 번 지나간 노드로 다시 돌아오는 경로가 없는(비순환) 그래프. 작업의 실행 순서나 의존 관계를 표현할 때 쓰인다. `following from` 은 이 DAG 구조에서 부모가 조상 위치에 있지만 자식의 결과에 의존하지 않는 관계를 뜻한다.
> 

**배기지 (Baggage)**

- 스팬에 전달되고 공유되는 값, HTTP 헤더에 포함

> 📌 **배기지가 어떤 상황에 쓰이는가:**
"이 요청은 VIP 사용자가 보낸 것이다"라는 정보를 모든 서비스에 전달하고 싶을 때. 배기지에 `user_tier=vip` 를 넣으면, 해당 요청이 거치는 모든 서비스에서 이 값을 읽을 수 있다. 단, 모든 서비스를 거치면서 헤더에 포함되기 때문에 너무 큰 데이터를 넣으면 성능에 영향을 준다.
> 

**캐리어 (Carrier)**

- 내용물을 담는 공간

> 📌 **캐리어가 무엇인가:**
배기지와 스팬 컨텍스트를 실어 나르는 그릇. 대부분의 경우 HTTP 헤더가 캐리어 역할을 한다.
>

## 로그 (Logs)

- 애플리케이션 실행 시 생성되는 텍스트 라인
- 에러와 경고 확인, 문제 원인 이해
- 로그를 중앙 집중적으로 수집, 관리 → 로그 유형 표준화

---

### 로그 포함 정보

**타임스탬프**

- 시간대 정보 포함
- GMT, UTC 권장 → 서로 다른 시간대의 서비스 통일
- 시간순 정렬

**식별자**

- 저장하려는 데이터에 가능한 많은 고유 식별자 포함
- 주요 키 정보가 로그에 포함되어야 함

**소스**

- 호스트, 클래스, 모듈, 함수, 파일명 등
- 병목 구간과 잠재적 성능 문제 식별에 효과적

**라벨/카테고리**

- 로그 데이터의 유형, 로그 라벨
    - `ERROR`, `DEBUG`, `INFO`, `WARN`

---

## 상관관계 (Correlation)

- 로그 → 메트릭: 집계
- 로그 → 추적: 시간순 정렬

**로그 - 추적**

- 로그 파일에는 추적을 식별할 수 있는 추적 ID 추가 가능
- 스팬 컨텍스트 내 메타데이터에 로그 파일명 추가 → 풍부한 정보 제공, 단 너무 많은 메타데이터는 성능 저하 유발

**추적 - 메트릭**

- 프로메테우스가 제공하는 이그젬플러 활용
- **이그젬플러 (Exemplar):** 프로메테우스 내부 저장소, 추적 저장
    - 템포의 메트릭 제너레이터
    - 예거의 메트릭 스토리지

> 📌 **이그젬플러가 어떤 상황에 유용한가:**
그라파나 대시보드에서 "에러율이 갑자기 뛴 시점"을 보고 있을 때, 이그젬플러가 있으면 그 데이터 포인트에 "이 시점의 실제 추적 ID"가 붙어 있다. 클릭 한 번으로 해당 추적의 상세 내용(어느 서비스에서 에러가 났는지)으로 이동할 수 있다 — 메트릭에서 추적으로 바로 점프하는 것.
> 

**메트릭 - 로그**

- 직접적인 연결 방법은 딱히 없다
- 네이밍을 유사하게 일치, 타임스탬프를 사용하여 검색 범위를 좁힘
- **카디널리티:** 메트릭으로 변환 가능

## 모니터링 vs 옵저버빌리티

- 관측가능성 : 시스템에서 외부로 출력되는 값만 이용해서, 시스템의 내부 상태를 이해하고 예측하는 것
- 모니터링: 블랙박스+화이트박스
- 관측가능성: 화이트박스+예측
- 추적(tracing), 계측(instrumentation), 텔레메트리(telemetry)

- 상세한 디버깅은 어려움 - 목적이 다름
- 상태 정보 제공 → 장애 발생 시 신속한 대응 가능
- 원하는 계측을 추가 → 발생 가능성이 높은 장애 예측
- 클라우드 네이티브처럼 분산되고 복잡한 시스템에서 발생하는 이벤트에 대한 문맥(context) 정보 제공
- 가시성 : 인프라와 네트워크 레벨에서 문제 원인을 이해, 해결방법과 절차

## 메트릭

- 측정 데이터 집계 및 수치화
    
    ex) 큐의 대기 메세지 개수, 사용 중인 CPU와 메모리의 크기, 서비스에서 초당 처리하는 개수
    
- 전체적인 시스템 상태 보고에 유용 → 히스토그램, 게이지 등 차트로 시각적 표현
- SLI(Service Level Indicater/서비스 수준 지표)로 정량화
    - SLO(Service Level Objective/서비스 수준 목표)라는 임계 기준을 달성해야 함, 상한과 하한 범위 지정

### 골든시그널

1. 지연
    - 에러가 아님, 잠재적 장애를 예측할 수 있는 신호
    - 서비스 요청 후 완료까지 걸리는 시간
    - 그라파나에서 제공하는 히스토그램을 사용 → PromQL로 지연 계산
2. 에러
    - 에러의 상태 변화를 시계열로 출력
        - 정상 → 대기 로 변환, 에러가 계속 활성 상태면 firing 상태로 변하고 알림 전송
3. 트래픽
    - 발생하는 요청량
        - 시스템의 유형, 초당 요청수, 네트워크 입출력 등
    - 양이 급격하게 많아진다면 지연과 에러 발생 → 장애로 연결 방지
4. 포화
    - 전체 총대역폭과 처리 가능한 트래픽에 대비해서 현재 트래픽
        - 얼마나 채워졌는지, 얼마나 채울 수 있는지
    - 지연 증가는 포화의 선행 지표일 때도 있음
    - 오토스케일링, DB 샤딩

### 메트릭 유형

- 카운터 : 모니터링 하는 이벤트의 누적 개수 혹시 크기
- 게이지 : 현재 상태를 표현
- 요약 : 응답의 크기와 대기시간을 추적하는데 사용, 측정한 이벤트의 합계와 카운트
- 히스토그램 : 시간 경과에 따른 추세와 데이터가 단일 범주에서 어떻게 분포되어 있는지를 표현

## 추적

- 마이크로서비스가 트랜잭션을 처리하는 과정에서 발생하는 세부적인 정보
- 시스템을 이동하는 경로, 트랜잭션을 처리하는 과정에서 발생하는 대기시간, 지연시간, 병목현상이나 에러를 일으키는 원인을 메타데이터로 출력

### 스팬

- 수행 시간 정보, 하위 동작의 시작과 소요 시간 정보
    - 동작 정보, 타임라인, 부모스팬과의 의존성 포함
    - 태그, 로그, 스팬 콘텍스트, 배기지 정보 포함
- 스팬 컨텍스트
    - 새로운 스팬을 생성하기 위해서는 다른 스팬을 참조해야 함
    - 스팬 콘텍스트로 표현된 메타데이터를 쓸 수 있는 inject와 읽을 수 있는 extract 메서드 제공
        - 주입과 추출을 통해 헤더에 전달
- 스팬 레퍼런스
    - 두 스팬 사이의 인과관계
    - 스팬들이 동일한 추적에 속하는 것을 추적자가 알 수 있게 함
    - child of / following from 둘 중 하나의 관계를 가짐
        - child of : 루트 스팬이 작업을 완료하기 전에 하위 범위에 대한 종속성을 가짐
        - following from : 루트 스팬이 DAG(방향성 비순환 그래프)의 조상, 의존성x 독립적으로 처리
        - 예거를 통해 한눈에 관찰가능
- 배기지 : 스팬에 전달되고 공유되는 값, HTTP 헤더에 포함
- 캐리어 : 내용물을 담는 공간

## 로그

- 애플리케이션 실행 시 생성되는 텍스트 라인
- 에러와 경고 확인, 문제 원인 이해
- 로그를 중앙 집중적으로 수집, 관리 → 로그 유형 표준화

### 로그 포함정보

- 타임스탬프
    - 시간대 정보 포함
    - GMT, UTC 권장 → 서로 다른 시간대의 서비스 통일
    - 시간순 정렬
- 식별자
    - 저장하려는 데이터에 가능한 많은 고유식별자 포함
    - 주요 키정보 로그에 포함되어야 함
- 소스
    - 호스트, 클래스, 모듈, 함수, 파일명 등
    - 병목 구간과 잠재적 성능 문제 식별에 효과적
- 라벨/카테고리
    - 로그 데이터의 유형, 로그 라벨
        - ERROR, DEBUG, INFO, WARN

## 상관관계

- 로그→메트릭 : 집계
- 로그 → 추적 : 시간순 정렬

#### 로그-추적

- 로그 파일에는 추적을 식별할 수 있는 추적ID 추가 가능
- 스팬 콘텍스트 내 메타데이터에 로그 파일명 추가 → 풍부한 정보 제공, but 너무 많은 메타데이터는 성능 저하 유발

#### 추적-메트릭

- 프로메테우스가 제공하는 이그젬플러 활용
- 이그젬플러 : 프로메테우스 내부 저장소, 추적 저장
    - 템포의 메트릭 제너레이터
    - 예거의 메트릭 스토리지

#### 메트릭-로그

- 딱히 없다
- 네이밍을 유사하게 일치, 타임스탬프를 사용하여 검색 범위를 좁힘
- 카디널리티 : 메트릭으로 변환 가능

## 오픈소스

- 오픈텔레메트리 : 계측을 위한 표준화된 도구, API, SDK 모음
- 로키 : 그라파나의 로그 관리 시스템, 각 로그 스트림에 대한 레이블 집합을 인덱싱
- 그라파나 미미르 : 프로메테우스용 오픈소스
- 타노스 : 다수의 프로메테우스를 통합할 수 있는 글로벌 뷰 제공, 대용량 메트릭 관리를 위한 장기 저장소
- 예거 : 분산 콘텍스트 전파를 포함한 마이크로서비스 기반 분산 시스템 모니터링과 추적을 위해서 사용, 다양한 백엔드 저장소와 통합 가능
- 그라파나 템포 : 그라파타 오픈소스 추적 솔루션
